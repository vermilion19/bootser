# 42dot Open Platform Engineering — 공고 기반 심화 정리

공고의 문장 하나하나가 곧 면접 주제입니다. 이 문서는 공고를 해부해 각 요구사항 뒤에 있는 기술 주제를 "왜 그런지" 수준까지 정리한 학습 자료입니다. 기존 면접 문항 문서(핵심 10문항, 나머지 문항)와 함께 사용하되, 이 문서는 "기반 지식"을 채우는 용도입니다.

---

# 0. 공고 해부 — 이 팀은 무엇을 하는 팀인가

공고 원문: "내부와 외부의 수많은 API의 입구를 책임지며, 보안과 유연성을 갖춘 인증·인가 체계부터 고가용성 플랫폼 운영, API 거버넌스까지"

이 한 문장에 팀의 정체성이 다 있습니다.

- **"API의 입구"** = API Gateway 운영 조직. 모든 트래픽이 지나가는 관문을 만들고 지킴.
- **"내부와 외부"** = 사내 서비스뿐 아니라 외부 파트너(다른 회사, 개발자)에게 API를 개방하는 Open Platform. 그래서 팀 이름이 Open Platform Engineering.
- **"인증·인가 체계"** = IAM. 수많은 클라이언트(사용자, 앱, 파트너 서버, 차량/디바이스)의 신원을 확인하고 권한을 통제.
- **"고가용성 플랫폼 운영"** = 이 관문이 죽으면 전부 죽음. SPOF 제거가 존재 이유 수준의 과제.
- **"API 거버넌스"** = API의 수명주기 관리. 버전, 정책, 문서화, 사용량, 파트너별 계약(플랜/쿼터), 표준 준수.

**면접관의 관점**: 42dot은 현대차그룹의 SDV(Software Defined Vehicle) 기업입니다. 차량, 모빌리티 서비스, 외부 파트너가 전부 API로 연결되는 플랫폼에서, "누가 어떤 API를 어디까지 쓸 수 있는가"를 통제하는 팀입니다. 클라이언트가 사람만이 아니라 차량·디바이스·파트너 서버라는 점이 일반 웹서비스 인증팀과 다른 지점이고, 이게 당신의 IoT 디바이스 인증 경험이 연결되는 접점입니다.

# 1. 주요업무 ①: "API Gateway, 서비스 간 연동 구조를 이해하고, 외부 트래픽 흐름에 맞춘 보안 정책 및 접근 제어 로직을 개발"

## 1-1. API Gateway의 존재 이유

MSA에서 서비스가 수십 개가 되면, 클라이언트가 각 서비스를 직접 호출하는 구조는 유지 불가능해집니다. 인증을 서비스마다 구현해야 하고, 클라이언트는 서비스 위치를 다 알아야 하고, 정책 변경은 전 서비스 배포가 됩니다. Gateway는 이 횡단 관심사(cross-cutting concerns)를 한 곳으로 모은 단일 진입점입니다.

Gateway가 담당하는 것: 라우팅, 인증/인가(1차), rate limiting, 로깅/추적, 요청/응답 변환, TLS 종료, 캐싱, 장애 격리(circuit breaker).

**트래픽 방향 용어 (면접에서 쓰면 좋음)**
- **North-South 트래픽**: 외부 → 내부. Gateway의 영역.
- **East-West 트래픽**: 내부 서비스 간. 서비스 메시(Istio)의 영역.
- 공고의 "외부 트래픽 흐름에 맞춘"은 north-south를 명시한 것. Gateway와 메시는 대체가 아니라 분담 관계.

## 1-2. Gateway의 내부 구조 — 요청의 수명주기

어떤 Gateway든(SCG, Tyk, Kong, Envoy) 본질은 같습니다:

```
요청 수신 → 라우트 매칭 → [미들웨어/필터 체인: pre 단계]
  → 백엔드 프록시 → [체인: post 단계] → 응답 반환
```

- **라우트 매칭**: Path/Host/Header 등 조건(Predicate)으로 어느 백엔드로 보낼지 결정
- **pre 필터**: 인증, rate limit, 헤더 조작 — 백엔드에 닿기 전 차단/가공
- **post 필터**: 응답 헤더 추가, 로깅, 메트릭

Spring Cloud Gateway로 치면 GlobalFilter(전체 적용)/GatewayFilter(라우트별 적용)이고, Tyk으로 치면 미들웨어 체인 + 정책(Policy)입니다. **제품이 달라도 개념은 매핑됩니다** — 이게 "SCG 경험으로 Tyk에 적응할 수 있다"는 주장의 근거입니다.

**SCG의 기술적 특성 (본인 사이드 프로젝트 방어용)**
- WebFlux(Netty) 기반 논블로킹. 이벤트 루프 스레드 소수가 수만 커넥션을 처리.
- 따라서 필터 안에서 블로킹 호출(JDBC, RestTemplate, Thread.sleep)을 하면 이벤트 루프가 막혀 전체 처리량이 붕괴. 이것이 WebFlux 최대 안티패턴.
- Redis 연동도 Lettuce(Netty 기반 논블로킹)를 쓰므로 rate limit의 Redis 왕복 중에도 스레드는 다른 요청을 처리. 단, 네트워크 RTT만큼의 지연 자체는 추가됨 (블로킹 여부와 지연은 별개 개념).

## 1-3. "보안 정책 및 접근 제어 로직" — Gateway에서 하는 보안

Gateway 레벨 보안의 층위를 구분해서 말할 수 있어야 합니다:

1. **인증 (누구인가)**: JWT 검증, API Key 확인, mTLS 클라이언트 인증서
2. **거친 인가 (이 API 그룹에 접근 가능한가)**: 스코프/플랜 기반. "파트너 A는 차량 조회 API 그룹만"
3. **트래픽 통제 (얼마나 쓸 수 있는가)**: rate limit, quota(기간 총량), 동시 연결 제한
4. **입력 방어**: 요청 크기 제한, 허용 메서드/컨텐츠타입 제한, IP allow/deny
5. **세밀한 인가는 Gateway가 아닌 각 서비스에서**: 리소스 소유권("이 차량이 이 사용자 것인가") 같은 도메인 지식이 필요한 판단. Gateway에 도메인 지식이 쌓이면 결합과 배포 병목이 생김.

**API Key vs OAuth 토큰 (Open Platform에서 중요한 구분)**
- API Key: 발급이 간단, 파트너 서버 간 통신에 흔함. 그러나 만료/스코프 개념이 약하고 유출 시 회수가 수동. "식별"에 가깝지 "인증"으로는 약함.
- OAuth Client Credentials + 단기 토큰: 발급 절차는 무겁지만 만료·스코프·회수가 표준화됨.
- 실무 답: 파트너 온보딩은 API Key(간편) 또는 client credentials, 민감 API는 단기 토큰 요구. "파트너 성숙도와 API 민감도에 따라 계층화"가 좋은 답.

## 1-4. 트래픽 관리 심화 — Rate Limiting

**알고리즘 4종과 선택 기준**

| 알고리즘 | 동작 | 특성 | 용도 |
|---|---|---|---|
| Fixed Window | 시간 칸막이별 카운트 | 경계에서 2배 버스트 통과 결함 | 간단한 내부용 |
| Sliding Window | 연속된 과거 N초 집계 | 정확하지만 저장/계산 비용 | 정확성 중요할 때 |
| Token Bucket | 일정 속도로 토큰 충전, 요청이 소모 | 버킷 용량만큼 순간 버스트 허용 | API 트래픽 표준 |
| Leaky Bucket | 고정 속도로만 유출 | 버스트를 평탄화 | 다운스트림 보호 |

암기 축: **Token = 버스트 허용, Leaky = 버스트 평탄화.** API 플랫폼은 정상 사용자의 스파이크를 봐주는 Token Bucket이 기본값.

**분산 rate limit의 원자성 문제**
- 인스턴스별 로컬 카운터는 전역 제한 불가 → Redis 중앙 카운터
- "GET으로 조회 → 판단 → INCR" 두 명령으로 나누면 그 사이에 다른 요청이 끼어드는 레이스 컨디션
- 해법: **Lua 스크립트**. Redis는 단일 스레드로 Lua 스크립트 전체를 원자적으로 실행. "남은 토큰 계산 + 차감 + 만료 설정"이 한 번에 수행됨. SCG RedisRateLimiter가 이 방식.

**Redis 왕복 지연까지 줄이는 고급 패턴 (꼬리 질문 대비)**
- 로컬 근사 + 중앙 정밀의 2단 구조
- 토큰 배치 선할당: 인스턴스가 토큰 N개를 미리 받아 로컬 소진, 왕복 빈도 1/N
- 본질은 정확도 vs 지연 트레이드오프. "요구 정확도에 따라 선택"이 정답 프레임.

**응답 규약**: 429 Too Many Requests + Retry-After, X-RateLimit-Limit/Remaining/Reset 헤더. 파트너가 백오프를 구현할 수 있게 하는 것까지가 플랫폼의 책임.

## 1-5. 장애 격리 3종 세트 (Resilience4j 사이드 경험의 이론화)

**Circuit Breaker** — "죽은 서비스는 빨리 포기"
- 위험한 건 죽은 서비스가 아니라 **느린 서비스**. Gateway의 커넥션/리소스가 느린 응답 대기에 묶여 멀쩡한 경로까지 마비 → 연쇄 장애.
- 상태 전이: CLOSED(정상, 실패율 집계) → 임계 초과 시 OPEN(호출 없이 즉시 실패) → 대기 후 HALF-OPEN(일부만 통과시켜 회복 확인) → 성공 시 CLOSED / 실패 시 OPEN.
- HALF-OPEN의 존재 이유: OPEN→CLOSED 직행 시 회복 중인 서비스에 전 트래픽이 쏟아져 재붕괴. "간 보기" 단계가 필수.

**Bulkhead** — "공간적 격리"
- 서비스별 동시 호출 수/리소스 풀을 분리. A 폭주가 B용 리소스를 침범 못 함. CB가 시간적 격리라면 Bulkhead는 공간적 격리.

**Retry의 함정** — retry storm
- 과부하로 타임아웃 → 클라이언트 일제 재시도 → 트래픽 증폭 → 완전 사망.
- 안전 3원칙: 지수 백오프 + 지터(동시 재시도 분산) + 멱등 요청에만. (TelemetryHub의 reconnect storm 대응과 동일 원리 — 본인 경험 연결 포인트)

**Load Shedding**
- 한계 도달 시 전부 느리게 처리하지 말고 일부를 즉시 429/503으로 거절. "Gateway는 죽지 않고 거절한다."

# 2. 주요업무 ②: "안정성, 보안, 확장성을 고려한 백엔드 아키텍처 설계 및 코드 품질 개선을 주도"

## 2-1. 인증 시스템의 고가용성 — 이 팀의 1급 과제

인증이 죽으면 전부 죽는다는 것이 이 팀의 숙명. 설계 원칙은 하나로 수렴합니다: **"핫패스(매 요청 경로)에서 인증 서버 의존을 제거하라."**

**Introspection vs 로컬 검증 (핵심 구도)**

| | Introspection (원격) | 로컬 검증 |
|---|---|---|
| 방식 | 매 요청마다 인증 서버에 유효성 질의 | 캐싱된 공개키로 서명을 자체 검증 |
| 인증 서버 부하 | 전체 트래픽에 비례 | 로그인/발급 시에만 |
| 인증 서버 장애 시 | 전 서비스 마비 | 기존 로그인 사용자 무영향 (신규 로그인만 불가) |
| 무효화 즉시성 | 즉시 반영 | 어려움 (TTL까지 유효) |

- "로컬"의 의미: 검증하는 그 서버(Gateway/각 서비스) **자기 내부에서**, 외부에 묻지 않고. JWT + RS256 + JWKS 공개키 캐싱으로 가능해짐.
- 트레이드오프 보완: 짧은 Access TTL(5~15분) + 필수 케이스만 블랙리스트(jti를 Redis에 잔여 TTL만큼) + Refresh는 서버 관리.
- 한 줄 답변: "introspection은 인증 서버를 병목이자 SPOF로 만들므로 로컬 검증으로 핫패스에서 제거하고, 즉시 무효화의 공백은 짧은 TTL과 블랙리스트로 보완합니다."

**무중단 키 회전 (운영 디테일, 우대사항 "운영 경험" 직결)**
1. 새 키페어 생성, 공개키를 JWKS에 추가 (구키+신키 공존)
2. 신규 토큰은 신키로 서명 (토큰 헤더의 kid로 어느 키인지 표시)
3. 구키로 서명된 토큰이 전부 만료된 후 구키를 JWKS에서 제거
- 검증 측은 kid를 보고 캐시에서 키를 찾고, 캐시에 없는 kid면 JWKS 재조회. 이 로직이 있어야 회전이 무중단이 됨.

## 2-2. Gateway 자체의 SPOF 제거

1. **Stateless 유지 + 수평 다중화**: 상태(세션, rate limit 카운터)는 Redis로 외부화. LB 뒤에 N대, 헬스체크로 자동 제외.
2. **의존성 격리**: Gateway보다 Gateway의 의존물(JWKS, Redis, 디스커버리)이 장애 전파 경로. 캐싱 + fallback 정책 + CB로 격리.
3. **fail-open vs fail-closed 판단 기준**: rate limit용 Redis 장애 → 보통 fail-open(허용)+알림. 블랙리스트 조회 장애 → 민감 경로(결제, 관리자)만 fail-closed. "일괄 정답이 아니라 경로별 위험도로 결정"이 성숙한 답.
4. **Graceful shutdown**: LB에서 먼저 제외 → 진행 중 요청 drain → 종료. 배포가 장애가 되지 않게.

## 2-3. "코드 품질 개선을 주도" — 이 문구가 묻는 것

- 테스트: 인증/인가는 **거부 케이스(401/403)가 성공 케이스보다 중요**. 엔드포인트×역할 매트릭스 테스트, 만료는 Clock 주입으로 시간 제어. (커버리지 80% 경험을 이 언어로 재포장)
- 리팩토링 경험 서사: "동작 변경 없이 구조 개선 + 테스트가 안전망" 구도로 본인 사례 1개 준비 (예: 배치 페이징 전환, N+1 제거).
- 코드 리뷰 문화에 대한 자기 생각 1문단 준비 (자격요건 "커뮤니케이션 및 협업"과 연결).

# 3. 주요업무 ③: "사용자와 API의 인증 및 인가 흐름을 설계하고 구현"

주목: "사용자**와 API**의". 사람 인증만이 아니라 **기계(서비스, 파트너 서버, 디바이스) 인증**을 명시했습니다. 이게 이 팀의 특수성이자 당신의 접점입니다.

## 3-1. OAuth 2.0 — 인가 위임 프레임워크

**본질**: 비밀번호를 넘기지 않고 제3자에게 제한된 권한을 위임하는 표준.

**등장인물 4**: Resource Owner(사용자) / Client(우리 서비스) / Authorization Server(토큰 발급) / Resource Server(자원 API)

**Grant Type = 토큰을 발급받는 시나리오** (환경마다 안전한 절차가 다름)

| Grant Type | 대상 | 핵심 |
|---|---|---|
| Authorization Code | 백엔드 있는 웹앱 | code를 백채널에서 secret과 함께 토큰 교환 |
| Authorization Code + PKCE | SPA, 모바일 | secret 없이 code 교환을 보호 |
| Client Credentials | 서버 간, 디바이스 | 사람 없음. client 자격만으로 발급 |
| Refresh Token | 갱신 | 재로그인 없이 Access 재발급 |
| (폐기) Implicit | 구 SPA | 토큰을 URL로 직접 전달 → 노출 위험으로 폐기 |

**Authorization Code에서 code를 거치는 이유** (자주 나옴)
- code는 브라우저(프론트채널: URL, 히스토리에 남는 위험한 경로)로 전달되는 **일회용 임시 교환권**일 뿐.
- 진짜 토큰 교환은 서버 간(백채널)에서 client_secret과 함께. → 토큰이 브라우저에 노출되지 않음.
- Implicit는 이 분리를 없애고 토큰을 프론트채널로 직접 던진 것 → 폐기.

**PKCE — client_secret이 없는 환경의 code 보호**
- 문제: SPA/모바일은 코드가 사용자 기기에 내려가 secret을 숨길 수 없음 → code를 가로챈 공격자가 토큰 교환 가능.
- 해법: 요청마다 임시 비밀 생성. code_verifier(랜덤 원본, 클라이언트 메모리에만) → code_challenge(SHA-256 해시)를 로그인 요청에 첨부 → 토큰 교환 때 원본 제출 → 서버가 해시 대조.
- 공격자는 4단계에서 code를 훔쳐도 원본 verifier를 몰라 교환 불가. 해시는 역산 불가.
- 판단 기준 한 줄: **"PKCE는 비밀을 숨길 곳 없는 클라이언트(Public Client)가 code를 직접 교환할 때 필요."** 백엔드가 secret으로 교환하는 구조(Confidential Client)는 필수 아님(OAuth 2.1은 웹에도 권장 추세). SDK가 ID Token을 직접 주는 네이티브 흐름은 code 자체가 없어 PKCE 논외.

**Confidential vs Public Client** (이 용어를 쓰면 이해도가 드러남)
- Confidential: secret을 서버에 숨길 수 있음 (백엔드 웹앱)
- Public: 숨길 수 없음 (SPA, 네이티브 앱) → PKCE 필수

## 3-2. OIDC — OAuth 위의 인증 표준

- OAuth의 Access Token은 "권한 증표"일 뿐 "사용자가 누구인지"를 보장하지 않음. OAuth만으로 로그인 구현 시 **token substitution**(타 앱용 토큰 재사용) 공격에 취약.
- OIDC가 추가한 것: **ID Token(JWT)** — sub, iss, **aud**(발급 대상 클라이언트), 인증 시각 클레임을 서명과 함께. aud 검증으로 "우리 앱에 발급된 인증 증명"임을 못박음.
- 소비 주체 구분: ID Token은 Client가(신원 확인), Access Token은 Resource Server가(접근 판단). ID Token으로 API 호출하면 안 됨.
- 부속 규격: Discovery(.well-known/openid-configuration — 엔드포인트/알고리즘 메타데이터), UserInfo 엔드포인트, 표준 스코프(openid, profile, email).
- **소셜 로그인 = OIDC의 사용 사례.** 본인 구현 경험을 이 언어로 설명할 것.

**모바일 네이티브 SDK 흐름의 서버 검증 체크리스트 (본인 서비스 실사례)**
- SDK가 ID Token을 직접 발급 → 서버는: ① 서명(구글/애플 공개키) ② **aud = 우리 client_id** ③ iss = 정당한 발급자 ④ exp. aud 누락 시 token substitution 구멍.

## 3-3. JWT 심화

**구조**: header(alg, kid).payload(claims).signature — 각각 Base64URL. **서명이지 암호화가 아님**: 위조 불가 ≠ 열람 불가. 민감정보 금지.

**표준 클레임**: sub(주체) iss(발급자) aud(대상) exp(만료) iat(발급시각) jti(토큰 고유 ID — 블랙리스트 키로 사용)

**HS256 vs RS256**
- HS256(대칭): 서명=검증 키. 검증하는 모든 곳에 비밀키 배포 → 한 곳 유출 시 전체 위조 가능. 단일 서비스용.
- RS256(비대칭): 인증 서버만 개인키 서명, 나머지는 공개키 검증. 유출 격리 + JWKS 배포 + kid 회전. **MSA/Gateway 표준.**

**알고리즘 공격 (보안팀 면접 단골)**
- alg:none 공격 — 서명 없음을 선언한 토큰을 통과시키는 취약 구현
- RS256→HS256 다운그레이드 — 공개키를 HMAC 비밀키로 쓰게 유도
- 방어: 서버가 허용 알고리즘을 화이트리스트로 고정, 토큰의 alg를 신뢰하지 않음

**무효화 전략 3종**: 짧은 TTL(피해 시간 제한) / jti 블랙리스트(즉시 차단 필요 이벤트만, 잔여 TTL만큼 Redis 저장) / Refresh 서버 관리(삭제 시 갱신 차단). + 사용자 단위 일괄 무효화는 token_version 클레임 비교.

**Refresh Token 보안**: 저장은 HttpOnly+Secure+SameSite 쿠키(웹) / Keychain·Keystore(모바일). **Rotation**(사용 시마다 재발급+구토큰 폐기=일회용화) + **재사용 감지**(폐기된 토큰 재사용 = 탈취 신호 → 토큰 패밀리 전체 무효화). Rotation 중 응답 유실로 정상 사용자가 걸리는 문제 → grace period 절충.

## 3-4. "API의 인증" — 기계 주체의 인증 (당신의 차별화 영역)

사람 인증과 다른 점 3가지 (Vueroid 경험으로 뒷받침):
1. **비대화형**: 로그인 화면/리다이렉트 불가 → 사전 발급 자격 증명 기반. OAuth로는 Client Credentials Grant.
2. **긴 수명 연결**: MQTT/WebSocket은 연결 시 1회 인증 후 수 시간 유지 → 연결 중 자격 취소 반영 문제(주기 재검증, 서버 측 강제 종료).
3. **대량 자동 재접속**: reconnect storm → 클라이언트 지수 백오프+지터, 서버 수락 rate limit, 인증 경로 캐싱.

서비스 간(east-west) 인증 선택지: Client Credentials 토큰 / mTLS(메시가 자동화) / API Key(약함). 파트너 서버 인증도 동일 프레임.

**디바이스 자격 증명 원칙**: 디바이스별 고유 자격 증명(공용 키 금지 — 하나 유출=전체 유출), 서버 측 개별 폐기 체계.

# 4. 자격요건 딥다이브

## 4-1. "Tyk 또는 유사한 API Gateway 운영 및 커스터마이징"

Tyk 개요 (미경험 전제, 개념 매핑으로 방어):
- Go 기반 오픈소스/상용 Gateway. **제품형**: Gateway(데이터 플레인) + Dashboard(관리 플레인) + 개발자 포털 + 분석이 한 세트.
- 핵심 개념 매핑: API Definition(=라우트 정의), **Policy**(rate limit·quota·스코프를 묶어 키/파트너에 부여 — 파트너별 플랜의 구현체), 미들웨어 체인(=필터 체인, pre/post 훅), 커스텀 플러그인(Go/gRPC/JS/Python).
- SCG와의 구도: **프레임워크(SCG, 코드로 조립, 내부 MSA 진입점) vs 제품(Tyk, 정책으로 운영, 외부 API 상품화)**. Open Platform처럼 파트너 관리·포털·사용량 분석이 필요한 조직은 제품형이 맞음.
- 답변 전략: "Tyk 실무는 없지만 SCG로 Gateway의 동작 원리(라우팅·필터·인증·rate limit)를 직접 구현해봐서, Tyk의 Policy·미들웨어 개념을 매핑해 학습할 수 있다."

## 4-2. "IAM 관련 도메인 이해"

IAM의 구성 요소를 한 장으로:
- **Identity 수명주기**: 등록(provisioning) → 인증 → 권한 부여 → 갱신/변경 → 폐기(deprovisioning). "퇴사자/해지 파트너의 접근이 확실히 끊기는가"가 IAM의 실전 품질.
- **주체(Principal)의 종류**: 사용자, 서비스 계정, 디바이스, 파트너 조직. 각각 인증 방식이 다름 (3-4 참고).
- **인증 요소**: 지식(비밀번호)/소유(기기, OTP)/생체. MFA = 두 요소 이상.
- **연합 인증(Federation)**: 외부 IdP(구글, 사내 SSO)에 인증을 위임하고 결과만 신뢰 — OIDC가 그 표준. SSO는 한 번 인증으로 여러 서비스 접근.

## 4-3. RBAC / ABAC (우대사항 "복합 권한 모델"과 통합)

- **RBAC**: 사용자→역할→권한. 직관적, 관리 용이. 한계: 조건이 늘면 역할 폭발(role explosion). "부서×지역×등급" 조합이 전부 역할이 되는 순간 붕괴.
- **ABAC**: 주체 속성 × 리소스 속성 × 환경 속성(시간, IP)으로 정책 평가. 세밀하지만 정책 복잡도·디버깅 비용.
- **실무 하이브리드**: RBAC 뼈대(플랜/역할) + ABAC 보완(소유권, 조건). 토큰에는 역할/스코프/테넌트만, 세부 권한은 서비스 조회+캐싱.
- **인가 배치**: Gateway = 스코프 기반 거친 인가(무의미 트래픽 조기 차단), 서비스 = 도메인 인가(리소스 소유권). 한쪽에 몰면 결합 또는 낭비.
- **멀티테넌시 격리**: 모든 판단의 기준은 "클라이언트가 보낸 tenant_id"가 아니라 **"토큰의 tenant_id"**. 쿼리 레벨 tenant 필터 강제.
- **정책 외부화**: OPA(Open Policy Agent) — 정책을 코드에서 분리해 선언적으로 관리·감사. 개념만 알아도 우대 어필.

# 5. 우대사항 딥다이브

## 5-1. "Kubernetes, Istio, Envoy 등 클라우드 네이티브 환경에서의 보안 통합"

미경험 영역. 목표 수위: "개념을 정확히 알고, 내 경험과 매핑해 말한다."

**K8s 최소 개념**: Pod(배포 최소 단위) / Deployment(replica·롤링업데이트) / Service(Pod 집합의 안정적 접근점) / Ingress(외부 노출) / HPA(자동 확장). Docker 실무 경험에서 컨테이너 기초는 있음을 연결.

**서비스 메시 핵심 아이디어**: 각 Pod 옆에 Envoy 프록시를 **사이드카**로 배치, 서비스 간 모든 통신이 프록시를 경유 → 통신 제어(암호화·인증·라우팅·관측)를 앱 코드에서 인프라로 이동.

**mTLS**: 일반 TLS는 클라이언트가 서버만 검증, mTLS는 **상호 검증**. 메시에서는 워크로드마다 고유 인증서(SPIFFE 신원)가 자동 발급·회전되고 사이드카끼리 mTLS. 효과 = 내부 트래픽 암호화 + **호출자 서비스의 암호학적 신원 보장**. 앱은 인증서 관리를 모름.

**인가 정책**: 그 신원 기반으로 "결제 서비스는 주문 서비스만 호출 가능"을 선언적 정책(Istio AuthorizationPolicy)으로. JWT 검증도 프록시 레벨 수행 가능.

**연결 멘트**: "내가 Gateway 필터 코드로 했던 검증·접근 제어의 상당 부분을 메시는 인프라 정책으로 대신한다. 단 메시는 east-west, Gateway는 north-south라 대체가 아닌 보완." + "정책 운영·디버깅의 실제 난이도는 겪어봐야 아는 영역이라 입사 후 빠르게 채우고 싶다."

## 5-2. "감사 로깅, 정책 기반 접근 제어 등 엔터프라이즈 보안 기능 운영"

**감사 로그 설계 프레임**
- 목적: "누가, 언제, 무엇을, 어떻게, 결과는"의 사후 재구성. 특히 인증 이벤트(로그인 성공/실패, 토큰 발급/폐기, 권한 변경)는 필수.
- 필드: 행위자(사용자/클라이언트/IP), 시각, 행위(API·리소스), 결과(성공/실패·사유), trace id. 금지: 비밀번호, 토큰 원문(jti만), 민감 개인정보.
- 파이프라인: 요청 경로에서 동기 기록은 본 트래픽에 부담 → 이벤트 발행 → Kafka 버퍼 → 컨슈머 배치 적재. (TelemetryHub의 수집 파이프라인·배치 upsert와 동일 패턴 — 본인 경험 직결)
- 유실 요구가 엄격하면: Transactional Outbox로 본 트랜잭션과 발행의 원자성 보장 (사이드에서 구현해본 패턴).
- 저장: 검색형(OpenSearch) 또는 시간 파티셔닝 RDB, 핫/콜드 수명주기, append-only 권한 분리(변조 방지).

**정책 기반 접근 제어(PBAC)**: 권한 판단을 코드 분기에서 꺼내 정책 문서(OPA Rego 등)로 관리. 이점 = 변경 이력·감사 가능, 배포 없이 정책 갱신, 보안팀과 협업 접점.

## 5-3. "신규 기술 도입 및 개선을 위한 PoC 수행 및 전파"

PoC 방법론 4단계 (질문이 안 나와도 답변 태도로 스며들게):
1. **평가 기준 선정의**: 도입 배경 요구사항 문서화 (통합 가능성, 성능 예산 p99, 운영 요건). 기준 없는 PoC는 "써보니 좋더라"로 끝남.
2. **리스크 큰 시나리오부터 검증**: 실패하면 도입 무산인 것부터 (예: 우리 토큰 검증을 플러그인으로 구현 가능한가).
3. **정량 측정**: 동일 조건 부하 테스트, 평균 아닌 **p95/p99**, 장애 시나리오 거동 포함. (Prometheus/Grafana 사이드 경험 연결)
4. **전파**: 기준 대비 충족표 + 미충족 항목의 우회 비용 + 도입/보류 권고 + 재현 가능한 문서.
- 본인 소재: WebSocket→MQTT 전환 검토(LTE 비용) 경험을 이 프레임으로 재구성 가능.

# 6. 경험 매핑표 — 공고 문장 ↔ 내 무기

| 공고 요구 | 내 경험 | 면접 변환 멘트 |
|---|---|---|
| 사용자와 API의 인증/인가 | Spring Security + OAuth2 소셜 로그인(웹 code flow + 모바일 SDK flow), MQTT/WebSocket 디바이스 인증 | "사람과 기계 양쪽 주체의 인증을 실무로 다룸. 디바이스 인증은 Client Credentials 개념에 해당" |
| API Gateway 운영/커스터마이징 | SCG GlobalFilter로 JWT 인증 필터, Redis rate limit 직접 구현 | "제품 운영은 아니지만 Gateway의 원리를 코드 수준에서 구현·검증" |
| 트래픽 관리 | Redis rate limit, Resilience4j CB/Bulkhead/RateLimiter, 부하 테스트 | "장애 격리 3종을 직접 조합해 장애 전파 차단을 검증" |
| 고가용성 아키텍처 | 다중 인스턴스 WebSocket + Redis Pub/Sub, Stateless 설계, CloudWatch 관측 | "다중 인스턴스 환경의 상태 외부화와 관측을 운영" |
| 안정성/확장성 리팩토링 | 45초→0.8초 쿼리 최적화, 배치 페이징 전환, N+1 제거 | "측정(실행 계획) 기반 개선 + 테스트 안전망" |
| 감사 로깅 | Kafka 파이프라인, 배치 적재, Outbox (사이드) | "감사 로그 적재는 내가 만든 텔레메트리 파이프라인과 동일 패턴" |
| 다양한 클라이언트 환경 인증 플로우 | 웹/모바일/디바이스 3종 인증 흐름 분기 설계 (실서비스) | "클라이언트 특성(secret 보관 가능 여부, code 유무)에 따라 흐름을 의도적으로 분리" |
| PoC | WebSocket→MQTT 전환 검토 | "비용·성능 기준을 세우고 검증 후 전환" |

공백 영역과 태도: Tyk(개념 매핑으로), K8s/Istio(개념+학습 계획), RBAC/ABAC 운영(설계 원칙으로). 공통 화법 = **현재 이해 → 내 경험과의 연결 → 검증/학습 계획.**

# 7. 이 문서에서만 나올 수 있는 심화 예상 질문 (기존 문서 미포함분)

1. API 거버넌스가 뭐라고 생각하나? 파트너에게 API를 개방할 때 버전 정책(하위호환, deprecation)을 어떻게 운영하겠나?
2. API Key와 OAuth 토큰 기반 파트너 인증의 장단점을 비교하고 언제 무엇을 쓸지 설명하라.
3. quota(월 총량)와 rate limit(초당)의 차이는? 각각 어떻게 구현이 달라지나? (quota는 정확한 누적 집계 필요 → 별도 카운팅/정산 파이프라인)
4. 파트너별 요금제(플랜)에 따라 접근 API·쿼터가 다르다. 토큰/정책/Gateway 설정 중 어디에 이 정보를 두겠나?
5. alg:none / 알고리즘 다운그레이드 공격을 설명하고 방어법을 말하라.
6. 외부 IdP(애플 로그인)가 장애일 때 우리 서비스의 graceful degradation 시나리오는?
7. north-south와 east-west 트래픽의 보안 책임을 어떻게 나누겠나? (Gateway vs 메시)
8. 감사 로그와 애플리케이션 로그의 차이는? (목적: 사후 재구성 vs 디버깅, 요구: 완전성·변조방지 vs 유연성)
9. Gateway 신규 도입 PoC에서 "성능"을 어떻게 측정 설계하겠나? (동일 조건, p99, 장애 주입까지)
10. 차량/디바이스처럼 토큰 갱신 주기가 긴 클라이언트의 인증은 무엇이 다른가? (본인 경험 + 오프라인 기간, 자격 회수 반영 지연 문제)
