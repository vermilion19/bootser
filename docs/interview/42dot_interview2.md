## Q1. 소셜 로그인에서 provider(구글/카카오 등)마다 응답 형식이 다른데 어떻게 추상화했나요?

### 모범 답변

provider마다 사용자 정보의 구조가 전부 다릅니다. 구글은 최상위에 sub, email이 오지만, 카카오는 kakao_account 안에 중첩되어 있고, 네이버는 response 객체로 감싸져 있습니다. 이걸 분기문으로 처리하면 provider가 늘 때마다 인증 코드 전체를 수정해야 합니다.

그래서 저는 **공통 인터페이스 + provider별 구현체** 구조로 추상화했습니다. 서비스가 실제로 필요로 하는 정보(식별자, 이메일, 이름, provider 타입)를 담는 표준 모델(예: OAuth2UserInfo)을 정의하고, provider별 구현체가 각자의 응답 JSON을 이 모델로 변환하는 책임만 가집니다. 신규 provider 추가는 구현체 하나를 추가하는 것으로 끝나고, 인증 이후의 회원 가입/연동 로직은 표준 모델만 바라보므로 전혀 수정되지 않습니다.

Spring Security의 OAuth2 Client를 쓰면 registrationId로 provider를 식별할 수 있어, 이 값을 팩토리의 분기 키로 사용해 구현체를 선택하는 구조로 구성했습니다.

또 하나 중요한 지점은 **계정 통합 정책**입니다. 같은 이메일로 구글과 카카오에 각각 가입하면 별개 계정이 될지, 하나로 합칠지를 정해야 합니다. provider가 이메일을 안 주는 경우(카카오 비즈 미인증 등)도 있어서, provider + provider별 고유 ID 조합을 기본 식별자로 삼고 이메일은 보조 수단으로 쓰는 것이 안전하다고 판단했습니다.

### 예상 꼬리 질문

- **Q. provider의 API 스펙이 바뀌면 어떻게 대응하나요?**
  → 변환 로직이 구현체 한 곳에 격리되어 있어 수정 범위가 명확. 변환 실패를 감지하는 로그/알림을 두면 스펙 변경을 조기 발견 가능.

- **Q. 소셜 계정 연동 해제(unlink) 시 고려사항은?**
  → 마지막 로그인 수단을 해제하면 계정 접근 불가가 되므로, 최소 1개 인증 수단 보장 검증 필요. provider 측 연동 해제 API 호출도 함께.

## Q2. JWT 페이로드에 넣으면 안 되는 정보는 무엇이고, 토큰이 커지면 어떤 문제가 생기나요?

### 모범 답변

JWT는 **암호화가 아니라 서명**입니다. Base64 인코딩일 뿐이라 누구나 디코딩해서 내용을 볼 수 있습니다. 따라서 넣으면 안 되는 것은:

1. **민감 정보** - 비밀번호(해시라도), 주민번호, 전화번호, 상세 개인정보. 토큰이 로그, 브라우저, 프록시 등 여러 곳을 거치므로 유출 경로가 많습니다.
2. **자주 바뀌는 정보** - 포인트 잔액, 상세 권한 목록 같은 것. 토큰 만료 전까지 갱신이 안 되어 정합성 문제가 생깁니다.
3. **커질 수 있는 정보** - 소속 그룹 전체 목록, 권한 전체 목록 등.

토큰이 커지면 생기는 문제는 실질적입니다. JWT는 **모든 요청의 헤더에 실려 다니므로**, 토큰이 4KB면 초당 1만 요청 기준 대역폭이 순수 토큰만으로 상당히 소모됩니다. 또 웹서버·프록시의 헤더 크기 제한(보통 8KB 안팎)에 걸려 431 에러가 나거나, 쿠키에 담는 경우 쿠키 한도(4KB)를 초과할 수 있습니다.

원칙은 "**토큰에는 식별과 거친 인가에 필요한 최소한만**"입니다. sub, 테넌트, 역할/스코프 정도만 담고, 상세 정보는 서비스가 필요 시 조회하고 캐싱하는 구조가 맞습니다.

### 예상 꼬리 질문

- **Q. 그래도 토큰에 민감 정보를 꼭 담아야 한다면?**
  → JWE(암호화된 JWT) 사용 가능. 다만 키 관리 복잡도가 크게 올라가므로, 대부분은 "토큰에 안 담는 설계"가 정답.

- **Q. 클레임이 위조되지 않는다는 건 어떻게 보장되나요?**
  → 서명 검증으로 보장. 단 "위조 불가"와 "열람 불가"는 다름을 구분해서 말할 것.

## Q3. 세션 방식과 JWT 방식의 트레이드오프를 설명해 주세요. Stateless가 항상 장점인가요?

### 모범 답변

**세션 방식**은 서버가 상태를 가집니다. 모든 요청마다 세션 저장소를 조회하지만, 그 대가로 즉시 무효화, 실시간 권한 반영, 동시 접속 제어가 자연스럽게 됩니다. 다중 인스턴스에서는 세션 저장소(Redis 등)를 공유해야 하고, 이 저장소가 병목이자 장애 지점이 됩니다.

**JWT 방식**은 검증이 로컬에서 끝나 저장소 의존이 없고, 수평 확장과 서비스 간 신뢰 전달(MSA)에 유리합니다. 대신 발급된 토큰을 회수할 수 없어 즉시 무효화가 어렵고, 이를 보완하려 블랙리스트를 붙이는 순간 부분적으로 Stateful로 돌아갑니다.

그래서 **Stateless가 항상 장점은 아니라고 생각합니다.** 판단 기준은 시스템의 요구사항입니다:

- 단일 서비스 + 즉시 세션 제어가 중요(관리자 시스템, 금융) → 세션이 오히려 단순하고 안전
- 다수 서비스가 하나의 인증을 공유(MSA, 플랫폼) + 트래픽 규모가 큼 → JWT의 로컬 검증이 구조적으로 유리
- 현실적인 다수의 시스템 → JWT(짧은 TTL) + Refresh Token 서버 관리 + 예외적 블랙리스트, 즉 하이브리드

Vueroid Cloud에서는 디바이스·다중 인스턴스 환경 특성상 토큰 기반이 맞았고, 사이드 프로젝트 MSA에서도 Gateway 로컬 검증 구조를 택했습니다. 다만 두 경우 모두 "완전한 Stateless"가 아니라 Refresh 관리 등 일부 상태를 인정하는 절충이었습니다.

### 예상 꼬리 질문

- **Q. 세션 저장소로 Redis를 쓰면 JWT와 성능 차이가 실제로 큰가요?**
  → Redis 조회는 밀리초 미만이라 절대 성능 차이는 작음. 본질적 차이는 성능보다 "인증 상태의 소유권"(중앙 vs 분산)과 장애 결합도. 규모가 커질수록 매 요청 중앙 조회의 비용과 리스크가 누적됨.

- **Q. sticky session은 왜 안 쓰나요?**
  → 인스턴스 교체/배포 시 세션 유실, 부하 불균형. 클라우드 환경의 탄력적 확장과 근본적으로 안 맞음.

## Q4. Spring Cloud Gateway의 필터 체인 구조를 설명해 주세요.

### 모범 답변

Spring Cloud Gateway는 요청이 들어오면 **Route 매칭 → 필터 체인 실행 → 백엔드 프록시** 순으로 동작합니다.

1. **Route와 Predicate** - 각 Route는 Predicate(Path, Header, Method 등 조건)와 필터 목록, 대상 URI로 구성됩니다. 요청이 Predicate에 매칭되면 해당 Route가 선택됩니다.
2. **필터의 두 종류** - **GatewayFilter**는 특정 Route에만 적용되고, **GlobalFilter**는 모든 Route에 적용됩니다. 실행 시에는 둘이 하나의 체인으로 합쳐지고 Order 값으로 순서가 결정됩니다.
3. **pre / post 처리** - 필터는 chain.filter(exchange) 호출 **이전 코드가 pre 필터**(요청 가공, 인증), **이후(then)에 붙는 코드가 post 필터**(응답 가공, 로깅)입니다. WebFlux 기반이라 리액티브 체인으로 연결됩니다.

사이드 프로젝트에서는 JWT 인증을 GlobalFilter의 pre 단계에서 구현했습니다. 토큰 검증 실패 시 chain.filter를 호출하지 않고 401 응답을 바로 쓰는 방식으로 요청을 조기 차단했고, 검증 성공 시 클레임을 내부 헤더로 mutate해서 다음 필터로 넘겼습니다. 인증처럼 모든 요청에 필요한 것은 GlobalFilter로, 특정 API 그룹에만 필요한 것(예: 특정 경로 rate limit)은 Route 필터로 나누는 기준을 적용했습니다.

주의할 점은 SCG가 **WebFlux(논블로킹)** 기반이라는 것입니다. 필터 안에서 블로킹 호출(JDBC, RestTemplate)을 하면 이벤트 루프가 막혀 전체 처리량이 무너집니다. 외부 조회가 필요하면 리액티브 클라이언트를 쓰거나 별도 스케줄러로 격리해야 합니다.

### 예상 꼬리 질문

- **Q. 필터 순서(Order)가 왜 중요한가요? 예를 들어주세요.**
  → 인증 필터는 로깅·rate limit보다 앞? 뒤? 논쟁 가능: rate limit을 인증보다 앞에 두면 비인증 트래픽의 폭주로부터 인증 로직을 보호할 수 있음. 목적에 따라 순서 설계가 달라짐을 보여줄 것.

- **Q. Gateway에서 응답 바디를 수정할 수 있나요?**
  → 가능하지만(ModifyResponseBody) 바디 전체 버퍼링이 필요해 메모리·스트리밍 비용이 큼. 가급적 헤더 수준 조작에 그치는 것이 좋음.

- **Q. WebFlux를 안 써봤다면 어려운 점이 뭘 것 같나요?**
  → 블로킹 코드 혼입 실수, 디버깅 스택트레이스 추적 어려움, ThreadLocal 기반 도구(MDC 등) 사용 제약. 솔직하게 인지하고 있는 리스크를 말하는 것이 좋음.

## Q5. Tyk 같은 전용 API Gateway 제품과 Spring Cloud Gateway의 차이는 뭐라고 생각하나요?

### 모범 답변

Tyk을 실무에서 운영해본 경험은 없어서, 조사하고 이해한 수준에서 말씀드리면 가장 큰 차이는 **"제품이냐, 프레임워크냐"**라고 생각합니다.

**Tyk**은 Go로 작성된 완성형 Gateway 제품입니다. API Key 관리, 개발자 포털, 대시보드, 사용량 분석, 버전 관리, 플랜별 쿼터 같은 **API 관리(Management) 기능**이 내장되어 있고, 설정과 정책 중심으로 운영합니다. 커스터마이징은 플러그인(Go, gRPC, JS 등)으로 합니다. 외부 파트너에게 API를 상품처럼 제공하는 "Open Platform" 성격의 조직에 적합한 도구입니다.

**Spring Cloud Gateway**는 개발자가 코드로 조립하는 **프레임워크**입니다. 자유도가 높아 도메인 특화 로직을 자바 생태계 안에서 자연스럽게 녹일 수 있지만, 관리 기능(포털, 키 발급, 분석)은 전부 직접 만들어야 합니다. 내부 MSA의 진입점 용도에 적합합니다.

정리하면 SCG는 "내부 라우팅 + 커스텀 로직"에, Tyk은 "외부 공개 API의 수명주기 관리"에 강점이 있다고 이해하고 있습니다. 42dot처럼 외부 파트너 대상 API 플랫폼을 운영하는 조직이 Tyk 계열을 쓰는 이유도 여기에 있다고 생각합니다. 저는 SCG로 Gateway의 동작 원리(라우팅, 필터, 인증, rate limit)를 직접 구현해봤기 때문에, Tyk의 개념(정책, 미들웨어 체인)을 매핑해서 학습하는 데 진입 장벽이 낮을 것이라 생각합니다.

### 예상 꼬리 질문

- **Q. Tyk 플러그인으로 커스텀 인증을 붙인다면 어떤 구조일 것 같나요?**
  → 요청 수명주기의 특정 훅(auth 단계)에 커스텀 미들웨어를 등록하는 구조로 이해. SCG의 필터 체인과 개념적으로 유사하다고 답하며, 정확한 스펙은 도입 시 문서 기반으로 검증하겠다고 마무리.

- **Q. Gateway를 직접 만들지 말고 제품을 사야 하는 시점은 언제라고 보나요?**
  → 외부 파트너 수·API 상품화 요구·감사/분석 요구가 커져서, 관리 기능을 직접 만드는 비용이 제품 도입 비용을 넘는 시점. "코어 비즈니스가 아닌 것은 사고, 차별화 지점만 만든다"는 기준 제시.

## Q6. Gateway가 SPOF가 되지 않게 하려면 어떻게 구성하겠습니까?

### 모범 답변

Gateway는 모든 트래픽의 단일 진입점이므로, 설계상 가장 위험한 SPOF 후보입니다. 세 계층으로 대응하겠습니다.

1. **인스턴스 다중화** - Gateway를 Stateless로 유지하고(상태는 Redis 등 외부화) 여러 인스턴스를 L4/L7 로드밸런서 뒤에 둡니다. 헬스체크로 비정상 인스턴스를 자동 제외하고, 배포는 롤링/블루그린으로 무중단 처리합니다. Gateway가 Stateless라야 이 구조가 성립하므로, 세션이나 로컬 캐시에 의존하는 설계를 피하는 것이 전제입니다.
2. **의존성 격리** - Gateway 자체보다 Gateway가 의존하는 것들(인증 키 조회, rate limit용 Redis, 서비스 디스커버리)이 장애 전파 경로가 되는 경우가 많습니다. JWKS 캐싱, Redis 장애 시 fallback 정책(fail-open/closed), Circuit Breaker로 특정 백엔드 서비스의 장애가 Gateway 스레드/커넥션을 고갈시키지 않도록 격리합니다. 사이드 프로젝트에서 Resilience4j의 Circuit Breaker와 Bulkhead로 이 패턴을 구성해봤습니다.
3. **과부하 자체로부터의 보호** - 트래픽 폭주 시 Gateway가 먼저 죽지 않도록 rate limit, 커넥션·타임아웃 상한을 설정하고, 초과 트래픽은 빠르게 거절(fail fast)해 뒤 서비스를 보호합니다. "Gateway는 죽지 않고 거절한다"가 원칙입니다.

그리고 이 모든 것의 전제로, 부하 테스트로 인스턴스당 한계치를 미리 파악하고 그 이전에 스케일아웃하는 용량 계획과 모니터링(지연, 에러율, 스레드/커넥션 지표)이 필요합니다.

### 예상 꼬리 질문

- **Q. 로드밸런서 자체가 SPOF 아닌가요?**
  → 맞음. 클라우드 관리형 LB(AWS ALB/NLB)는 내부적으로 다중화되어 있고, 온프레미스라면 LB 이중화(VRRP 등) + DNS 레벨 분산까지 고려. "어디까지 우리가 책임질 계층인가"를 구분해 답변.

- **Q. Gateway 배포 중 진행 중인 요청은 어떻게 되나요?**
  → Graceful shutdown: 새 요청 수신 중단 → 진행 중 요청 완료 대기(drain) → 종료. LB 헬스체크에서 먼저 제외 후 종료하는 순서가 중요.

## Q7. 외부 트래픽이 급증할 때 Gateway 레벨에서 할 수 있는 방어는 무엇이 있나요?

### 모범 답변

목표는 두 가지입니다. **Gateway 자신이 살아남는 것**, 그리고 **뒤의 서비스를 보호하는 것**. 계층별로:

1. **Rate Limiting** - 사용자/테넌트/API별 요청 제한으로 특정 주체의 폭주를 차단합니다. 정상 급증(이벤트)과 비정상(공격, 버그 클라이언트)을 구분하는 1차 수단입니다.
2. **Circuit Breaker** - 특정 백엔드 서비스가 느려지거나 죽으면 해당 경로를 빠르게 차단해 Gateway의 스레드·커넥션이 느린 서비스에 묶이는 것을 막습니다. 느린 서비스 하나가 Gateway 전체를 마비시키는 것이 전형적인 연쇄 장애 패턴입니다.
3. **타임아웃과 재시도 정책** - 백엔드 호출 타임아웃을 짧게 설정하고, 재시도는 멱등한 요청에만 제한적으로. 급증 상황에서 무분별한 재시도는 트래픽을 증폭시키는 자충수가 됩니다.
4. **부하 차단(Load Shedding)** - 한계에 근접하면 우선순위 낮은 트래픽부터 즉시 429/503으로 거절합니다. 모든 요청을 느리게 처리하는 것보다 일부를 빠르게 거절하는 것이 전체 시스템에 낫습니다.
5. **캐싱** - 변하지 않는 응답(설정, 공개 데이터)은 Gateway/CDN 레벨 캐시로 백엔드 도달 자체를 줄입니다.

사이드 프로젝트에서 Resilience4j로 Circuit Breaker, Bulkhead, RateLimiter를 조합해 "한 서비스의 장애가 전체로 번지지 않는" 구조를 직접 검증해봤고, 이 경험이 Gateway 방어 설계에 그대로 이어진다고 생각합니다.

### 예상 꼬리 질문

- **Q. Circuit Breaker의 상태 전이(Closed/Open/Half-Open)를 설명해 주세요.**
  → Closed(정상, 실패율 집계) → 임계 초과 시 Open(즉시 차단) → 대기 후 Half-Open(일부 요청만 통과시켜 회복 확인) → 성공 시 Closed 복귀. Half-Open의 존재 이유(회복 감지 + 폭주 방지)까지 말하면 좋음.

- **Q. 재시도가 위험한 이유를 구체적으로?**
  → retry storm: 장애로 느려진 서비스에 재시도가 몰려 부하가 배가 → 회복 불가. 지수 백오프 + 지터 + 재시도 예산(budget) 개념으로 답변.

## Q8. [이력서] 45초 걸리던 쿼리를 0.8초로 줄였다고 했는데, 실행 계획에서 무엇을 보고 무엇을 바꿨나요?

### 모범 답변 (뼈대 - [확인 필요] 부분을 실제 구현으로 채울 것)

대용량 영상 메타데이터를 다중 조건으로 검색하는 API였습니다. 데이터가 누적되면서 특정 조건 조합에서 45초가 걸리는 slow query가 발생했습니다.

**진단** - PostgreSQL의 EXPLAIN ANALYZE로 실행 계획을 확인했습니다. [확인 필요: 실제로 본 문제. 예시 - (a) 조건 컬럼에 인덱스가 없거나 함수/타입 변환 때문에 인덱스를 못 타고 Seq Scan 발생, (b) 인덱스는 탔지만 선택도가 낮아 대량의 히트 후 필터링, (c) 불필요한 조인/서브쿼리가 비용의 대부분 차지]

**개선 1: 쿼리 자체 정리** - 실행 계획상 비용이 큰 불필요한 쿼리/조인을 제거했습니다. [확인 필요: 무엇을 제거했는지 - 예: 화면에서 안 쓰는 컬럼의 조인, 중복 서브쿼리]

**개선 2: 인덱스 재설계** - 검색 조건의 사용 패턴을 분석해 복합 인덱스를 재설계했습니다. 이때 원칙은 등치(=) 조건 컬럼을 앞에, 범위 조건 컬럼을 뒤에 두고, 카디널리티가 높은 컬럼을 우선하는 것이었습니다. [확인 필요: 실제 인덱스 구성]

**개선 3: 커서 기반 페이징 전환** - offset 방식은 offset만큼 읽고 버리기 때문에 뒤 페이지로 갈수록 선형으로 느려집니다. 정렬 키(예: created_at + id) 기준 "마지막으로 본 값 이후(WHERE (created_at, id) < (?, ?))"를 조건으로 거는 커서 방식으로 바꿔, 페이지 깊이와 무관하게 인덱스 range scan 한 번으로 조회되도록 했습니다.

**결과** - 45초가 0.8초로 줄었고, 페이지 깊이에 따른 성능 저하가 사라졌습니다.

### 예상 꼬리 질문

- **Q. 커서 페이징의 단점은? 어떻게 감수했나요?**
  → 임의 페이지 점프(예: 7페이지로 바로) 불가, 정렬 기준 변경 시 커서 재설계 필요. 백오피스 특성상 무한 스크롤/다음 페이지 흐름이 주 사용 패턴이라 수용 가능했다는 식으로 요구사항과 연결해 답변.

- **Q. 커서의 정렬 키가 유니크하지 않으면 어떤 문제가 생기나요?**
  → 같은 값의 행이 페이지 경계에서 누락/중복됨. 그래서 유니크한 tie-breaker(id)를 정렬 키에 포함해 (created_at, id) 복합 커서로 구성.

- **Q. 인덱스를 늘리면 쓰기 성능이 떨어지는데 그 판단은?**
  → 이 테이블은 조회가 압도적인 읽기 중심 워크로드 + 쓰기는 배치 적재라 허용 가능. "무조건 추가"가 아니라 워크로드 특성으로 판단했음을 강조.

- **Q. 0.8초도 느린 편인데 더 줄인다면?**
  → 커버링 인덱스로 테이블 접근 제거, 자주 쓰는 조건 조합의 부분 인덱스, 통계성 조회라면 집계 테이블/머티리얼라이즈드 뷰 사전 계산, 캐싱. "다음 단계 아이디어"를 준비해두면 좋은 인상.

## Q9. [이력서] Blur의 포인트 중복 차감을 동시성 제어로 해결했다고 했는데, 어떤 방식이었고 다중 인스턴스에서도 유효한가요?

### 모범 답변 (뼈대 - [확인 필요] 부분을 실제 구현으로 채울 것)

포인트 결제 시 같은 사용자의 요청이 동시에 들어오면, 두 요청 모두 잔액 확인을 통과한 뒤 각각 차감하는 **check-then-act 레이스 컨디션**으로 중복 차감이 발생할 수 있는 구조였습니다.

[확인 필요: 실제 적용 방식. 아래 중 실제로 쓴 것을 중심으로 답변 구성]

**(a) 비관적 락으로 해결한 경우** - 포인트 레코드 조회 시 SELECT ... FOR UPDATE(JPA @Lock PESSIMISTIC_WRITE)로 행 락을 걸어, 한 트랜잭션이 잔액 확인~차감을 마칠 때까지 다른 요청이 대기하도록 했습니다. 결제는 충돌 시 재시도보다 정합성이 우선이고 동일 사용자의 동시 요청 빈도는 낮아, 락 대기 비용이 수용 가능하다고 판단했습니다.

**(b) 낙관적 락으로 해결한 경우** - 포인트 엔티티에 @Version을 두고, 커밋 시점 버전 충돌이 나면 실패시키거나 재시도했습니다. 락 점유가 없어 처리량에 유리하지만 충돌 시 재시도 설계가 필요했습니다.

**(c) 조건부 UPDATE로 해결한 경우** - UPDATE point SET balance = balance - ? WHERE user_id = ? AND balance >= ? 처럼 차감과 잔액 검증을 원자적 한 문장으로 처리하고, 영향받은 행 수가 0이면 잔액 부족으로 처리했습니다.

**다중 인스턴스 유효성** - 위 방식들은 모두 **DB가 동시성의 중재자**이므로, 애플리케이션 인스턴스가 몇 대든 유효합니다. 인스턴스 수와 무관하게 같은 DB의 같은 행을 두고 경합하기 때문입니다. 반대로 synchronized나 JVM 로컬 락은 인스턴스 하나 안에서만 유효해서 다중 인스턴스에선 무의미합니다.

다만 DB 락 경합이 병목이 되는 규모라면, 사이드 프로젝트에서 다뤄본 **Redisson 분산 락**으로 DB 앞단에서 직렬화하거나, 요청에 멱등키를 부여해 중복 요청 자체를 차단하는 방식으로 확장할 수 있다고 생각합니다.

### 예상 꼬리 질문

- **Q. 비관적 락과 낙관적 락의 선택 기준은?**
  → 충돌 빈도와 재시도 비용. 충돌이 잦고 실패가 비싸면(결제) 비관적, 충돌이 드물고 재시도가 싸면 낙관적. "정답"보다 판단 기준을 보여줄 것.

- **Q. 락을 잡은 트랜잭션이 외부 API(PG사 등)를 호출하면?**
  → 락 점유 시간이 외부 지연에 종속되어 위험. 외부 호출은 트랜잭션/락 범위 밖으로 분리하고, 상태 기반(결제 진행중 상태 기록)으로 재설계해야 함.

- **Q. 멱등키(idempotency key)는 어떻게 동작하나요?**
  → 클라이언트가 요청마다 고유 키를 부여, 서버는 키 단위로 처리 결과를 저장. 같은 키 재요청 시 저장된 결과를 반환해 이중 처리 원천 차단. 결제 API 표준 패턴(Stripe 등).

## Q10. [이력서] 테스트 커버리지 80%를 달성했다고 했는데, 인증/인가 로직은 어떻게 테스트했나요?

### 모범 답변

먼저 커버리지 수치에 대해 정확히 말씀드리면, 저희는 라인 커버리지 기준이었고 [확인 필요: 라인/브랜치 기준], 수치 자체가 품질을 보장하지 않는다는 것도 인지하고 있습니다. 80%는 목표라기보다 "리팩터링할 때 회귀를 잡아주는 안전망"으로서의 최소선이었습니다.

인증/인가 테스트는 계층별로 나눴습니다.

1. **단위 테스트** - 토큰 파싱/검증, 만료 판정 같은 순수 로직은 의존성 없이 단위 테스트로. 정상 토큰, 만료 토큰, 서명 위조 토큰, 형식 오류 토큰 같은 경계 케이스를 케이스별로 검증했습니다.
2. **인가 규칙 테스트** - Spring Security 환경에서 spring-security-test의 MockMvc 지원(mockUser, JWT 모킹 등)을 활용해, "이 역할로 이 엔드포인트에 접근하면 200/403"을 엔드포인트×역할 매트릭스로 검증했습니다. 인가 테스트의 핵심은 **성공 케이스보다 거부 케이스**라고 생각합니다. 권한 없는 접근이 403인지, 미인증이 401인지가 실제 보안 사고를 막는 테스트입니다.
3. **통합 테스트** - 소셜 로그인처럼 외부 IdP가 끼는 흐름은 외부 API를 모킹해 우리 쪽 처리(사용자 정보 변환, 가입/연동 분기)를 검증했습니다.

이 테스트들이 있어서 인증 로직을 리팩터링하거나 Spring Security 버전을 올릴 때 회귀를 빠르게 잡을 수 있었습니다.

### 예상 꼬리 질문

- **Q. 커버리지가 높아도 놓치는 것은?**
  → 검증(assertion) 없는 실행만 해도 커버리지는 오름. 동시성 버그, 설정 오류(프로파일별 시큐리티 설정), 시간 의존 로직(토큰 만료)은 커버리지로 안 잡힘. 만료 테스트는 Clock 주입으로 시간을 제어해 테스트했다는 식으로 확장 가능.

- **Q. MQTT/WebSocket 같은 비동기 연동은 어떻게 테스트했나요?**
  → [확인 필요: 실제 방식] 예시 답변 - 브로커 연동부는 인터페이스로 분리해 비즈니스 로직은 단위 테스트, 연동 자체는 테스트용 브로커(embedded/Testcontainers)로 통합 테스트. Awaitility 등으로 비동기 결과 대기.

## Q11. 로그인 API에 브루트포스 공격이 들어오면 어떤 계층에서 어떻게 막겠습니까?

### 모범 답변

단일 수단이 아니라 계층 방어로 답하겠습니다.

1. **네트워크/엣지 계층** - WAF, IP 기반 rate limit으로 명백한 대량 공격을 가장 바깥에서 차단합니다. 다만 공격자가 IP를 분산(봇넷)하면 IP 기반만으로는 부족합니다.
2. **Gateway 계층** - 로그인 엔드포인트에 IP별 + 계정별 이중 rate limit을 겁니다. IP 분산 공격에는 계정 기준 제한이, 계정 순회 공격(credential stuffing - 여러 계정에 유출 비번 대입)에는 IP 기준 제한이 각각 대응합니다.
3. **애플리케이션 계층** - 계정별 실패 횟수 추적: 임계 초과 시 점진적 지연(백오프)이나 CAPTCHA를 먼저 적용합니다. 곧바로 계정 잠금을 하면, 공격자가 일부러 실패를 반복해 남의 계정을 잠그는 **잠금 DoS**가 가능하므로, 잠금은 최후 수단 + 소유자에게 알림·해제 수단과 함께 설계합니다.
4. **탐지와 대응** - 실패율 급증, 특정 IP 대역의 이상 패턴을 모니터링 지표로 두고 알림을 겁니다. 유출 비밀번호 목록과의 대조(가입/변경 시)도 예방 수단입니다.

추가로, 로그인 응답에서 "아이디가 없음"과 "비밀번호 틀림"을 구분해주면 계정 존재 여부를 노출하게 되므로, 동일한 오류 메시지와 유사한 응답 시간을 유지하는 것도 함께 챙겨야 합니다.

### 예상 꼬리 질문

- **Q. 응답 시간까지 왜 신경 쓰나요?**
  → 존재하는 계정만 해시 검증을 수행하면 응답 시간 차이로 계정 존재가 유추됨(타이밍 공격). 존재하지 않는 계정도 더미 해시 연산을 수행하는 방어가 있음.

- **Q. CAPTCHA의 한계는?**
  → 해석 대행 서비스·AI로 우회 가능, UX 비용. 그래서 위험 신호가 있을 때만 조건부 노출(risk-based)하는 것이 일반적.

## Q12. 감사 로깅(Audit Log)을 설계한다면 무엇을 기록하고, 대량 로그를 어떻게 저장·조회하겠습니까?

### 모범 답변

**무엇을 기록하나** - 감사 로그의 목적은 "누가, 언제, 무엇을, 어떻게 했고, 결과가 무엇이었나"를 사후에 재구성하는 것입니다. 최소 필드는: 행위자(사용자 ID, 클라이언트/디바이스, IP), 시각, 행위(API, 메서드, 대상 리소스), 결과(성공/실패, 거부 사유), 요청 식별자(trace id). 인증 이벤트(로그인 성공/실패, 토큰 발급/폐기, 권한 변경)는 특히 필수입니다. 반대로 **넣으면 안 되는 것**은 비밀번호, 토큰 원문, 민감 개인정보입니다. 토큰은 jti 같은 식별자만 남기고 마스킹합니다.

**어떻게 저장하나** - 감사 로그는 쓰기가 압도적이고(모든 요청), 수정이 없고(append-only), 조회는 드물지만 조건 검색이 필요합니다. 요청 처리 경로에서 동기로 DB에 쓰면 본 트래픽에 부담을 주므로, **비동기 파이프라인**으로 분리하겠습니다: 애플리케이션은 로그 이벤트를 발행만 하고, Kafka 같은 버퍼를 거쳐 컨슈머가 저장소에 적재하는 구조입니다. 사이드 프로젝트에서 Kafka 기반 이벤트 파이프라인과 배치 적재(개별 insert를 배치 upsert로 전환해 lag 해소)를 직접 다뤄봤는데, 감사 로그 적재는 정확히 같은 패턴입니다.

**저장소와 보관** - 검색 조건(기간+행위자+행위)에 맞는 저장소를 선택합니다. 규모가 크면 Elasticsearch/OpenSearch 같은 검색 특화 저장소, 정형 조회 중심이면 시간 기준 파티셔닝한 RDB도 가능합니다. 기간별 수명 주기(핫: 검색 가능, 콜드: 객체 스토리지 아카이브)와 보존 기간 정책, 그리고 감사 로그 특성상 **변조 방지**(append-only 권한 분리, 접근 통제)까지가 설계 범위라고 생각합니다.

### 예상 꼬리 질문

- **Q. 비동기로 쓰면 로그 유실 가능성이 있는데 괜찮나요?**
  → 감사 로그의 요구 수준에 따라 다름. 규제 대상이면 at-least-once 보장(발행 실패 시 로컬 스풀, Kafka acks 설정)이 필요. Transactional Outbox 패턴으로 "본 트랜잭션과 로그 발행의 원자성"을 보장하는 방법도 사이드에서 구현해봄.

- **Q. trace id는 왜 필요한가요?**
  → MSA에서 하나의 사용자 행위가 여러 서비스 로그로 흩어짐. 동일 trace id로 연결해야 행위 단위 재구성 가능. Gateway에서 발급해 전 서비스로 전파.

## Q13. Kubernetes/Istio 실무 경험이 없다고 하셨는데, 서비스 메시에서 mTLS와 인가 정책이 어떻게 동작하는지 아는 대로 설명해 주세요.

### 모범 답변

실무 운영 경험은 없어서, 학습한 개념 수준에서 솔직하게 말씀드리겠습니다.

**서비스 메시의 기본 구조** - Istio 같은 서비스 메시는 각 서비스 파드 옆에 Envoy 프록시를 **사이드카**로 붙여, 서비스 간 모든 통신이 프록시를 거치게 합니다. 통신 제어(암호화, 인증, 라우팅, 관측)를 애플리케이션 코드에서 인프라 계층으로 내리는 것이 핵심 아이디어입니다.

**mTLS** - 일반 TLS는 클라이언트가 서버만 검증하지만, mTLS는 **양쪽이 서로의 인증서를 검증**합니다. 서비스 메시에서는 각 워크로드에 고유 인증서(SPIFFE 형식의 워크로드 신원)가 자동 발급·회전되고, 사이드카끼리 mTLS로 통신합니다. 효과는 두 가지입니다: 내부 트래픽 암호화, 그리고 "호출자가 어떤 서비스인지"에 대한 암호학적 신원 보장. 애플리케이션 코드는 인증서 관리를 전혀 몰라도 됩니다.

**인가 정책** - 그 신원을 바탕으로 "결제 서비스는 주문 서비스만 호출 가능" 같은 서비스 간 접근 제어를 선언적 정책(Istio의 AuthorizationPolicy)으로 적용할 수 있습니다. 사용자 수준 인가(JWT 검증)도 프록시 레벨에서 처리 가능합니다.

제 경험과 연결하면, 제가 Gateway 필터로 직접 구현했던 "검증·접근 제어를 코드로"의 상당 부분을 메시가 "인프라 정책으로" 대신하는 것이라고 이해하고 있습니다. 다만 Gateway/애플리케이션 레벨 인가(도메인 지식 필요)와 메시 레벨 인가(서비스 신원 기반)는 대체가 아니라 상호 보완이라고 생각하고, 실무에서 정책 운영·디버깅의 실제 난이도는 직접 겪어봐야 안다고 생각해 입사 후 빠르게 채우고 싶은 영역입니다.

### 예상 꼬리 질문

- **Q. K8s의 기본 개념(Pod, Service, Deployment)은 어디까지 아시나요?**
  → Pod(배포 최소 단위), Deployment(replica·롤링업데이트 관리), Service(파드 집합에 대한 안정적 접근점/로드밸런싱), Ingress(외부 노출), HPA(부하 기반 자동 확장) 수준으로 개념 정리해 답변. Docker 실무 경험에서 컨테이너 기초는 있음을 연결.

- **Q. 그럼 서비스 메시가 있으면 API Gateway는 필요 없나요?**
  → 아님. 메시는 east-west(내부 서비스 간), Gateway는 north-south(외부→내부) 트래픽 담당. 외부 인증, rate limit, API 상품화는 여전히 Gateway 영역.

## Q14. 새로운 Gateway 제품(예: Tyk)을 도입해야 한다면 PoC를 어떻게 설계하겠습니까?

### 모범 답변

PoC의 목적은 "좋은 제품인가"가 아니라 "**우리 요구사항과 제약에서 동작하는가**"를 검증하는 것이라고 생각합니다. 4단계로 설계하겠습니다.

1. **평가 기준 정의 (가장 중요)** - 도입 배경이 되는 핵심 요구사항을 먼저 문서화합니다. 예: 우리 인증 체계(JWT/OIDC)와의 통합 가능성, 파트너별 정책 분리, 커스텀 로직 확장 방식(플러그인), 목표 처리량과 지연 예산(p99 몇 ms 추가까지 허용), 운영 요건(모니터링 연동, 배포 방식, HA 구성). 기준 없이 시작하는 PoC는 "써보니 좋더라"로 끝납니다.
2. **핵심 시나리오 검증** - 전 기능이 아니라, 실패하면 도입이 무산되는 리스크 큰 시나리오부터 검증합니다. 예: 우리 토큰 검증을 커스텀 미들웨어로 구현 가능한가, 테넌트별 rate limit이 요구 수준으로 동작하는가. "가장 의심스러운 것부터"가 원칙입니다.
3. **정량 측정** - 동일 조건에서 부하 테스트로 처리량, 지연 분포(평균이 아닌 p95/p99), 자원 사용량을 기존 대비 측정합니다. 사이드 프로젝트에서 Prometheus/Grafana로 지표를 관찰하며 병목을 검증해본 방식 그대로입니다. 장애 시나리오(백엔드 다운, 설정 오적용)에서의 거동도 함께 봅니다.
4. **결과 공유와 의사결정** - 기준 대비 충족/미충족을 표로 정리하고, 미충족 항목의 우회 가능성과 비용, 운영 학습 곡선까지 포함해 "도입/보류/대안" 권고로 마무리합니다. PoC 결과를 팀이 재사용할 수 있게 문서와 재현 가능한 구성으로 남기는 것까지가 전파라고 생각합니다.

### 예상 꼬리 질문

- **Q. PoC에서 흔히 하는 실수는 뭐라고 생각하나요?**
  → 해피패스만 검증(장애·엣지 미검증), 프로덕션과 다른 환경에서 측정, 평가 기준을 나중에 정해 결론에 끼워 맞추기, 운영 비용(러닝커브, 온콜) 누락.

- **Q. 실제로 기술 검증을 해본 경험이 있나요?**
  → WebSocket→MQTT 전환 검토(LTE 비용 절감) 경험을 소재로: 전환 배경, 검증한 것, 판단 근거로 스토리 구성 가능. [본인 경험으로 구체화 권장]
