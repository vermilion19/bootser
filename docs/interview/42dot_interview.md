# 42dot Open Platform Engineering

이력서(Vueroid Cloud, Gateway 사이드 프로젝트)와 42dot 공고(API Gateway, IAM, OAuth2/OIDC, 트래픽 관리)를 대조해 뽑은 핵심 질문입니다. 각 문항은 모범 답변과 예상 꼬리 질문으로 구성되어 있으며, 꼬리 질문에는 답변 포인트를 함께 적었습니다.

## Q1. OAuth 2.0 Authorization Code Grant 흐름을 처음부터 끝까지 설명해 주세요.

### 모범 답변

Authorization Code Grant는 사용자 자격 증명을 클라이언트가 직접 다루지 않고, 인가 서버를 통해 토큰을 안전하게 발급받는 흐름입니다.

1. 사용자가 클라이언트(웹앱)에서 로그인을 시도하면, 클라이언트는 사용자를 인가 서버의 authorize 엔드포인트로 리다이렉트합니다. 이때 client_id, redirect_uri, scope, state를 함께 보냅니다.
2. 사용자가 인가 서버에서 직접 로그인하고 권한 동의를 하면, 인가 서버는 redirect_uri로 **Authorization Code**를 전달합니다.
3. 클라이언트의 백엔드가 이 코드와 client_secret을 가지고 인가 서버의 token 엔드포인트에 요청해 **Access Token(및 Refresh Token)**을 교환받습니다.
4. 이후 클라이언트는 Access Token으로 리소스 서버 API를 호출합니다.

핵심은 3단계입니다. 토큰 교환이 브라우저(프론트채널)가 아닌 서버 간 통신(백채널)에서 일어나기 때문에, Access Token이 브라우저 히스토리나 리퍼러에 노출되지 않습니다. 실무에서 Vueroid Cloud의 소셜 로그인을 구현할 때 이 흐름을 사용했고, state 파라미터로 CSRF를 방어했습니다.

### 예상 꼬리 질문

- **Q. Implicit Grant는 왜 이제 권장되지 않나요?**
  → 토큰이 URL fragment로 브라우저에 직접 전달되어 히스토리·리퍼러로 유출될 수 있고, 토큰 발급 주체 확인(client 인증)이 없기 때문. 현재는 SPA도 Authorization Code + PKCE가 표준.

- **Q. PKCE가 뭔가요? 왜 필요한가요?**
  → client_secret을 안전하게 보관할 수 없는 퍼블릭 클라이언트(SPA, 모바일)를 위한 확장. 클라이언트가 code_verifier를 만들고 그 해시(code_challenge)를 authorize 요청에 보낸 뒤, 토큰 교환 시 원본 verifier를 제출해 "코드를 가로챈 공격자"가 토큰으로 교환하지 못하게 막음.

- **Q. state 파라미터의 역할은?**
  → CSRF 방어. 클라이언트가 생성한 랜덤 값을 리다이렉트 후 돌려받아 검증함으로써, 공격자가 자신의 authorization code를 피해자 세션에 주입하는 것을 방지.

## Q2. OAuth 2.0과 OpenID Connect(OIDC)의 차이는 무엇인가요?

### 모범 답변

OAuth 2.0은 **인가(Authorization)** 프레임워크이고, OIDC는 그 위에 **인증(Authentication)** 계층을 얹은 표준입니다.

OAuth 2.0의 Access Token은 "이 클라이언트가 이 리소스에 접근해도 된다"는 권한 증명일 뿐, "지금 사용자가 누구인지"를 보장하지 않습니다. 토큰만 보고 사용자를 식별하는 것은 스펙상 정의되지 않은 사용법입니다.

OIDC는 여기에 **ID Token(JWT)**을 추가합니다. ID Token에는 sub(사용자 식별자), iss(발급자), aud(대상 클라이언트), 인증 시각 등 사용자 신원 클레임이 서명과 함께 담기고, 클라이언트는 서명 검증만으로 "인가 서버가 이 사용자를 인증했다"는 사실을 확인할 수 있습니다. 또한 UserInfo 엔드포인트, Discovery(.well-known), 표준 스코프(openid, profile, email) 같은 규격이 정의되어 있어 구현 간 상호운용성이 높습니다.

정리하면, "API 접근 권한"이 필요하면 OAuth 2.0, "로그인(사용자 신원 확인)"이 필요하면 OIDC입니다. 소셜 로그인은 사실상 OIDC의 사용 사례입니다.

### 예상 꼬리 질문

- **Q. OAuth만으로 로그인 기능을 구현하면 어떤 문제가 생기나요?**
  → Access Token은 대상(audience) 검증 개념이 약해, 다른 앱에 발급된 토큰을 가로채 재사용하는 token substitution 공격에 취약. ID Token은 aud 클레임으로 "이 클라이언트에게 발급된 인증 증명"임을 보장.

- **Q. ID Token과 Access Token의 용도 차이는?**
  → ID Token은 클라이언트가 소비(사용자 신원 확인)하고, Access Token은 리소스 서버가 소비(API 접근 허용 판단). ID Token을 API 호출에 쓰면 안 됨.

- **Q. Discovery 문서(.well-known/openid-configuration)에는 뭐가 있나요?**
  → authorize/token/userinfo/jwks 엔드포인트 URL, 지원 스코프·알고리즘 등 메타데이터. 클라이언트가 이걸 읽어 설정을 자동화.

## Q3. Gateway에서 JWT 검증 필터를 구현하셨는데, 무엇을 검증했고 서명 알고리즘은 무엇을 선택했나요?

### 모범 답변

사이드 프로젝트에서 Spring Cloud Gateway의 GlobalFilter로 JWT 인증 필터를 구현했습니다. 검증 항목은 네 가지였습니다.

1. **서명 검증** - 토큰이 위변조되지 않았는지
2. **exp(만료 시각)** - 만료된 토큰 거부
3. **iss(발급자)** - 우리 인증 서버가 발급한 토큰인지
4. **토큰 구조·타입** - Bearer 스킴, JWT 형식 자체의 유효성

알고리즘은 초기에 HS256(대칭키)으로 시작했지만, MSA 구조에서는 **RS256(비대칭키)이 적합**하다고 판단했습니다. HS256은 서명과 검증에 같은 비밀키를 쓰기 때문에 토큰을 검증해야 하는 모든 서비스에 비밀키를 배포해야 하고, 그중 한 곳만 유출돼도 누구나 토큰을 위조할 수 있습니다. RS256은 인증 서버만 개인키로 서명하고, Gateway와 각 서비스는 공개키로 검증만 하므로 유출 리스크가 격리됩니다. 공개키는 JWKS 엔드포인트로 배포하고 키 회전(kid 기반)도 가능합니다.

### 예상 꼬리 질문

- **Q. aud(audience) 클레임은 왜 검증하나요?**
  → 다른 서비스용으로 발급된 토큰이 우리 API에 재사용되는 것을 방지. 특히 여러 API·클라이언트가 있는 플랫폼 환경에서 필수.

- **Q. JWKS 엔드포인트를 매 요청마다 호출하면 어떻게 되나요?**
  → 인증 서버가 병목·SPOF가 됨. 공개키를 로컬 캐싱하고, kid가 캐시에 없을 때만 갱신하는 방식으로 처리. 캐시 TTL과 키 회전 주기를 함께 설계.

- **Q. alg 헤더를 신뢰하면 안 되는 이유는?**
  → alg: none 공격, RS256→HS256 다운그레이드 공격(공개키를 HMAC 비밀키로 쓰게 유도)이 존재. 서버는 허용 알고리즘을 화이트리스트로 고정해야 함.

- **Q. Gateway 필터에서 검증 실패 시 응답은 어떻게 처리했나요?**
  → 401 Unauthorized 반환, WWW-Authenticate 헤더 포함. 만료(401)와 권한 부족(403)을 구분.

## Q4. JWT는 발급 후 무효화가 어렵습니다. 강제 로그아웃이나 토큰 탈취 시 어떻게 대응하겠습니까?

### 모범 답변

JWT는 Stateless라는 장점이 곧 단점이 됩니다. 서버가 상태를 안 가지니 "이미 발급된 토큰"을 서버가 취소할 방법이 기본적으로 없습니다. 실무적으로는 세 가지를 조합합니다.

1. **Access Token TTL을 짧게(5~15분)** - 탈취돼도 피해 시간을 제한하는 것이 1차 방어선입니다. 긴 세션은 Refresh Token이 담당합니다.
2. **블랙리스트(Deny List)** - 강제 로그아웃·탈퇴·비밀번호 변경 시 해당 토큰의 jti(토큰 ID)를 Redis에 남은 TTL만큼 저장하고, Gateway 검증 시 조회합니다. 완전한 Stateless는 포기하지만, "전체 토큰 저장"이 아니라 "취소된 토큰만 저장"이므로 조회 비용이 작고 Redis 캐시로 충분히 감당됩니다.
3. **Refresh Token 서버 관리** - Refresh Token은 애초에 서버(DB/Redis)에 저장해 상태를 관리합니다. 강제 로그아웃 시 Refresh Token을 삭제하면, Access Token이 만료되는 최대 15분 뒤에는 세션이 완전히 종료됩니다.

즉 "즉시 차단이 꼭 필요한 이벤트"는 블랙리스트로, "일반적인 세션 종료"는 짧은 TTL + Refresh 삭제로 처리하는 이원 전략입니다.

### 예상 꼬리 질문

- **Q. 블랙리스트를 쓰면 결국 Stateful인데, 그럼 세션 방식과 뭐가 다른가요?**
  → 세션은 모든 요청이 저장소 조회 필수(정상 케이스도 조회). 블랙리스트는 서명 검증은 로컬에서 끝나고 예외 케이스만 조회하므로 저장소 의존도와 데이터 크기가 훨씬 작음. 장애 시 폴백 전략(블랙리스트 조회 실패 시 허용/거부)도 선택 가능.

- **Q. 비밀번호 변경 시 기존 토큰을 모두 무효화하려면?**
  → 사용자별 token_version(또는 pwd_changed_at)을 두고 토큰 클레임과 비교. 버전 불일치 시 거부. 사용자 단위 무효화에 유용.

- **Q. Redis가 죽으면 블랙리스트 조회가 안 되는데 어떻게 하나요?**
  → 보안 우선이면 fail-closed(거부), 가용성 우선이면 fail-open(허용) 선택의 문제. 서비스 성격에 따라 다르며, 저라면 로그인/결제 같은 민감 경로만 fail-closed로 분리.

## Q5. Access Token과 Refresh Token을 어디에 저장하고, Refresh Token 탈취에는 어떻게 대비하나요?

### 모범 답변

웹 클라이언트 기준으로 말씀드리면:

- **Access Token**: 메모리(JS 변수) 보관이 가장 안전합니다. localStorage는 XSS에 그대로 노출됩니다. 메모리 보관 시 새로고침하면 사라지는 문제는 Refresh로 재발급받아 해결합니다.
- **Refresh Token**: **HttpOnly + Secure + SameSite 쿠키**에 저장합니다. HttpOnly라 JS로 접근할 수 없어 XSS로 직접 탈취가 불가능하고, SameSite로 CSRF를 완화합니다. 경로를 재발급 엔드포인트(/auth/refresh)로 제한해 노출 면적을 줄입니다.

Refresh Token 탈취 대비는 **Rotation + 재사용 감지**가 핵심입니다.

1. **Rotation**: Refresh할 때마다 새 Refresh Token을 발급하고 이전 토큰은 즉시 폐기합니다. 토큰이 일회용이 됩니다.
2. **재사용 감지(Reuse Detection)**: 이미 폐기된(사용된) Refresh Token으로 재발급 요청이 오면, 탈취를 의심할 수 있는 강력한 신호입니다. 이때 해당 사용자의 토큰 패밀리 전체를 무효화하고 재로그인을 요구합니다. 정상 사용자든 공격자든 한쪽이 이미 사용한 토큰을 다른 쪽이 쓰는 순간 감지됩니다.

### 예상 꼬리 질문

- **Q. 모바일 앱이라면 저장 위치가 달라지나요?**
  → iOS Keychain / Android Keystore 같은 OS 보안 저장소 사용. 쿠키 개념이 없으므로 토큰을 직접 저장하되 OS 레벨 암호화에 위임.

- **Q. Rotation 중 네트워크 오류로 클라이언트가 새 토큰을 못 받으면?**
  → 정상 사용자가 구 토큰으로 재요청 → 재사용 감지에 걸려 로그아웃되는 문제. grace period(수 초간 직전 토큰 허용)를 두거나, 응답 유실 시 동일 토큰 1회 재시도를 허용하는 완충 설계 필요. 보안과 UX의 트레이드오프.

- **Q. SameSite=Strict과 Lax의 차이는?**
  → Strict은 외부 사이트발 요청에 쿠키 미전송(외부 링크로 진입 시 로그인 풀림 UX 문제), Lax는 top-level GET 내비게이션만 허용. 보통 Lax + CSRF 토큰 조합.

## Q6. RBAC과 ABAC의 차이를 설명하고, 파트너사마다 접근 가능한 API가 다른 B2B 플랫폼이라면 어떻게 설계하겠습니까?

### 모범 답변

**RBAC(Role-Based)**은 사용자에게 역할(Role)을 부여하고 역할에 권한(Permission)을 묶는 모델입니다. "ADMIN은 모든 API, PARTNER_VIEWER는 조회 API만" 같은 식으로 직관적이고 관리가 쉽지만, 조건이 세분화될수록 역할이 폭발(role explosion)하는 한계가 있습니다.

**ABAC(Attribute-Based)**은 사용자 속성, 리소스 속성, 환경 속성(시간, IP 등)을 조합한 정책으로 판단합니다. "파트너사 A 소속이면서, 자신의 조직 리소스에 대해, 업무 시간에만 허용" 같은 세밀한 제어가 가능하지만 정책이 복잡해지고 디버깅이 어렵습니다.

B2B 플랫폼이라면 **RBAC을 뼈대로, ABAC을 보완재로** 쓰는 하이브리드를 택하겠습니다.

1. 파트너사(테넌트) 단위로 구독 플랜/스코프를 정의 - 어떤 API 그룹에 접근 가능한지 (RBAC 성격)
2. 토큰에 tenant_id와 scope를 클레임으로 포함
3. **Gateway에서 거친 필터링** - 스코프 기반으로 "이 API 그룹에 접근 가능한가"를 차단 (호출 자체를 조기 차단해 내부 리소스 보호)
4. **각 서비스에서 세밀한 판단** - "이 리소스가 이 테넌트 소유인가" 같은 데이터 수준 인가는 도메인을 아는 서비스가 수행 (ABAC 성격)

인가를 Gateway 한 곳에 다 몰면 Gateway가 모든 도메인 지식을 알아야 해서 결합이 커지고, 서비스에만 두면 무의미한 트래픽이 내부까지 들어옵니다. 계층별 분담이 현실적인 답이라고 생각합니다.

### 예상 꼬리 질문

- **Q. 권한 정보를 토큰에 넣을 때의 문제는?**
  → 권한 변경이 토큰 만료까지 반영 안 됨(짧은 TTL로 완화), 권한이 많으면 토큰 비대화. 대안은 토큰엔 역할/스코프만 넣고 세부 권한은 서비스가 조회+캐싱.

- **Q. 정책을 코드에 하드코딩하지 않으려면?**
  → 정책 엔진 분리(예: OPA - Open Policy Agent). 정책을 선언적으로 관리하고 서비스는 평가 요청만. 감사·변경 이력 관리에도 유리. (실무 경험은 없지만 개념 학습 어필 포인트)

- **Q. 멀티테넌시에서 테넌트 간 데이터 격리는 어떻게 보장하나요?**
  → 모든 쿼리에 tenant_id 조건 강제(공통 필터/인터셉터 레벨), 토큰의 tenant_id와 요청 리소스의 소유 테넌트 대조. "클라이언트가 보낸 tenant_id"가 아닌 "토큰의 tenant_id"를 신뢰.

## Q7. API Gateway에 Rate Limiting을 구현한다면 어떤 알고리즘을 쓰고, 다중 인스턴스 환경에서는 어떻게 처리하겠습니까?

### 모범 답변

대표 알고리즘은 네 가지입니다. Fixed Window는 구현이 단순하지만 윈도우 경계에서 순간적으로 2배 트래픽이 통과하는 버스트 문제가 있습니다. Sliding Window는 이를 보완하지만 비용이 더 듭니다. **Token Bucket**은 평균 속도를 제한하면서 버킷 용량만큼의 순간 버스트를 허용해, API 트래픽처럼 자연스러운 스파이크가 있는 환경에 적합합니다. Leaky Bucket은 유출 속도를 고정해 균일한 처리율이 필요할 때 씁니다.

일반적인 API Gateway라면 **Token Bucket**을 선택하겠습니다. 정상 사용자의 짧은 버스트는 허용하면서 지속적인 과다 호출만 제한하는 특성이 UX와 보호를 모두 만족시키기 때문입니다.

다중 인스턴스 환경에서는 인스턴스별 로컬 카운터로는 전역 제한이 안 되므로, **Redis를 중앙 카운터 저장소**로 씁니다. 이때 "조회 후 증가" 방식은 레이스 컨디션이 생기므로, 토큰 계산과 차감을 **Lua 스크립트로 원자적으로** 수행해야 합니다. Spring Cloud Gateway의 RedisRateLimiter도 이 방식(Lua 기반 Token Bucket)으로 동작하며, 사이드 프로젝트에서 이 구조를 직접 구성해봤습니다. 키는 요구사항에 따라 사용자별/테넌트별/API별로 설계합니다.

### 예상 꼬리 질문

- **Q. Redis가 죽으면 Rate Limiter는 어떻게 동작해야 하나요?**
  → fail-open(전부 허용) vs fail-closed(전부 차단) 선택. 일반적으로 rate limit은 보호 장치이므로 fail-open + 알림이 무난하지만, 과금이 걸린 API라면 fail-closed 고려. 로컬 fallback limiter를 두는 절충안도 있음.

- **Q. 429 응답에는 어떤 헤더를 담아야 하나요?**
  → Retry-After, X-RateLimit-Limit / Remaining / Reset. 클라이언트가 백오프를 구현할 수 있게 하는 것이 플랫폼의 매너.

- **Q. 파트너별로 제한량이 다르면(플랜별 차등) 어떻게 설계하나요?**
  → 토큰/API Key에서 테넌트를 식별하고, 테넌트별 limit 설정을 조회(캐싱)해 버킷 파라미터를 동적으로 적용.

- **Q. Redis 왕복이 매 요청마다 발생하면 지연이 늘지 않나요?**
  → 맞음. 절충으로 로컬에서 대략 거르고 Redis로 정밀 제한하는 2단 구조, 또는 일정 단위로 토큰을 로컬에 배분(batch)하는 방식이 있음. 정확도와 지연의 트레이드오프.

## Q8. 인증/인가 시스템이 죽으면 전체 서비스가 마비됩니다. 고가용성 관점에서 어떻게 설계하겠습니까?

### 모범 답변

인증 시스템의 고가용성 설계 원칙은 "**핫패스(매 요청 경로)에서 인증 서버 의존을 최소화한다**"입니다.

1. **토큰 검증의 로컬화** - 매 요청마다 인증 서버에 유효성을 묻는 introspection 방식은 인증 서버를 전체 트래픽의 병목이자 SPOF로 만듭니다. JWT + 공개키(JWKS) 로컬 검증으로 전환하면, 핫패스에서 인증 서버 호출이 사라집니다. 인증 서버는 로그인/재발급 시에만 필요해져 장애 반경이 "신규 로그인 불가"로 축소되고, 이미 로그인한 사용자는 영향을 받지 않습니다.
2. **JWKS 캐싱** - 공개키는 각 서비스/Gateway가 캐싱하고, 키 회전(kid)에 대비해 백그라운드 갱신을 둡니다. 인증 서버가 잠시 죽어도 캐시된 키로 검증이 계속됩니다.
3. **인증 서버 자체의 다중화** - Stateless하게 구성해 다중 인스턴스 + 로드밸런서로 수평 확장하고, 세션성 데이터(Refresh Token 등)는 Redis/DB로 외부화합니다. 저장소도 replica 구성.
4. **장애 격리** - 로그인, 토큰 재발급, 관리 API를 분리해 한쪽 과부하가 전체로 번지지 않게 하고, Circuit Breaker로 하위 의존(DB, 외부 IdP) 장애의 전파를 차단합니다. 사이드 프로젝트에서 Resilience4j로 이 패턴을 적용해봤습니다.
5. **모니터링** - 토큰 발급 실패율, 검증 지연, JWKS 갱신 실패를 핵심 지표로 알림을 겁니다. Vueroid Cloud에서 CloudWatch 기반 알림을 구성해본 경험을 그대로 확장할 수 있는 영역입니다.

### 예상 꼬리 질문

- **Q. 로컬 검증의 단점은 없나요?**
  → 즉시 무효화가 어려움(Q4의 트레이드오프와 동일). 짧은 TTL + 블랙리스트로 보완. introspection과 로컬 검증은 "일관성 vs 가용성/성능"의 선택.

- **Q. 키 회전은 어떻게 무중단으로 하나요?**
  → 새 키를 JWKS에 먼저 추가(구키+신키 공존) → 새 토큰은 신키로 서명 → 구키로 서명된 토큰이 모두 만료된 후 구키 제거. kid로 어떤 키로 검증할지 식별.

- **Q. 외부 IdP(구글 로그인 등)가 죽으면?**
  → 신규 소셜 로그인만 실패. 기존 세션은 우리 토큰이므로 유지. 로그인 화면에서 해당 provider만 비활성화하는 graceful degradation.

## Q9. Gateway에서 인증을 통과시킨 후, 내부 서비스로 사용자 정보를 어떻게 전달하겠습니까?

### 모범 답변

일반적인 방식은 Gateway가 JWT를 검증한 뒤 사용자 ID, 테넌트, 역할 같은 필요한 정보를 **내부 헤더(예: X-User-Id)로 주입**해 전달하는 것입니다. 각 서비스가 토큰 파싱을 반복하지 않아도 되는 장점이 있습니다. 사이드 프로젝트의 Gateway 필터에서도 검증 후 클레임을 헤더로 변환해 내부 서비스로 넘기는 구조를 구현했습니다.

다만 이 방식의 치명적 위험은 **헤더 위조**입니다. 외부 클라이언트가 X-User-Id: admin을 직접 넣어 보내면? 그래서 반드시 함께 해야 하는 것이:

1. **Gateway에서 외부 유입 내부 헤더를 무조건 제거(strip) 후 재주입** - 신뢰의 시작점을 Gateway로 고정
2. **내부 서비스는 Gateway를 통해서만 접근 가능하도록 네트워크 격리** - 내부 서비스가 외부에 직접 노출되면 헤더 주입 방식 전체가 무너짐

더 견고한 대안으로는 (a) Gateway가 검증 후 원본 JWT를 그대로 전달하고 각 서비스도 서명을 재검증하는 방식(zero-trust에 가깝고, 내부 침투 시에도 방어), (b) Gateway가 내부용 토큰을 재서명해 전달하는 방식이 있습니다. 신뢰 모델을 어디까지 가져갈지에 따른 선택이며, 내부망을 신뢰할 수 있으면 헤더 방식, 못 믿으면 토큰 재검증 방식이 맞다고 생각합니다.

### 예상 꼬리 질문

- **Q. 서비스 간 호출(사용자 요청이 아닌)의 인증은 어떻게 하나요?**
  → 서비스 계정 개념 필요. OAuth2 Client Credentials Grant로 서비스용 토큰 발급, 또는 서비스 메시(Istio 등)의 mTLS로 전송 계층에서 상호 인증. (K8s/Istio는 실무 경험 없음을 전제로 개념 답변)

- **Q. 원본 JWT를 계속 전달하면 내부 로그에 토큰이 남을 위험은?**
  → 로깅 필터에서 Authorization 헤더 마스킹 필수. 감사 로그에는 토큰이 아닌 사용자 ID·jti만 기록.

- **Q. Gateway에서 인가(권한 판단)까지 다 하면 안 되나요?**
  → 스코프 수준의 거친 인가는 Gateway, 리소스 소유권 같은 도메인 인가는 서비스. Gateway에 도메인 지식이 쌓이면 결합·배포 병목 발생 (Q6과 연결).

## Q10. Vueroid Cloud에서 디바이스(블랙박스) 인증은 어떻게 처리했나요? 사용자 인증과 디바이스 인증은 무엇이 다른가요?

### 모범 답변

Vueroid Cloud에서는 두 종류의 주체를 인증해야 했습니다. **사람(사용자)**은 Spring Security 기반 로그인과 소셜 로그인(OAuth2)으로 처리했고, **디바이스(블랙박스)**는 MQTT 연결 시점에 디바이스 식별자와 자격 증명을 검증하는 방식으로 처리했습니다. WebSocket 실시간 채널 역시 연결 수립(핸드셰이크) 시점에 인증을 수행해 인증되지 않은 연결이 세션을 점유하지 못하게 했습니다.

이 경험에서 체감한 사용자 인증과 디바이스 인증의 차이는:

1. **상호작용 불가** - 디바이스는 아이디/비번을 입력하거나 리다이렉트를 탈 수 없으므로, 사전 발급된 자격 증명(디바이스 키 등) 기반의 비대화형 인증이 필요합니다. OAuth2로 치면 Client Credentials Grant에 해당하는 모델입니다.
2. **수명이 긴 연결** - 사용자 API는 요청 단위 인증이지만, MQTT/WebSocket은 한 번 인증 후 연결이 수 시간~수일 유지됩니다. 연결 후 자격이 취소된 디바이스를 어떻게 끊을지(연결 시 인증 + 주기적 재검증)가 별도 과제가 됩니다.
3. **대량·자동 재접속** - 디바이스는 네트워크 불안정으로 재접속이 폭주할 수 있어, 인증 시스템이 이 부하를 견뎌야 합니다. 실제로 MQTT 브로커 연결 장애를 디버깅하며 keep-alive와 재접속 처리를 다뤄봤습니다.

42dot의 Open Platform도 차량·디바이스·외부 파트너 등 이질적인 클라이언트의 인증을 다루는 팀으로 이해하고 있는데, "사람이 아닌 주체의 인증"을 실무에서 다뤄본 이 경험이 직접적으로 연결된다고 생각합니다.

### 예상 꼬리 질문

- **Q. 디바이스 자격 증명이 유출되면 어떻게 대응하나요?**
  → 디바이스 단위 자격 증명 폐기(revoke) 체계 필요. 전 디바이스 공용 키는 절대 금지(하나 유출 = 전체 유출). 디바이스별 고유 자격 증명 + 서버측 상태 관리로 개별 차단.

- **Q. 수만 대 디바이스가 동시에 재접속(reconnect storm)하면?**
  → 클라이언트 측 지수 백오프 + 지터(jitter)로 분산, 서버 측 연결 수락 rate limit. 인증 경로에 캐시를 둬 DB 부하 차단.

- **Q. WebSocket은 연결 후 토큰이 만료되면 어떻게 하나요?**
  → 정책 선택지: (a) 연결 유지하되 만료 시점에 서버가 종료 통지 후 재연결 유도, (b) 애플리케이션 레벨 재인증 메시지로 갱신. 연결 수명과 토큰 수명의 불일치는 실시간 시스템의 공통 과제.

- **Q. MQTT ACL은 어떻게 구성했나요?**
  → 디바이스별로 자신의 토픽에만 publish/subscribe 가능하도록 제한. 실무에서 ACL 설정 경로 문제를 직접 디버깅한 경험 있음(구체적 트러블슈팅 스토리로 전개 가능).

## 마무리 전략 요약

1. **깊게 파일 영역(확실히 준비)**: Q1~Q5 (OAuth2/OIDC/JWT). 이력서에 명시된 경험이라 반드시 꼬리물기가 들어옴.
2. **사이드 프로젝트로 되돌리기**: Tyk, K8s 등 공백 질문이 오면 "실무 경험은 없지만 사이드에서 Spring Cloud Gateway + Redis rate limit + Resilience4j를 직접 구현하며 개념을 검증했다"로 연결.
3. **실무 경험의 재해석**: MQTT 디바이스 인증, WebSocket 인증, CloudWatch 모니터링을 "플랫폼 인증·관측" 언어로 번역해 말하기 (Q10 참고).
4. **모르는 것 인정 + 판단 기준 제시**: "안 써봤습니다"로 끝내지 말고 "안 써봤지만, 도입한다면 이런 기준으로 검토하겠다"까지. 우대사항의 PoC 역량 어필과 연결됨.
