# 1. Vueroid Cloud (핵심 — 가장 깊게 파일 프로젝트)

## 1-1. MQTT 디바이스 연동

**Q. MQTT를 선택한 이유는? HTTP나 WebSocket 대비 장점은?**
- 뼈대: "저전력·불안정 네트워크의 디바이스에 맞는 경량 pub/sub 프로토콜 + LTE 비용 절감을 위해 WebSocket에서 전환"
- 전환 스토리(비용 검토 → 검증 → 전환)는 PoC 역량 어필 소재로도 재사용

**Q. QoS 레벨은 뭘 썼고 왜?**
- 뼈대: QoS 0(최대 1회)/1(최소 1회, 중복 가능)/2(정확히 1회, 비쌈) 트레이드오프 + 실제 선택 [확인 필요]
- 꼬리 대비: QoS 1이면 중복 수신 처리(멱등성)를 어떻게 했나

**Q. 디바이스 인증은 어떻게? 사용자 인증과 뭐가 다른가?** (모의 Q5 출제됨)
- 뼈대: "비대화형 주체 → 사전 발급 자격 증명(Client Credentials 모델) + 식별(시리얼)과 인증(비밀)의 분리 + 디바이스별 고유 자격 증명(공용 키 금지)"
- 교훈: 시리얼"만"으로 인증한다고 답하면 "시리얼 유출되면?"에 무너짐. 식별/인증 분리를 먼저 말할 것

**Q. 수만 대가 동시 재접속(reconnect storm)하면?**
- 뼈대: "클라이언트 지수 백오프+지터 / 서버 수락 rate limit / 인증 경로 캐싱(자격·상태 정보를 캐싱해 DB를 폭풍에서 격리)"
- TelemetryHub에서 RECONNECT_STORM 시나리오 실험한 경험 반드시 소환

**Q. MQTT 브로커 연결 장애를 디버깅했다는데 구체적으로?**
- 실무 스토리 그대로: keep-alive 타임아웃, ACL 설정 경로, Docker 볼륨 경로 불일치, 패킷 크기 제한 중 실제 겪은 것 1~2개를 증상→가설→확인→해결 구조로
- MQTT ACL 운영 경험은 "Kafka ACL·토픽 권한 설계와 같은 원리"로 확장 (모의 문답의 비동기 인가 논리)

## 1-2. WebSocket + Redis Pub/Sub (모의 Q18 출제됨 — 답변 완성본 있음)

**Q. 문제 상황과 Redis Pub/Sub 선택 이유는? Kafka는 왜 안 썼나?**
- 뼈대: "발신 서버가 수신자의 접속 서버를 모름 → 세션 위치 추적(상태 관리 추가) vs 브로드캐스트(단순) 중 서버 수가 적어 후자 선택. Kafka는 이 유스케이스(즉시 전달, 유실 허용)에 영속성·재처리가 무용 + 운영 비용. 반대로 유실 불허용인 사이드 프로젝트에선 Kafka — 같은 기준(보존 요구)에서 갈린 선택"
- 교훈: "broadcasting 해야만 했다"로 끝내지 말고 선택지 비교를 드러낼 것

**Q. Redis Pub/Sub은 전달 보장이 없는데 메시지 유실은 문제 안 됐나?**
- 뼈대: "실시간 상태 중계라 끊긴 클라이언트는 재접속 시 현재 상태를 새로 받으면 됨 — 유실 가능성이 이 요구사항에선 약점이 아님"

**Q. WebSocket 연결의 인증은? 연결 후 토큰이 만료되면?**
- 뼈대: "핸드셰이크(HTTP인 순간) 시점 인증 + 연결 수명과 토큰 수명의 불일치는 별도 과제(만료 시 종료 통지 후 재연결 유도 or 앱 레벨 재인증)"

## 1-3. 인증/인가 (Spring Security + OAuth2)

**Q. OAuth 2.0과 OIDC 차이? 본인 소셜 로그인은 뭘 쓴 건가?** (모의 Q6 — 처음에 못 답한 문제)
- 뼈대: "OAuth=인가(출입증), OIDC=OAuth+ID Token(신분증). ID Token 기반 신원 확인이므로 내 소셜 로그인은 OIDC"
- 앵커 3개: 출입증/신분증 · ID Token · aud가 token substitution 방어

**Q. ID Token 검증 시 뭘 확인하나?** (모의 Q7 — 못 답한 문제)
- 뼈대: "서명(위조)·exp(시효)·iss(출처)·aud(수신자) — 각각 빼먹으면 가짜 토큰/영구 재사용/가짜 발급자/token substitution"
- 암기: "안 고쳤나, 안 죽었나, 어디서 왔나, 나한테 온 거 맞나"

**Q. 자체 JWT에는 aud 검증 안 하던데?** (스스로 발견한 것 — 강점으로 전환)
- 뼈대: "발급자 1, 검증점 1, 대상 서비스 1인 구조라 오용 경로가 없어 불필요 판단. 여러 API/파트너 대상 발급이 시작되는 순간(=42dot 같은 플랫폼) aud가 필수가 됨"

**Q. 웹/모바일 인증 흐름이 왜 다른가? PKCE는 왜 없나?**
- 뼈대: "웹=백엔드가 secret으로 code 교환(Confidential Client, PKCE 필수 아님). 모바일=SDK가 ID Token 직접 발급, code 자체가 없어 PKCE 논외. PKCE는 secret 못 숨기는 클라이언트가 code를 직접 교환할 때 필요"

**Q. Access/Refresh 전략은?** (모의 Q8 — 합격점 받은 답)
- 뼈대: "Access 3분(유출 피해 최소화 우선, Refresh 빈도 증가는 규모상 수용) + Refresh로 재발급"
- 확정 꼬리: "Refresh 유출되면?" → Rotation + 재사용 감지(폐기 토큰 재사용 = 탈취 신호 → 패밀리 전체 무효화) + HttpOnly 쿠키. 이 3종 세트 미준비 시 Q8이 미끼가 됨

## 1-4. 배치 최적화 (모의 Q16 연계)

**Q. 대량 데이터를 영속성 컨텍스트로 다루면 무슨 문제?**
- 뼈대: "스냅샷 포함 2배 메모리 + 1차 캐시 무한 누적 → OOM / 더티체킹은 건별 UPDATE만 생성 → 쿼리 폭증"
- 해법: "벌크는 native로 통합(+컨텍스트 clear가 세트), 건별 처리는 페이징 청크 + flush/clear로 메모리 상한 고정"

**Q. N+1이 뭐고 어떻게 해결했나?** (모의 Q13 — 선 긋기 전략 필수)
- 뼈대: "내 사례는 JPA 지연 로딩발이 아니라 루프 내 건당 쿼리라는 구조 문제 — 넓은 의미의 N+1. [확인: 통합 방식]으로 쿼리 수를 상수화"
- 선 그은 뒤 확정 꼬리: "그럼 JPA N+1은?" → EAGER도 발생(JPQL은 직역이라 뒤늦게 건별 쿼리), 차이는 통제 가능성 → 전부 LAZY + 필요 시 fetch join/BatchSize
- 심화 꼬리: fetch join+페이징 = 행 뻥튀기로 LIMIT 불가 → 메모리 페이징(OOM 위험) → 페이징은 BatchSize(IN 묶음 로딩)와 조합 / MultipleBagFetch = 컬렉션 2개 조인 시 카테시안 곱으로 조립 불가 → "컬렉션 fetch join은 1개까지, 나머지 BatchSize"

# 2. 영상 DB 데이터 관리 플랫폼

**Q. 45초→0.8초, 실행 계획에서 뭘 봤고 뭘 바꿨나?**
- 뼈대: "EXPLAIN ANALYZE로 [확인: Seq Scan/비효율 조인] 확인 → 불필요 쿼리 제거 + 조건 패턴 기반 복합 인덱스 재설계(등치 앞, 범위 뒤) + 커서 페이징 전환"

**Q. 커서 페이징의 단점은? 왜 감수 가능했나?**
- 뼈대: "임의 페이지 점프 불가, 정렬 변경 시 재설계 — 백오피스 특성상 순차 탐색이 주 패턴이라 수용"
- 확정 꼬리: "커서 정렬 키가 유니크하지 않으면?" → 경계에서 누락/중복 → (created_at, id) 복합 커서로 tie-break

**Q. 0.8초도 느린데 더 줄인다면?**
- 뼈대: "커버링 인덱스(테이블 접근 제거) → 부분 인덱스 → 집계 테이블/캐싱 — 다음 단계 아이디어 제시"

# 3. Blur

**Q. 포인트 중복 차감을 어떻게 해결했나? 다중 인스턴스에서도 유효한가?**
- 뼈대: "check-then-act 레이스가 원인. [확인: 실제 방식]으로 확인~차감을 원자화. DB가 중재자인 방식(FOR UPDATE/버전/조건부 UPDATE)은 인스턴스 수와 무관하게 유효 — synchronized는 JVM 로컬이라 무의미"
- 확정 꼬리: "비관 vs 낙관 선택 기준?" → 충돌 빈도 × 실패 비용. 결제는 정합성 우선이라 비관/조건부가 자연스러움
- 심화 꼬리: "락 잡은 채 외부 API(PG) 호출하면?" → 락 점유가 외부 지연에 종속 — 외부 호출은 트랜잭션 밖으로, 상태 기반 재설계

**Q. @Transactional 동작 원리는?** (모의 Q11~12 — 프록시로 완성한 답)
- 뼈대: "프록시가 빈 대신 등록되어 pre(트랜잭션 시작)/post(커밋/롤백)를 감쌈 → 내부 호출(this)은 프록시 미경유라 REQUIRES_NEW 등 속성 무시 → 별도 빈 분리로 해결. REQUIRES_NEW는 커넥션 추가 점유라 대량 반복 시 풀 고갈 주의(배치 경험과 연결)"

# 4. 사이드 프로젝트 ① MSA 웨이팅 플랫폼 (SCG 집중 — 서류 합격 근거)

## SCG 3문장 뼈대 (모의 Q2 완성본 — 최우선 암기)
1. "GlobalFilter 구현 — 인증은 전 요청의 전제라 라우트별 누락 실수가 불가능한 deny-by-default + 공개 경로만 화이트리스트"
2. "pre에서 서명·만료 검증, 성공 시 클레임을 mutate로 내부 헤더 주입 후 chain.filter"
3. "실패 시 chain 미호출 + 401 완결 — 백엔드 전달 자체가 체인 끝의 필터(NettyRoutingFilter)라 체인이 끊기면 요청이 백엔드에 도달 불가"

**Q. 왜 SCG였나?** (모의 Q1)
- 뼈대: "게이트웨이는 커넥션을 물고 기다리는 I/O 집약 컴포넌트 → WebFlux 논블로킹이 구조적 적합 + 인증 로직을 자바로 일급 관리. '익숙해서'는 보조 이유로만"

**Q. 헤더 위조 방어는?** (모의 Q3 — 60점 받은 답)
- 뼈대: "두 겹 — ①인증 여부 무관하게 외부 유입 내부 헤더 무조건 strip 후 재주입(덮어쓰기만으론 공개 경로로 우회됨) ②내부 서비스 네트워크 격리. 이 전제가 불가한 환경이면 각 서비스 JWT 재검증(Zero Trust)"

**Q. 게이트웨이 SPOF는?** (모의 Q4)
- 뼈대: "3층 — ①Stateless(상태는 Redis 외부화) + LB 다중화 + graceful shutdown ②의존물 격리(Redis fail-open/closed 정책, 느린 백엔드는 타임아웃+CB) ③한계 초과 시 load shedding — 죽지 않고 거절"

**Q. HS256이었다면?**
- 뼈대: "검증 주체가 게이트웨이뿐이라 HS256로 시작했지만, 검증 주체가 늘면 검증 키=발급 키인 HS256은 위조 능력이 복제됨 → RS256+JWKS로 발급 권한을 인증 서버에 격리하는 게 맞음"
- JWKS = "발급자가 공개키를 kid와 함께 게시하는 표준 엔드포인트, 검증자는 캐싱하고 모르는 kid만 재조회"

**Q. Kafka 이벤트 기반에서 컨슈머의 권한 판단은?**
- 뼈대: "인가 시점의 이동 — 프로듀서가 발행 전 인가 완료 + 토픽 produce/consume은 브로커 ACL로 통제. 메시지 도달 자체가 검증 통과의 증거. 민감도별 토픽 분리가 곧 인가 설계, 처리 불가 메시지는 DLT로. MQTT ACL 운영 경험과 같은 원리"

**Q. Transactional Outbox를 왜 썼나?**
- 뼈대: "DB 커밋과 이벤트 발행의 원자성 — 커밋 후 발행 실패 시 유실을 막기 위해 트랜잭션 내 Outbox 저장 + 별도 폴링 발행으로 at-least-once"
- 꼬리: "그럼 중복 발행은?" → at-least-once의 대가, 컨슈머 멱등 처리와 세트

# 5. 사이드 프로젝트 ② TelemetryHub

**Q. MQTT와 Kafka를 왜 둘 다 썼나?**
- 뼈대: "역할 분담 — MQTT는 대량 디바이스 수집(경량, 디바이스 친화), Kafka는 버퍼링·재처리·팬아웃(영속, 컨슈머 속도 분리). Stream Processor가 느려져도 Kafka가 백프레셔 흡수"

**Q. 정확히 한 번 집계는 어떻게?**
- 뼈대: "eventId 기반 dedup을 Kafka Streams + 로컬 RocksDB로 in-process 처리 + 조건부 upsert(더 최신일 때만 갱신)로 재처리·실시간 쓰기 충돌 방어"

**Q. eventTime/ingestTime 이중 저장은 왜?**
- 뼈대: "디바이스 시계를 신뢰할 수 없음 — 집계는 eventTime, tie-break는 ingestTime, 차이로 네트워크 지연 모니터링"

# 6. 답변 태도 규칙 (모의 문답에서 확인된 약점 → 처방)

1. **첫 조각에서 멈추지 말 것** (Q3·Q4·Q9·Q15·Q20 공통): 답하기 전 속으로 "몇 겹이지? + 내 프로젝트에서 관련된 게 뭐지?" 두 가지 자문. 두 갈래 질문(Q20 유형)은 갈래 수부터 세기
2. **용어 0.5초 검증** (3회 발생: 리소스/주체, 지역변수/필드, CPU/OS 스레드): 핵심 명사를 입 밖에 내기 전 반대말과 대조. 틀린 용어 하나가 맞는 이해 전체를 의심받게 함
3. **결론보다 이유** (Q9 교훈): "게이트웨이는 못 한다"보다 "주체는 알지만 리소스를 모른다"를 기억 — 이유가 틀리면 결론도 철회당함
4. **선택의 순간을 드러내기** (Q18 교훈): "~했다" 대신 "A와 B 중 ~라서 A" — 시니어 면접은 선택의 이유를 듣는 자리
5. **모르면 쪼개기** (Q21 교훈): 얼어붙으면 소리 내어 "①누가 ②뭘 들고 ③누가 검증 ④제약은 어디서"로 분해 — 쪼개면 아는 문제가 됨
6. **보안 답변엔 우회로까지** (Q3 교훈): "이렇게 막는다" 뒤에 "이 방어가 뚫리는 경로와 그 대비"를 스스로 붙이기
7. **본인 사례가 교과서와 다르면 선 먼저 긋기** (Q13 교훈): "제 경우는 정확히는 ~였습니다"로 시작해 꼬리를 내 영토로 가져오기
8. **"주니어" 자기 규정 금지**: 5년 차 경력자의 정체성으로. "제가 잘은 모르지만" 화법 제거
