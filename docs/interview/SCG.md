# 1. SCG란 무엇인가 — 한 문단 정의부터

**면접용 정의 한 문단:**

> "Spring Cloud Gateway는 Spring 생태계의 API Gateway 프레임워크로, Spring WebFlux(Netty) 위에서 동작하는 논블로킹 리액티브 게이트웨이입니다. 요청을 Route로 매칭하고, 필터 체인으로 인증·rate limit·헤더 조작 같은 횡단 관심사를 처리한 뒤 백엔드로 프록시합니다. 저는 MSA 사이드 프로젝트에서 모든 외부 트래픽의 단일 진입점으로 SCG를 두고, JWT 인증 필터와 rate limiting을 직접 구현했습니다."

## 왜 Gateway가 필요했나 (내 프로젝트의 문제의식)

MSA로 서비스를 나누는 순간 세 가지 문제가 생깁니다:

1. **인증 중복** — Waiting, Restaurant, Promotion, Auth 각 서비스가 전부 JWT 검증 코드를 가져야 함. 검증 로직 변경 = 전 서비스 배포.
2. **클라이언트의 부담** — 클라이언트가 서비스별 주소를 다 알아야 하고, 서비스가 늘거나 위치가 바뀌면 클라이언트가 깨짐.
3. **횡단 관심사의 분산** — rate limit, 로깅, CORS를 서비스마다 따로 하면 정책 일관성이 무너짐.

→ "모든 요청이 지나가는 한 지점"에 이걸 모으는 것이 Gateway. 저는 이 진입점을 SCG로 구현했습니다.

## 왜 하필 SCG였나 (선택 이유 — 반드시 물어봄)

**답변 프레임: 대안과 비교해서 말하기**

- **vs Nginx/HAProxy**: 정적 라우팅·프록시는 훌륭하지만, JWT 검증처럼 우리 도메인 로직(자체 클레임 구조, 헤더 주입 규칙)을 넣으려면 Lua 등 별도 스택이 필요. 저는 인증 로직을 Java/Spring 코드로 다른 서비스와 같은 언어·같은 테스트 방식으로 관리하고 싶었습니다.
- **vs Netflix Zuul 1**: 서블릿 기반 블로킹 모델(요청당 스레드). Gateway는 대량의 동시 커넥션을 다루는 I/O 집약 컴포넌트라 논블로킹이 구조적으로 유리. Zuul 1은 유지보수 모드이기도 함.
- **vs Spring Cloud Netflix 스택 전반**: SCG가 Spring 공식 후속. Eureka·Spring Cloud LoadBalancer와의 통합이 자연스러움.
- **정리 멘트**: "Gateway는 커넥션을 많이 물고 있는 I/O 집약 컴포넌트라 논블로킹 모델이 적합하고, 인증 같은 커스텀 로직을 Spring 코드로 일급 관리할 수 있다는 점에서 SCG를 선택했습니다."

### 예상 꼬리 질문
- **Q. 그럼 Kong이나 Tyk 같은 제품형은 왜 안 썼나?**
  → "1인 사이드 프로젝트에서 목적이 '게이트웨이의 동작 원리를 코드 수준에서 이해하는 것'이었기 때문에 프레임워크가 맞았습니다. 반대로 외부 파트너 관리·포털·사용량 분석이 필요한 Open Platform이라면 제품형이 맞다고 생각합니다." (42dot 상황 인지를 드러내는 답)

# 2. SCG의 아키텍처 — WebFlux와 논블로킹

## 요청 하나의 여정

```
클라이언트 요청
   ↓
Netty 이벤트 루프가 수신 (요청당 스레드 아님!)
   ↓
Gateway Handler Mapping — Route의 Predicate들을 평가해 매칭되는 Route 선택
   ↓
필터 체인 실행 — GlobalFilter + 해당 Route의 GatewayFilter가 Order 순으로 하나의 체인으로 합쳐짐
   ↓ (pre 로직: chain.filter() 호출 "이전" 코드)
백엔드로 프록시 (논블로킹 HTTP 클라이언트)
   ↓ (post 로직: chain.filter().then(...) "이후" 코드)
응답 반환
```

## 핵심: 논블로킹이 왜 Gateway에 맞는가

- 전통 서블릿(MVC) 모델: 요청 1개 = 스레드 1개 점유. 백엔드 응답을 기다리는 동안 스레드가 놀면서 묶임. 동시 커넥션 수 = 스레드 수 한계.
- WebFlux(Netty) 모델: 소수의 이벤트 루프 스레드가 수만 커넥션을 처리. 백엔드 응답 대기 중에도 스레드는 다른 요청을 처리. I/O 대기가 대부분인 Gateway 워크로드에 정확히 부합.

## 치명적 함정: 필터 안의 블로킹 호출 (반드시 물어봄)

이벤트 루프 스레드는 소수(보통 CPU 코어 수)입니다. 필터 안에서 블로킹 호출(JDBC, RestTemplate, Thread.sleep)을 하면 그 스레드가 멈추고, **그 스레드가 처리하던 다른 모든 요청이 함께 밀립니다.** 요청 하나가 아니라 게이트웨이 전체 처리량이 무너지는 것. 이것이 WebFlux 최대 안티패턴.

- 대응: 외부 호출은 리액티브 클라이언트(WebClient, ReactiveRedisTemplate/Lettuce)로. 불가피한 블로킹은 별도 스케줄러(boundedElastic)로 격리.
- **내 프로젝트 연결**: rate limit의 Redis 호출도 Lettuce(Netty 기반 논블로킹)라 이벤트 루프를 막지 않음. 단, 네트워크 왕복 지연 자체는 존재 — "블로킹 여부"와 "지연"은 별개 개념이라는 것까지 말하면 상급.

### 예상 꼬리 질문
- **Q. WebFlux를 쓰면서 어려웠던 점은?**
  → 솔직 답변 소재: 블로킹 코드가 섞이지 않게 신경 써야 함, 스택트레이스가 리액티브 체인이라 디버깅이 어려움, ThreadLocal 기반 도구(MDC 로깅 등)가 그대로 안 됨. "어려움을 알고 썼다"가 포인트.
- **Q. 그럼 백엔드 서비스들도 다 WebFlux로 만들었나?**
  → 아니어도 됨. Gateway만 논블로킹이고 뒤 서비스는 MVC여도 무방. "각 컴포넌트의 워크로드 특성에 맞는 모델 선택" — Gateway는 I/O 집약이라 WebFlux, 도메인 서비스는 팀 생산성·생태계 고려해 MVC. 이 구분 자체가 좋은 답변.

# 3. Route / Predicate / Filter — 3대 구성 요소

## Route: "이 조건이면 저기로"

Route = Predicate(조건) + Filter 목록 + 대상 URI. 저는 [확인 필요: yml 설정 vs Java DSL 중 무엇으로 정의했는지] 방식으로 정의했습니다.

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: waiting-service
          uri: lb://WAITING-SERVICE     # lb:// = Eureka 디스커버리 + 로드밸런싱
          predicates:
            - Path=/api/waiting/**
```

- **lb:// 의 의미**: 고정 주소가 아니라 Eureka에 등록된 서비스 인스턴스 목록에서 Spring Cloud LoadBalancer가 인스턴스를 선택. 인스턴스가 늘거나 죽어도 Gateway 설정 변경이 없음 — Scale-out 설계의 연결 고리.

## Predicate: 매칭 조건

Path 외에도 Method, Header, Host, Query, 시간(After/Before) 등. AND 조합 가능. "라우팅 결정에 필요한 조건은 선언적으로, 로직은 필터로"가 역할 구분.

## Filter: pre / post 로직

**GatewayFilter vs GlobalFilter (내 선택의 핵심)**

| | GatewayFilter | GlobalFilter |
|---|---|---|
| 적용 범위 | 지정한 Route에만 | 모든 Route에 |
| 등록 방식 | Route 설정에 명시 | 빈으로 등록하면 자동 적용 |
| 적합한 용도 | 특정 경로 전용 처리 | 인증, 공통 로깅 등 전역 관심사 |

**나는 JWT 인증을 GlobalFilter로 구현했다. 이유:**

> "인증은 특정 라우트가 아니라 게이트웨이를 지나는 모든 요청의 전제 조건이기 때문입니다. GatewayFilter로 라우트마다 붙이면 새 라우트 추가 시 인증을 빠뜨리는 휴먼 에러가 가능한 구조가 됩니다. 기본은 전부 인증(GlobalFilter), 예외(로그인, 헬스체크 등 공개 경로)만 화이트리스트로 제외하는 deny-by-default 구조가 보안적으로 안전하다고 판단했습니다."

이 "deny-by-default" 한 마디가 보안 사고방식을 보여주는 킬러 멘트입니다.

**pre / post의 코드상 구분**

```java
@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    // ── 여기가 pre: chain.filter() 호출 전 ──
    // 토큰 추출, 검증, 실패 시 여기서 차단
    return chain.filter(exchange)          // 다음 필터/백엔드로 진행
        .then(Mono.fromRunnable(() -> {
            // ── 여기가 post: 응답이 돌아온 후 ──
            // 응답 헤더, 로깅 등
        }));
}
```

### 예상 꼬리 질문
- **Q. 필터 순서(Order)는 어떻게 정했고 왜 중요한가?**
  → 나는 Ordered 인터페이스로 순서를 명시 [확인 필요: 실제 order 값과 앞뒤에 둔 필터]. 순서가 중요한 이유의 예: rate limit을 인증보다 앞에 두면 비인증 폭주 트래픽으로부터 인증 로직(서명 검증 CPU 비용)을 보호할 수 있고, 반대로 인증을 앞에 두면 사용자 단위 rate limit이 가능. "무엇을 보호하고 무엇 기준으로 제한할 것인가"에 따른 설계 결정이라고 답하기.
- **Q. 특정 경로만 인증을 제외하려면 GlobalFilter에서 어떻게 했나?**
  → [확인 필요: 실제 방식 — 경로 화이트리스트를 필터 내부에서 검사했는지, 설정으로 뺐는지] 어느 쪽이든 "공개 경로 목록을 설정으로 분리해 코드 수정 없이 관리"가 좋은 방향.

# 4. 내 JWT 인증 필터 — 구현 서사 (면접의 본론)

## 답변 뼈대: "무엇을 → 어떻게 → 왜"

**구성 요약 멘트:**

> "GlobalFilter와 Ordered를 구현한 JwtAuthorizationFilter를 만들어 모든 요청의 pre 단계에서 JWT를 검증했습니다. Authorization 헤더에서 Bearer 토큰을 추출해 서명·만료를 검증하고, 실패 시 chain.filter를 호출하지 않고 401을 즉시 반환해 요청을 게이트웨이에서 차단했습니다. 검증에 성공하면 클레임에서 사용자 식별 정보를 꺼내 내부 헤더로 주입해 뒤 서비스로 전달했고, 이 필터는 @ConditionalOnProperty로 토글 가능하게 만들어 로컬 개발·테스트 환경에서는 끌 수 있게 했습니다."

## 검증 항목 (Q. 정확히 뭘 검증했나?)

1. Authorization 헤더 존재 + Bearer 스킴 형식
2. **서명 검증** — 위변조 확인 [확인 필요: HS256이었는지 RS256이었는지 — 이 답이 꼬리 질문의 방향을 결정함]
3. **exp 만료 확인**
4. [확인 필요: iss/aud 검증 여부 — 안 했다면 "개선 여지로 인지하고 있다"로 답변 준비]

**HS256이었다면 이렇게 방어:**
> "사이드 프로젝트에서는 인증 서버와 게이트웨이만 검증하는 구조라 HS256으로 시작했지만, 검증 주체가 늘어나는 실무 MSA라면 비밀키 배포 범위가 넓어져 유출 리스크가 커지므로 RS256 + JWKS로 가는 것이 맞다고 생각합니다. 개인키는 인증 서버에만 두고 나머지는 공개키 검증만 하는 구조로요."
→ 현재 구현의 한계와 다음 단계를 함께 말하면 "성장 가능한 사람"으로 보임.

## 실패 처리 (Q. 검증 실패하면 어떻게 되나?)

```java
// 개념 코드
if (!valid) {
    exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
    return exchange.getResponse().setComplete();  // 체인 진행 없이 즉시 응답 종료
}
```

- 포인트: **chain.filter()를 호출하지 않는 것** 자체가 차단. 백엔드에 요청이 닿기 전에 게이트웨이에서 끝냄 → 내부 서비스 보호.
- 401(인증 실패: 토큰 없음/만료/위조)과 403(인가 실패: 신원은 확인됐으나 권한 없음)의 구분을 말할 수 있어야 함.

## 클레임 전달 (Q. 뒤 서비스는 사용자를 어떻게 아나?)

검증 후 exchange.mutate()로 요청에 내부 헤더(X-User-Id 등)를 추가해 전달 [확인 필요: 실제 헤더 이름과 전달한 클레임].

**반드시 따라오는 꼬리: "그 헤더를 클라이언트가 위조해서 보내면?"**
> "두 가지가 전제되어야 합니다. 첫째, 게이트웨이가 외부에서 들어온 내부 헤더를 먼저 제거(strip)한 뒤 재주입해 신뢰의 시작점을 게이트웨이로 고정하고, 둘째, 내부 서비스는 게이트웨이를 통해서만 접근 가능하도록 네트워크를 격리해야 합니다. 내부 서비스가 외부에 직접 노출되면 헤더 주입 방식 전체가 무너집니다."
[확인 필요: strip을 실제로 했는지. 안 했다면 "구현 당시 인지하지 못했고, 지금은 이 위험을 알고 있다"고 솔직하게 + 위 개선안 제시]

## @ConditionalOnProperty 토글 (Q. 왜 필터를 켜고 끄게 만들었나?)

> "로컬 개발과 통합 테스트에서 매번 유효한 토큰을 준비하는 비용이 커서, 프로파일/설정으로 인증 필터를 비활성화할 수 있게 했습니다. 다만 운영 환경에서 실수로 꺼지는 것이 최악의 사고이므로, 기본값은 활성화(matchIfMissing 방향)로 두는 것이 안전하다는 것도 인지하고 있습니다." [확인 필요: 실제 기본값 설정]

### 이 섹션 전체의 예상 꼬리 질문 모음
- Q. 토큰 파싱/검증 라이브러리는 뭘 썼나? (jjwt / nimbus 등) [확인 필요]
- Q. 검증 로직의 성능 비용은? → 서명 검증은 CPU 연산(특히 RSA는 HMAC보다 비쌈). 그래서 rate limit을 앞단에 두는 설계와 연결 가능.
- Q. 게이트웨이에서 인가(권한)까지 검사했나? → 스코프/역할 수준의 거친 인가는 게이트웨이, 리소스 소유권 같은 도메인 인가는 각 서비스 — 계층 분담 원칙으로 답변.
- Q. 토큰이 큰 경우(클레임 많음) 문제는? → 매 요청 헤더에 실리므로 대역폭·헤더 크기 제한(431). 토큰엔 식별·스코프 최소만.

# 5. Rate Limiting 구현 (Redis 연동)

## 구성 멘트

> "RedisRateLimiter 기반으로 요청 제한을 적용했습니다 [확인 필요: 내장 RedisRateLimiter 사용인지 직접 구현인지]. 다중 인스턴스에서도 전역 제한이 되도록 카운터를 Redis에 중앙화했고, 조회-차감 사이의 레이스 컨디션은 Lua 스크립트의 원자 실행으로 해결되는 구조입니다. KeyResolver로 제한 기준 키를 정의했습니다 [확인 필요: IP 기준인지 사용자 기준인지]."

## 알아야 할 내부 동작

- SCG 내장 RedisRateLimiter = **Token Bucket**. 설정 파라미터: replenishRate(초당 충전 토큰 = 평균 허용률), burstCapacity(버킷 크기 = 순간 버스트 허용량).
- 왜 Token Bucket인가: 평균은 제한하되 정상적인 짧은 스파이크는 허용 — API UX에 적합.
- Lua 스크립트가 "현재 토큰 계산 + 차감 + TTL 설정"을 Redis 단일 스레드에서 원자적으로 실행 → 분산 환경 레이스 해결.
- KeyResolver: "누구 기준으로 셀 것인가"를 정의하는 함수형 인터페이스. 사용자 기준이면 인증 필터가 먼저 실행되어 사용자 정보가 있어야 함 → **필터 순서 설계와 연결됨** (인증 → rate limit 순).

### 예상 꼬리 질문
- Q. Redis가 죽으면 rate limiter는? → fail-open(허용+알림) vs fail-closed(차단). 보호 장치 성격상 보통 fail-open, 과금 API면 closed 고려. "경로별 위험도로 결정."
- Q. Redis 왕복이 매 요청 추가되는데 괜찮나? → Lettuce 논블로킹이라 스레드는 안 막힘. 왕복 지연(1ms 미만)은 존재하나 수용 가능 수준. 초고트래픽이면 로컬 근사 + 중앙 정밀 2단, 토큰 배치 선할당으로 왕복 빈도 절감 — 정확도 vs 지연 트레이드오프.

# 6. Eureka 연동과 전체 아키텍처에서의 위치

## 구성 멘트

> "서비스 디스커버리는 Eureka를 사용해, 게이트웨이 라우트의 uri를 lb://서비스명으로 선언했습니다. 각 서비스 인스턴스가 Eureka에 자기를 등록하면 게이트웨이는 서비스명만으로 살아있는 인스턴스를 찾아 로드밸런싱합니다. 인스턴스를 늘리는 것만으로 처리량이 늘어나는 Stateless Scale-out 설계의 한 축입니다."

## 왜 이 구조인가

- 고정 IP/포트 라우팅은 인스턴스 증설·교체 때마다 게이트웨이 설정 변경 필요 → 디스커버리로 위치 투명성 확보.
- 게이트웨이 자신도 Stateless(상태는 Redis 외부화)라 다중 인스턴스 가능 — "게이트웨이가 SPOF"라는 지적에 대한 방어.

### 예상 꼬리 질문
- Q. Eureka가 죽으면? → 클라이언트(게이트웨이)는 마지막으로 받은 레지스트리 캐시로 당분간 동작. 신규 인스턴스 반영만 지연. "디스커버리 장애 ≠ 즉시 전면 장애"를 아는 것이 포인트.
- Q. K8s 환경이라면 Eureka가 필요한가? → K8s의 Service/DNS가 디스커버리를 대체. "플랫폼이 제공하면 애플리케이션 레벨 디스커버리는 걷어낼 수 있다" — K8s 개념 이해를 자연스럽게 어필할 기회.

# 7. 종합: "왜 그렇게 만들었나"를 관통하는 설계 원칙 3개

면접 마지막에 "설계에서 중요하게 생각한 것은?"이 오면 이 3개로 수렴시키기:

1. **단일 책임의 진입점** — 인증·제한·라우팅 같은 횡단 관심사를 게이트웨이로 모아 각 서비스는 도메인에만 집중. 단, 도메인 인가는 서비스에 남겨 게이트웨이 비대화를 피함 (계층 분담).
2. **deny-by-default 보안** — 전역 인증(GlobalFilter) + 명시적 화이트리스트. 새 라우트가 생겨도 기본이 "잠김".
3. **워크로드에 맞는 모델 + 상태 외부화** — I/O 집약 게이트웨이는 논블로킹(WebFlux), 전역 상태(rate limit)는 Redis로 외부화해 Stateless 유지 → 수평 확장과 게이트웨이 자체의 SPOF 제거.

# 9. 3분 스토리 — 통째로 말하는 연습용

> "MSA를 구성하면서 모든 외부 트래픽의 단일 진입점으로 Spring Cloud Gateway를 뒀습니다. 선택 이유는 두 가지였는데, 게이트웨이는 다수의 커넥션을 물고 백엔드 응답을 기다리는 I/O 집약 컴포넌트라 WebFlux 기반 논블로킹 모델이 구조적으로 맞았고, JWT 검증 같은 커스텀 로직을 Spring 코드로 다른 서비스와 동일하게 테스트·관리할 수 있었기 때문입니다.
>
> 인증은 GlobalFilter로 구현했습니다. 인증은 특정 라우트가 아닌 모든 요청의 전제 조건이라, 라우트별로 붙이다 빠뜨리는 사고가 불가능한 deny-by-default 구조를 원했습니다. pre 단계에서 토큰의 서명과 만료를 검증하고, 실패 시 체인을 진행하지 않고 401로 즉시 차단해 내부 서비스에 요청이 닿지 않게 했습니다. 성공 시 클레임을 내부 헤더로 주입해 뒤 서비스가 토큰을 다시 파싱하지 않도록 했고요.
>
> 다중 인스턴스 환경의 전역 rate limit은 Redis 카운터로 중앙화했고, WebFlux에서 블로킹 호출이 이벤트 루프를 막는 안티패턴을 피하기 위해 Redis 연동도 논블로킹(Lettuce) 경로로 구성했습니다. 라우팅은 Eureka 디스커버리와 lb:// URI로 인스턴스 증설에 설정 변경이 없는 구조를 만들었습니다.
>
> 한계도 알고 있습니다. 사이드 프로젝트 규모라 [HS256을 썼지만 검증 주체가 늘면 RS256+JWKS가 맞고 / iss·aud 검증, 헤더 strip 같은 부분은 개선 여지가 있으며 — 확인 후 선택], 실무 플랫폼이라면 키 회전, 감사 로깅, 파트너별 정책 같은 운영 요소가 본질이 된다는 것을 이번 준비 과정에서 더 구체적으로 이해하게 됐습니다."

마지막 문단이 중요합니다. 한계를 먼저 말하는 지원자는 방어할 필요가 없어집니다.
