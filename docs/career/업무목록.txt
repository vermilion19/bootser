1. Vueroid Cloud
══════════════════════════════════════════════════════
주요 역할: 백엔드 API 개발 및 실시간 통신 서버 구축
기술 스택: Java, Spring Boot, Redis, MQTT, WebSocket

────────────────────────────────────────────────────
▶ WebSocket 서버 다중화로 인한 세션 불일치 문제 해결
────────────────────────────────────────────────────
[문제]
서비스 트래픽 증가에 따라 WebSocket 서버를 수평 확장(Scale-out)하는 과정에서
클라이언트 A가 서버1에, 클라이언트 B가 서버2에 연결된 경우 두 클라이언트 간
메시지 전달이 불가능한 세션 불일치(Session Mismatch) 문제 발생.
각 서버 인스턴스가 자신의 메모리에 세션을 독립적으로 관리하기 때문에 발생하는
구조적 한계였음.

[해결]
Redis Pub/Sub을 서버 간 메시지 브로커로 도입.
클라이언트로부터 메시지를 수신한 서버가 Redis 채널에 Publish하면,
모든 서버 인스턴스가 해당 채널을 Subscribe하여 연결된 클라이언트에 전달.
특정 서버에 세션이 묶이지 않는 Stateless 구조로 전환.

[결과]
서버 대수에 관계없이 전체 클라이언트 간 실시간 메시지 전달이 가능한
수평 확장 가능한 WebSocket 아키텍처 확보.

[면접 예상 질문]
Q: Redis Pub/Sub은 메시지 유실 가능성이 있는데, Kafka가 아닌 Redis를 선택한 이유는?
A: 해당 기능은 채팅처럼 실시간성이 중요하고 일부 유실이 허용되는 알림성 메시지였음.
   Kafka 도입 시 Broker 운영 복잡도가 증가하고 라이브러리 의존성도 늘어나는 반면,
   Redis는 이미 세션 저장소로 인프라에 포함되어 있어 추가 비용 없이 활용 가능했음.
   메시지 보장이 필요한 비즈니스 이벤트라면 Kafka를 선택했을 것.

────────────────────────────────────────────────────
▶ IoT 디바이스 통신 프로토콜 최적화
────────────────────────────────────────────────────
[문제]
HTTP 기반 통신으로 IoT 디바이스와 서버 간 데이터를 주고받을 때,
매 요청마다 헤더 오버헤드가 크고 TCP 연결 수립 비용이 높아
배터리 소모와 저대역폭 환경에서의 통신 불안정 문제가 발생.

[해결]
MQTT(Message Queuing Telemetry Transport) 프로토콜 도입.
Publish/Subscribe 모델 기반으로 수십 바이트 수준의 경량 메시지 송수신 구현.
지속 연결(Persistent Session) 방식으로 불필요한 연결 재수립 비용 제거.

[결과]
디바이스 배터리 소모 감소 및 저대역폭·불안정 네트워크 환경에서도
안정적인 통신 환경 확보.

────────────────────────────────────────────────────
▶ 대용량 데이터 배치 처리 파이프라인 구축
────────────────────────────────────────────────────
[문제]
대량의 이력 데이터를 주기적으로 집계하는 로직이 단순 반복 처리(for loop + JPA save)
방식으로 구현되어 있어 처리 시간이 길고 운영 시간대 DB 부하가 집중되는 문제 발생.

[해결]
Spring Batch를 활용하여 청크(Chunk) 단위 처리 및 Job 스케줄링 도입.
업무 시간 외 시간대에 배치를 실행하여 운영 서비스 부하 분산.
Step 단위 재시작 지원으로 중간 실패 시 전체 재처리 없이 복구 가능하도록 구현.

[결과]
대용량 데이터 처리를 서비스 운영에 영향 없이 안정적으로 수행하는
재시작 가능한 배치 파이프라인 구축.

2. Blur
══════════════════════════════════════════════════════
주요 역할: 결제 및 포인트 시스템 코어 로직 개발
기술 스택: Java, Spring Boot, PostgreSQL

────────────────────────────────────────────────────
▶ 포인트 동시 차감 시 데이터 정합성 문제(Race Condition) 해결
────────────────────────────────────────────────────
[문제]
동일 유저의 포인트 차감 요청이 동시에 여러 건 발생하는 상황에서,
"잔액 조회 → 계산 → 업데이트" 과정이 원자적으로 처리되지 않아
Lost Update(갱신 손실) 문제가 발생. 포인트 잔액이 실제보다 더 많이 남거나
마이너스가 되는 데이터 정합성 오류 확인.

[해결]
DB의 SELECT ... FOR UPDATE 구문을 사용한 비관적 락(Pessimistic Lock) 적용.
포인트 Row에 대한 동시 접근을 직렬화하여 항상 최신 잔액 기준으로 차감 수행.
- 낙관적 락(Optimistic Lock) 미선택 이유: 금융성 데이터 특성상 충돌 발생 시
  재시도(Retry) 방식보다 선점(Lock) 방식이 적합하며, 동일 유저의 포인트 요청은
  충돌 빈도가 높아 재시도 오버헤드가 더 크다고 판단.

[결과]
동시 요청 환경에서도 포인트 잔액의 정합성을 보장하는 안전한 차감 로직 구현.

[면접 예상 질문]
Q: FOR UPDATE를 쓰면 데드락이나 성능 저하 우려가 있는데, 낙관적 락을 고려하지 않았나?
A: 포인트/금전 데이터는 정합성이 최우선이고, 한 유저에게 동시 요청이 몰리는 패턴에서
   낙관적 락은 충돌 → 재시도 → 재충돌을 반복할 수 있어 오히려 처리량이 떨어짐.
   비관적 락으로 확실하게 순차 처리하는 것이 안전하고 예측 가능하다고 판단했음.
   데드락 방지를 위해 락 획득 순서를 일관되게 유지하였음.

────────────────────────────────────────────────────
▶ 비즈니스 로직 처리 중 락 유지 시간 최소화
────────────────────────────────────────────────────
[문제]
"무료 포인트 우선 차감" 정책 구현을 위해 애플리케이션에서 데이터를 조회 후
무료/유료 분기 계산을 하고 다시 업데이트하는 방식을 사용했는데,
락을 잡고 있는 동안 애플리케이션 연산 시간이 포함되어 락 유지 시간이 길어지고
처리량(Throughput) 저하 가능성 발생.

[해결]
CTE(Common Table Expression)와 CASE 문을 활용하여 무료/유료 포인트 분기 및
차감 로직을 단일 SQL 쿼리 내에서 처리.
애플리케이션과 DB 간의 왕복(Round-trip)을 줄이고 DB 내에서 원자적으로 처리.

[결과]
락 유지 시간 단축으로 처리량 개선.
비즈니스 로직(무료 포인트 우선 차감)과 성능 요구사항(빠른 락 해제)을 동시에 충족.

[면접 예상 질문]
Q: 비즈니스 로직을 DB 쿼리에 넣으면 유지보수가 어렵지 않나?
A: 일반적으로는 애플리케이션 레이어가 맞지만, 이 경우는 락을 잡고 있는 시간을
   최소화하는 것이 성능에 직접적인 영향을 미치는 상황이었음.
   로직의 복잡도가 낮고 변경 빈도도 낮은 정책이라 SQL 내 처리가 적합하다고 판단.
   정책이 자주 바뀐다면 애플리케이션으로 올렸을 것.

3. ADAS Dataset
══════════════════════════════════════════════════════
주요 역할: 대용량 데이터 관리 API 개발
기술 스택: Java, Spring Boot, QueryDSL, PostgreSQL

────────────────────────────────────────────────────
▶ 대용량 테이블 페이징 성능 저하 문제 해결
────────────────────────────────────────────────────
[문제]
수백만 건 이상의 데이터셋 목록 조회 API에서 OFFSET 기반 페이징 사용 시,
페이지가 뒤로 갈수록 OFFSET 값이 커져 DB가 해당 오프셋까지 모든 Row를
스캔하는 Full Scan이 발생. 뒤 페이지로 갈수록 응답 시간이 선형으로 증가하는
성능 저하 문제 확인.

[해결]
No-Offset(커서 기반) 페이징 방식으로 전환.
마지막으로 조회한 Row의 기준값(커서)을 WHERE 조건에 사용하여
인덱스 Range Scan으로 필요한 데이터만 조회.

[결과]
페이지 번호에 관계없이 일정한 조회 성능 유지.
마지막 페이지 기준으로 OFFSET 방식 대비 응답 속도 개선.

────────────────────────────────────────────────────
▶ 커서 기반 페이징의 데이터 누락 엣지 케이스 해결
────────────────────────────────────────────────────
[문제]
정렬 기준 컬럼(created_at)이 동일한 데이터가 여러 건 존재하는 경우,
단일 커서 값만으로 페이징 시 해당 시간대의 일부 데이터가 누락되거나
중복 조회되는 엣지 케이스 발견. 실무에서 배치 인서트나 동시 등록 시 빈번히 발생.

[해결]
생성일시(created_at)와 PK(id)를 묶은 복합 커서(Composite Cursor) 전략 적용.
  - 정렬: ORDER BY created_at DESC, id DESC
  - 조건: WHERE (created_at < ?) OR (created_at = ? AND id < ?)
PK는 항상 고유하므로 어떤 경우에도 커서 위치가 유일하게 결정됨.

[결과]
정렬 기준 값이 완전히 동일한 데이터가 존재하더라도 누락 및 중복 없는 페이징 구현.

[면접 예상 질문]
Q: 정렬 기준(created_at)이 중복될 때 커서를 어떻게 설계하셨나요?
A: created_at 단독 커서는 동일 시간대 데이터에서 누락이 발생할 수 있어,
   PK를 2차 정렬 기준으로 추가한 복합 커서를 사용했음.
   ORDER BY created_at DESC, id DESC로 정렬 후
   WHERE (created_at < ?) OR (created_at = ? AND id < ?) 조건으로
   항상 고유한 커서 위치를 보장.

────────────────────────────────────────────────────
▶ 동적 검색 조건 처리 및 쿼리 안전성 확보
────────────────────────────────────────────────────
[문제]
날씨, 시간대, 도로 환경 등 다양한 필터 조건을 조합하는 검색 기능에서,
if문으로 쿼리 문자열을 직접 이어붙이는 방식은 조건 추가마다 분기가 폭발적으로
늘어나고 SQL Injection 위험도 존재하는 구조적 문제 확인.

[해결]
QueryDSL 도입. BooleanBuilder를 활용한 타입 안전한 동적 쿼리 빌더 패턴으로
각 조건을 독립적인 메서드로 분리하여 조합.

[결과]
조건 추가/수정 시 독립적으로 변경 가능한 유지보수성 높은 쿼리 레이어 구축.
컴파일 시점에 쿼리 오류 감지 가능.

4. Taxi-BSD
══════════════════════════════════════════════════════
주요 역할: 서비스 아키텍처 설계 및 API 개발

────────────────────────────────────────────────────
▶ 상태 전이 기반 데이터 모델링
────────────────────────────────────────────────────
[문제]
택시 배차 서비스의 다양한 상태(배차 요청 → 수락 → 운행 중 → 완료/취소)를
단일 테이블에 플래그 컬럼으로 관리하면 상태가 늘어날수록 null 컬럼이 많아지고
이력 추적이 불가능한 구조.

[해결]
비즈니스 도메인의 상태 전이 흐름을 명확히 정의하고,
현재 상태 테이블과 이력 테이블을 분리한 정규화된 스키마 설계.
상태 변경 시 이력 레코드를 append 방식으로 누적하여 감사(Audit) 추적 가능.

[결과]
요구사항 변경 및 신규 상태 추가에 유연하게 대응 가능한 데이터 모델 확보.

────────────────────────────────────────────────────
▶ RESTful API 설계 및 구현
────────────────────────────────────────────────────
클라이언트 요구사항을 분석하여 자원(Resource) 중심의 URI 설계와
HTTP 메서드(GET/POST/PUT/DELETE)를 명확하게 구분한 RESTful API 명세 설계 및 구현.

5. Vueroid-Line
══════════════════════════════════════════════════════
주요 역할: 서비스 기능 구현 및 유지보수

────────────────────────────────────────────────────
▶ 레거시 코드 레이어 분리 리팩토링
────────────────────────────────────────────────────
[문제]
기능이 추가되며 비즈니스 로직이 Controller 레이어에 직접 혼재된 구조.
로직 수정 시 변경 범위가 불명확하고 단위 테스트 작성이 불가능한 상태.

[해결]
역할에 따라 Controller(요청/응답 변환) → Service(비즈니스 로직) →
Repository(데이터 접근)로 레이어를 명확히 분리.
반복 코드는 공통 유틸로 추출하고, 각 레이어가 단일 책임을 가지도록 리팩토링.

[결과]
단위 테스트 작성 가능한 구조로 개선.
신규 기능 추가 및 버그 수정 시 변경 범위 명확화로 사이드 이펙트 감소.

────────────────────────────────────────────────────
비즈니스 요구사항에 기반한 REST API 개발 및 기능 유지보수 수행.


══════════════════════════════════════════════════════
[보완 메모] - 수치 추가 필요 (기억나는 경우 채워넣기)
══════════════════════════════════════════════════════
- ADAS Dataset: 페이징 개선 전/후 응답 시간 (예: 마지막 페이지 기준 X초 → Y초)
- Vueroid Cloud: WebSocket 동시 연결 수, 배치 처리 건수
- Blur: 동시 요청 테스트 시 정합성 오류 재현 건수
수치가 없더라도 "~문제를 발견하고, ~방식으로 해결했다"는 맥락 자체가 면접에서 충분한 화제가 됨.
