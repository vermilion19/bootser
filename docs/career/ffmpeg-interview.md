# FFmpeg 관련 면접 예상 질문 (백엔드 일반 · 경력 기준)

`blur_api`의 FFmpeg 처리 코드를 기반으로, **경력 백엔드 개발자 면접**에서 나올 만한 질문과
모범 답변·꼬리질문·개선 포인트를 정리한 문서입니다.

> 답변 전략: 경력 면접은 "옵션 암기"가 아니라 **트레이드오프 판단 / 장애 대응 / 운영 관점 / 설계 근거**를 봅니다.
> 항상 "왜 이렇게 했는가 → 한계는 무엇인가 → 어떻게 개선할 것인가" 3단으로 답하세요.

---

## 0. 30초 프로젝트 요약 (오프닝용)

> "영상 업로드 서비스에서 미디어 처리를 담당했습니다. ffmpeg/ffprobe 바이너리를 Java `ProcessBuilder`로 호출해
> 메타데이터 추출, 썸네일/WebP 미리보기, HLS 다중화질 변환, 구간 자르기, 회전, 블러, 타임라인 합성을 처리합니다.
> 인코딩은 NVIDIA GPU(CUDA + `h264_nvenc`)로 가속하고, ffmpeg 진행 로그를 파싱해 실시간 진행률을 클라이언트에 전달합니다.
> 계층은 Facade(업무 흐름) → Processor(경로/도메인 결합) → Util(순수 ffmpeg 실행)로 분리했습니다."

## 1. 아키텍처 / 설계 판단

### Q1. ffmpeg을 JNI 라이브러리(JavaCV/Xuggler)가 아니라 외부 프로세스로 호출한 이유는?
**핵심:** 트레이드오프를 설명하라는 질문.
- **장점**
  - ffmpeg 최신 빌드/필터/하드웨어 옵션을 그대로 사용 가능(바인딩 버전에 종속되지 않음).
  - 프로세스 격리 → ffmpeg 크래시가 JVM을 죽이지 않음(JNI는 네이티브 크래시 시 JVM 동반 사망 위험).
  - 실행 명령을 그대로 로깅 → 운영 중 동일 명령을 셸에서 재현/디버깅 가능.
- **단점/책임**
  - 프로세스 수명 관리(타임아웃, 좀비, stdout/stderr 버퍼)를 직접 구현해야 함.
  - 표준 출력 파싱(진행률) 등 "문자열 인터페이스"의 취약함.
- **꼬리질문:** "그럼 JNI는 언제 쓰겠나?" → 초저지연·프레임 단위 제어가 필요하거나, 프로세스 생성 오버헤드가 병목일 때.

### Q2. Facade → Processor → Util 3계층으로 나눈 이유와 각 책임은?
- **Util(`FFMPEGUtil`)**: 순수 ffmpeg 실행. 도메인/스토리지 모름 → 단위 테스트·재사용 쉬움.
- **Processor(`FFMPEGProcessor`)**: workDir/uuid 경로 조립 등 도메인 결합 지점.
- **Facade(`UploadFacade` 등)**: 정책 검사 → S3 업로드 → 메타 추출 → 썸네일 → DB 저장의 **업무 트랜잭션** 오케스트레이션.
- **포인트:** "ffmpeg 명령 변경이 업무 로직에 전파되지 않도록" 경계를 두었다고 설명.

### Q3. (경력 함정) `UploadFacade.uploadAndSaveV2`가 `@Transactional`인데, 내부에서 S3 업로드·ffmpeg·썸네일을 다 합니다. 문제는?
**이게 경력 면접에서 가장 날카로운 질문입니다.**
- **문제:** 트랜잭션이 열린 채로 **수 초~수십 초 걸리는 외부 I/O(S3, ffmpeg)** 가 실행됨.
  - DB 커넥션을 그 시간 내내 점유 → 커넥션 풀 고갈 → 동시 업로드 시 전체 API 지연.
  - 외부 작업 실패 시 롤백되지만, **S3에 올라간 파일·생성된 썸네일은 롤백 안 됨(보상 트랜잭션 필요)**.
- **개선:**
  - 무거운 미디어 작업은 트랜잭션 밖으로 빼고, DB write만 짧은 트랜잭션으로 감싼다.
  - 업로드 후 인코딩/썸네일은 **비동기 큐(메시지 큐 + 워커)** 로 분리, 상태를 `InferenceState`로 관리(이미 상태 enum이 있음).
  - 외부 리소스 정리는 실패 시 보상 로직 또는 주기적 orphan cleanup.

## 2. 프로세스 실행 · 안정성 (운영 관점, 가장 깊게 파는 영역)

### Q4. ffmpeg이 hang하면? 타임아웃/좀비 프로세스는 어떻게 처리했나?
- 코드: `process.waitFor(120, TimeUnit.SECONDS)` 후 미완료면 `destroyForcibly()`.
- **모범 답변 + 자기비판:**
  - 고정 120초는 **긴 영상에 부족, 짧은 영상엔 과함** → 영상 길이 비례 동적 타임아웃이 낫다.
  - `destroyForcibly()`는 SIGKILL → ffmpeg이 임시파일을 정리 못 하고 죽을 수 있음. 자식 프로세스가 더 있으면 고아 가능 → `process.descendants()`까지 정리 고려.
  - **버그 포인트(짚으면 가산점):** `run()`의 `if(result) process.destroyForcibly();` 는 `waitFor`가 `true`(정상 종료)일 때 destroy를 호출 → 정상 종료된 프로세스를 죽이고, **정작 타임아웃(false) 시엔 안 죽이는** 역전된 로직으로 보임. 주석은 60초인데 코드는 120초로 불일치.

### Q5. ffmpeg의 stdout/stderr를 안 읽으면 무슨 일이 생기나?
**경력자라면 반드시 알아야 하는 고전 함정.**
- OS 파이프 버퍼(보통 64KB)가 가득 차면 ffmpeg의 write가 블로킹 → **프로세스 데드락**(영원히 안 끝남).
- 해법: 출력을 계속 소비해야 함.
  - `runCheckTime()`: `redirectErrorStream(true)`로 합친 뒤 루프에서 `readLine()`으로 계속 읽음(+진행률 파싱).
  - `run()`: `inheritIO()`로 부모 프로세스 표준 스트림에 위임 → 버퍼 문제 회피.
- **꼬리질문:** "stdout/stderr 둘 다 있으면?" → 한쪽만 읽으면 다른 쪽 버퍼가 차서 데드락. `redirectErrorStream` 또는 별도 스레드로 동시에 읽어야 함.

### Q6. 진행률(progress)은 어떻게 구현했나?
- ffmpeg stderr의 `frame=... time=00:00:12.34 ...`에서 정규식 `time=(\S+)`로 현재 처리 시간 추출.
- `progress = time / (전체 duration) * 100`, 전체 duration은 ffprobe로 사전 측정.
- **스로틀링:** 1초에 한 번 + 값이 바뀔 때만 콜백 → 네트워크/로그 폭주 방지.
- **한계:** ffmpeg 진행률은 비선형(앞부분 빠르고 뒤가 느릴 수 있음), 2-pass면 부정확. "사용자 체감용 추정치"임을 명확히.

### Q7. 동시에 인코딩 요청이 몰리면? 리소스(특히 GPU) 관리는?
- **NVENC 동시 세션 제한:** 소비자급 GPU는 동시 인코딩 세션 수가 드라이버 레벨로 제한됨(예: 제한 N개) → 초과 시 인코더 초기화 실패.
- 현재 코드에 동시 실행 제한이 없다면 **개선점으로 제시:**
  - `Semaphore`/전용 스레드풀로 동시 ffmpeg 수 제한, 초과분은 큐 대기.
  - 또는 작업을 메시지 큐로 빼고 워커 수로 자연스럽게 제한.
- CPU/메모리/디스크 I/O도 유한 자원 → 한 노드에서 무한정 fork 금지.

### Q8. 임시 파일 / 디스크 관리는?
- workDir에 원본·세그먼트·썸네일·중간 산출물이 쌓임. `MakeFileHelper`는 `deleteFileList`로 중간 파일 정리.
- **리스크:** 작업 실패/프로세스 강제종료 시 임시파일 잔존 → 디스크 가득 참. 주기적 cleanup·TTL 디렉터리 권장.

## 3. ffmpeg 명령/옵션 이해 (지식 검증)

### Q9. `-ss`/`-t`를 `-i` 앞 vs 뒤에 두는 차이?
- **앞(input seeking):** 키프레임으로 점프 → 빠름, 그러나 정확도↓(키프레임 단위).
- **뒤(output seeking):** 디코딩하며 탐색 → 정확, 그러나 느림.
- 이 프로젝트는 정확도 위해 **`-i` 뒤**에 둠(`splitVideoByTime`). 빠른 미리보기엔 앞이 유리 → 상황별 선택.

### Q10. 컨테이너 vs 코덱? `-c copy`의 의미와 사용처는?
- 컨테이너(mp4/m3u8/ts) = 포장지, 코덱(h264/aac) = 압축방식.
- `-c copy` = 재인코딩 없이 스트림 복사 → 무손실·고속. 회전 코드에서 오디오 `-c:a copy`로 음질 보존 + 속도 확보.

### Q11. `-vf`와 `-filter_complex` 차이? 언제 무엇을?
- `-vf`: 단일 입력→단일 출력 비디오 필터 체인(scale, transpose, drawtext).
- `-filter_complex`: 다중 입력/출력·분기·합성. `split`/`overlay`/`amix` 쓰는 합성·블러·다중화질이 해당.
- 라벨로 그래프 구성: `[0:v]scale=...[v];[v][1:v]overlay=...[out]`.

### Q12. CRF/CQ vs 비트레이트(`-b:v`)의 차이? 언제 무엇을?
- **CRF(libx264)/CQ(NVENC):** 품질 기준 가변 비트레이트. 값↓=고화질. 용량 예측 어렵지만 화질 일정.
- **`-b:v`:** 목표 비트레이트. 용량/대역폭 예측 쉬움 → HLS variant처럼 **대역폭 등급을 정해야 하는 스트리밍에 적합**.
- 코드: HLS는 `-b:v`(5000k/3000k/1500k), 합성/자르기는 `-cq`/`-crf` 계열 혼용.

### Q13. `-preset`은 무엇을 조절하나?
- 인코딩 **속도 vs 압축효율** 트레이드오프. 느릴수록 같은 화질에 용량↓·CPU/시간↑.
- libx264: `ultrafast`~`veryslow`, NVENC: `p1`~`p7`. 코드의 `slow`/`p4`/`p5`는 품질·속도 균형점 선택.

## 4. HLS / 스트리밍

### Q14. HLS 구조(master.m3u8 / variant playlist / .ts)와 ABR을 설명하라.
- 영상을 짧은 `.ts` 세그먼트로 분할, `.m3u8` 플레이리스트가 목록 관리.
- master.m3u8은 여러 화질(variant)을 가리킴 → 플레이어가 네트워크 상황에 따라 화질 자동 전환(**ABR, Adaptive Bitrate**).
- 코드: `-var_stream_map`으로 화질-스트림 매핑, `%v`로 화질별 폴더 분리, `-hls_time 6`(세그먼트 6초), `-hls_list_size 0`(VOD 전체 유지).

### Q15. 왜 원본을 그냥 mp4로 주지 않고 HLS 다중화질로 만드나?
- 모바일/저대역폭에서 버퍼링 없이 시청(화질 자동 강등), 빠른 초기 재생(세그먼트 단위), CDN 캐싱 친화.
- 비용: 변환 시간·스토리지·복잡도 증가 → 트레이드오프.

### Q16. 세그먼트 길이(`-hls_time`)는 어떻게 정하나?
- 짧으면(2s) 화질 전환 빠르고 초기 지연↓, 그러나 요청 수·오버헤드↑.
- 길면(10s) 효율↑, 그러나 전환 반응↓. 6초는 일반적 절충값.

## 5. GPU / 하드웨어 가속 (이 프로젝트 차별점)

### Q17. NVENC/CUDA의 장단점은? 언제 CPU 인코딩이 나은가?
- **장점:** 인코딩 속도 압도적, CPU 부담↓ → 다중화질·대량 처리에 유리.
- **단점:** 동일 비트레이트 대비 화질이 libx264보다 약간 낮을 수 있음, GPU/드라이버 종속, 동시 세션 제한.
- **CPU가 나은 경우:** 최고 압축효율이 중요한 아카이빙, GPU 없는 환경, 소량 처리. (코드가 블러 강도를 CUDA 기준 `BLUR_WEIGHT=70`으로 보정한 것도 화질 특성 차이 때문.)

### Q18. `-hwaccel cuda`, `-hwaccel_output_format cuda`, `hwupload`/`hwdownload`의 의미는?
- `-hwaccel cuda`: GPU 디코딩.
- `-hwaccel_output_format cuda`: 디코딩 프레임을 **GPU 메모리에 유지** → CPU↔GPU 복사 최소화(성능 핵심).
- `hwdownload`/`hwupload`: GPU↔CPU 메모리 이동. `gblur` 등 일부 필터는 GPU 메모리에서 직접 못 써서 내렸다가 다시 올림.

### Q19. (날카로움) Dockerfile.base는 apt의 일반 ffmpeg을 설치합니다. NVENC가 동작하나요? 배포 리스크는?
- apt ffmpeg에 nvenc 인코더가 빌드돼 있어도 **런타임에 NVIDIA 드라이버 + GPU 접근(NVIDIA Container Toolkit, `--gpus all`)** 이 없으면 `h264_nvenc` 호출이 실패.
- **리스크:** GPU 없는 노드/로컬에 배포하면 인코딩이 통째로 실패. 코드 대부분이 NVENC 의존.
- **개선:** 런타임에 GPU/인코더 가용성 체크 → 없으면 **libx264 CPU 폴백**. 명령 빌드를 추상화해 인코더를 주입 가능하게.

## 6. 보안 (경력 면접 단골)

### Q20. 사용자 입력(파일명/텍스트)이 ffmpeg 명령에 들어가는데 커맨드 인젝션 위험은?
- `ProcessBuilder(List<String>)`는 **셸을 거치지 않고** 인자를 직접 전달 → 셸 메타문자 인젝션 방어됨(`Runtime.exec(String)`이었다면 위험).
- **남은 위험:** `drawtext=text=...`처럼 **필터 문자열 내부**에 사용자 텍스트가 들어가면 필터 구문(`:`, `'`, `\`)이 깨지거나 다른 필터 주입 가능 → **이스케이프 필수**.
- **추가:** 파일명/경로/확장자 화이트리스트, 길이 제한, MIME/실제 포맷 검증(확장자만 믿지 말고 ffprobe로 실제 코덱 확인).

### Q21. `URLDecoder.decode(path)` 후 검증 없이 ffmpeg에 넘기는데 path traversal 위험은?
- 디코딩 후 `../`로 작업 디렉터리 밖 파일 접근 가능성 → **경로 정규화 후 허용 루트(workDir) 하위인지 검증** 필요.
- 업로드 파일은 UUID 기반 경로로 강제해 사용자 입력 경로를 직접 안 쓰는 게 안전.

### Q22. 악성 미디어 파일(디코드 폭탄, 손상 파일)에 대한 방어는?
- ffmpeg/ffprobe 자체 취약점 가능성 → 버전 최신화, 처리 프로세스 격리(컨테이너/낮은 권한 유저/리소스 cgroup 제한).
- 타임아웃·메모리 제한·해상도/길이 상한 검증으로 자원 고갈 방어.

## 7. 테스트 / 운영 / 관측성

### Q23. ffmpeg 의존 코드를 어떻게 테스트하나?
- 순수 로직(명령어 빌드: `FFmpegCommandBuilder.build()`)은 **생성된 인자 리스트를 assert**하는 단위 테스트로 분리(바이너리 불필요).
- 실제 실행 테스트는 작은 샘플 미디어로 통합 테스트(CI에 ffmpeg 설치 필요) → 느리므로 분리.
- 이 분리가 "Util을 순수하게, 실행을 얇게" 둔 설계의 이점.

### Q24. 운영 중 인코딩 실패를 어떻게 진단하나?
- 코드가 실행 명령 전체를 로그로 남김(`printLog`) → 동일 명령 재현 가능.
- exit code·stderr 마지막 라인 수집, 작업 ID(fileId/uuid)로 추적, 실패율/소요시간 메트릭(p95) 수집.
- **개선:** 구조화 로깅 + 작업 상태 머신(대기/진행/성공/실패/재시도) + 재시도 정책(멱등하도록 출력 경로 결정적으로).

### Q25. 인코딩 작업의 멱등성/재시도는?
- 같은 입력→같은 출력 경로가 결정적이어야 재시도 안전. 부분 산출물(깨진 ts/mp4)을 임시 경로에 쓰고 **성공 시 atomic rename**(코드의 `renameTo`/`Files.move REPLACE_EXISTING` 패턴) → 소비자가 반쪽 파일을 보지 않게.

## 8. "이 코드에서 개선하고 싶은 점은?" (자기비판 — 반드시 준비)

면접 마무리에 자주 나오는 질문. 다음을 근거와 함께:
1. **`@Transactional` 안의 장시간 외부 I/O** → 커넥션 점유·롤백 불일치. 미디어 작업 비동기 분리.(Q3)
2. **`run()`의 `destroyForcibly()` 조건 역전 + 주석/코드 타임아웃 불일치(60s vs 120s).**(Q4)
3. **고정 120초 타임아웃** → 영상 길이 비례 동적화.
4. **NVENC 실패 시 CPU 폴백 부재** → 환경 의존 장애.(Q19)
5. **동시 실행 제한(세마포어/큐) 부재** → GPU 세션 초과·자원 폭주.(Q7)
6. **빈 catch 블록**(`rotationVideoMetadata`의 `catch(IOException e){}`) → 장애 은폐, 최소 로깅 필요.
7. **`drawtext` 등 필터 내 사용자 입력 이스케이프 미흡** → 보안.(Q20)
8. **`convertToM3U8For3CH`의 filter graph에서 `[v2out]`를 map하지만 filter_complex가 v2out을 생성하지 않는 듯** → 실제 동작 검증 필요(잠재 버그).
9. **동기 처리 모델** → 대량 트래픽 시 워커/큐 기반 비동기 파이프라인으로 전환.

### 시뮬레이션 A — `@Transactional` 안의 외부 I/O (가장 깊게 파임)

🧑‍💼 `uploadAndSaveV2`에 `@Transactional`이 붙어 있네요. 이 메서드가 하는 일을 순서대로 말해보세요.
🙋 정책 검사 → 로컬 디스크에 파일 저장 → S3 업로드 → ffprobe 메타 추출 → 썸네일 추출 → DB 저장입니다.

🧑‍💼 그 중 DB 작업은 몇 개죠? 나머지는요?
🙋 DB write는 `aiFileRepository.save` 몇 번뿐이고, 나머지(S3·ffmpeg·썸네일)는 전부 외부 I/O입니다. 즉 **수십 초 걸리는 외부 작업이 트랜잭션 경계 안에 들어가 있습니다.**

🧑‍💼 그게 왜 문제죠? 구체적으로.
🙋 트랜잭션이 열리는 순간 DB 커넥션 하나를 점유합니다. 외부 I/O가 끝날 때까지 그 커넥션을 **반납하지 않습니다.** 인코딩이 30초 걸리면 커넥션을 30초간 붙잡고, 동시 업로드가 풀 크기만큼만 들어와도 커넥션 풀이 고갈돼 **업로드와 무관한 다른 API까지 커넥션을 못 받아 전체 장애**로 번집니다.

🧑‍💼 풀 사이즈 늘리면 되지 않나요?
🙋 근본 해결이 아닙니다. 커넥션은 DB 측 자원(메모리·max_connections)이라 무한히 못 늘리고, 늘려도 "긴 작업이 커넥션을 점유한다"는 구조 자체는 그대로라 트래픽이 늘면 다시 터집니다. **점유 시간을 줄이는 게 정답**이지 풀을 키우는 건 미봉책입니다.

🧑‍💼 좋아요. 또 다른 문제는?
🙋 **롤백 불일치**입니다. ffmpeg이나 썸네일 단계에서 예외가 나면 DB는 롤백되지만, 그 전에 **S3에 올라간 원본 파일과 생성된 썸네일은 롤백되지 않습니다.** DB엔 레코드가 없는데 S3엔 파일이 남는 orphan이 생깁니다.

🧑‍💼 그럼 어떻게 고치겠어요?
🙋 두 단계로 나눕니다. (1) 업로드 직후엔 **짧은 트랜잭션**으로 메타데이터와 `status=PROCESSING` 레코드만 저장하고 즉시 커밋·응답. (2) 인코딩·썸네일은 **메시지 큐에 작업을 넣고 워커가 비동기 처리**, 완료 시 상태를 `READY`로 갱신합니다. 이미 `InferenceState` enum이 있으니 상태 머신으로 자연스럽게 표현됩니다.

🧑‍💼 비동기로 빼면 S3 orphan 문제는 해결되나요?
🙋 자동으로 해결되진 않습니다. 실패 시 보상 작업(업로드한 파일 삭제)을 워커에 넣거나, **주기적으로 "DB에 없는 S3 객체"를 청소하는 orphan cleanup 배치**를 둡니다. 업로드 경로를 uuid로 결정적으로 만들면 청소 대상 식별이 쉽습니다.

🧑‍💼 작업을 큐에 넣었는데 워커가 처리 중 죽으면요?
🙋 메시지를 ack하지 않으면 큐가 재전달하므로, **작업이 멱등(idempotent)** 해야 합니다. 같은 uuid면 출력 경로가 동일하고, 중간 산출물은 임시 경로에 쓴 뒤 성공 시 atomic rename → 재시도해도 깨진 결과가 노출되지 않습니다. 무한 재시도를 막기 위해 재시도 횟수 초과 시 **DLQ(Dead Letter Queue)** 로 보내 수동 점검합니다.

🧑‍💼 (마무리) 그럼 사용자는 인코딩이 끝났는지 어떻게 아나요?
🙋 업로드 응답은 즉시 주고 상태는 `PROCESSING`. 완료는 (1) 클라이언트 폴링 또는 (2) WebSocket/SSE 푸시로 알립니다. 마침 이 프로젝트는 진행률을 콜백으로 푸시하는 구조가 있어 그 채널을 재사용하면 됩니다.

### 시뮬레이션 B — 프로세스 출력 데드락 (CS 기본기 검증)

🧑‍💼 `runCheckTime`에서 `while ((line = reader.readLine()) != null)`로 출력을 계속 읽는데, 이 루프를 지우면 어떻게 되나요?
🙋 ffmpeg이 중간에 멈춰서 **영원히 안 끝날 수 있습니다.** 진행률을 못 받는 정도가 아니라 인코딩 자체가 데드락에 빠집니다.

🧑‍💼 왜 멈추죠? 인코딩은 자기 일만 하면 되잖아요.
🙋 ffmpeg은 진행 로그를 stderr로 계속 출력합니다. OS의 파이프 버퍼(보통 64KB 정도)가 가득 차면, 부모가 읽어가기 전까지 ffmpeg의 write() 시스템콜이 **블로킹**됩니다. 부모는 `waitFor`로 종료만 기다리고 출력을 안 읽으면, 서로 상대를 기다리는 **고전적 데드락**이 됩니다.

🧑‍💼 그럼 stdout, stderr 둘 다 있으면요?
🙋 한쪽만 읽으면 안 읽는 쪽 버퍼가 차서 똑같이 막힙니다. 그래서 이 코드는 `redirectErrorStream(true)`로 **stderr를 stdout에 합쳐 하나의 스트림으로 만들고** 단일 루프로 읽습니다. 합치지 않으려면 stderr 전용 읽기 스레드를 따로 둬야 합니다.

🧑‍💼 `run()` 메서드는 루프가 없는데 거긴 왜 안 막히죠?
🙋 거긴 `builder.inheritIO()`를 씁니다. 자식 프로세스의 표준 스트림을 **부모(JVM) 프로세스의 스트림에 직접 연결**해서, OS가 출력을 콘솔/상위로 흘려보냅니다. 별도 버퍼에 고이지 않으니 데드락이 없습니다. 대신 그 출력을 Java 코드에서 파싱할 순 없어서, 진행률이 필요한 쪽은 `inheritIO` 대신 직접 읽는 방식을 씁니다.

🧑‍💼 Java 표준 라이브러리에 이걸 더 안전하게 해주는 건 없나요?
🙋 `ProcessBuilder.redirectOutput/redirectError(Redirect.to(file))`로 파일에 흘리거나, Java 9+ 면 `Process`를 별도 스레드풀에서 스트림 펌핑하는 패턴을 씁니다. 핵심은 **"출력은 반드시 누군가 계속 소비한다"** 는 원칙입니다.

### 시뮬레이션 C — GPU 동시성과 폴백 (운영/장애 대응)

🧑‍💼 인코딩을 NVENC로 합니다. 트래픽이 몰려서 동시에 50개 업로드가 들어오면 어떻게 되나요?
🙋 현재 구조면 요청마다 ffmpeg 프로세스를 fork하므로 동시에 50개가 GPU 인코딩을 시도합니다. **NVENC는 GPU 드라이버 레벨에서 동시 인코딩 세션 수가 제한**돼 있어(소비자급 GPU는 보통 소수), 한계를 넘으면 인코더 초기화가 실패하고 그 작업들이 줄줄이 에러납니다.

🧑‍💼 그 제한을 코드는 어떻게 막고 있나요?
🙋 제가 본 범위에선 **명시적 동시 실행 제한이 없습니다.** 이게 개선점입니다. `Semaphore`로 동시 ffmpeg 수를 GPU 세션 한도 이하로 제한하거나, 아예 작업을 큐에 넣고 **워커 수로 동시성을 통제**해야 합니다.

🧑‍💼 세마포어로 막으면 51번째 요청은 어떻게 되죠?
🙋 동기 API라면 대기하다 타임아웃 위험이 있으니, 비동기 큐 모델이 낫습니다. 51번째는 큐에서 대기하고 사용자에겐 `PROCESSING` 상태로 즉시 응답합니다. 처리량이 부족하면 GPU 노드/워커를 수평 확장합니다.

🧑‍💼 GPU 없는 환경에 배포되면요? Dockerfile은 apt ffmpeg을 쓰던데.
🙋 apt ffmpeg에 nvenc가 빌드돼 있어도 **런타임에 NVIDIA 드라이버와 GPU 접근(NVIDIA Container Toolkit, `--gpus all`)** 이 없으면 `h264_nvenc` 호출이 즉시 실패합니다. 로컬·GPU 없는 노드에선 인코딩이 전부 죽습니다.

🧑‍💼 그럼 어떻게 견고하게 만들죠?
🙋 인코더를 하드코딩하지 않고 **추상화**합니다. 기동 시 `ffmpeg -encoders`로 `h264_nvenc` 가용성을 점검하거나, NVENC로 시도 후 실패하면 **libx264 CPU 인코딩으로 폴백**합니다. 명령 빌더에서 코덱·hwaccel 옵션을 주입 가능하게 만들면 환경별로 같은 코드가 동작합니다.

🧑‍💼 CPU 폴백하면 화질이나 결과가 달라지지 않나요?
🙋 달라집니다. NVENC와 libx264는 같은 비트레이트라도 화질 특성이 다릅니다. 이 프로젝트가 블러 강도를 CUDA 기준 `BLUR_WEIGHT=70`으로 보정한 것도 그 차이 때문입니다. 그래서 폴백은 "장애 시 동작 보장"용이고, 품질 파라미터는 인코더별로 따로 튜닝해야 합니다.

### 시뮬레이션 D — HLS / 다중화질 (도메인 깊이)

🧑‍💼 왜 원본 mp4 하나만 주지 않고 굳이 화질 여러 개로 HLS를 만드나요?
🙋 시청 환경이 다양하기 때문입니다. 모바일·저대역폭 사용자가 고화질 원본을 받으면 버퍼링이 심합니다. HLS는 짧은 세그먼트 단위로 받고, 네트워크 상황에 따라 플레이어가 화질을 자동 전환(ABR)해 끊김 없이 재생합니다.

🧑‍💼 그 "자동 전환"은 누가 결정하나요? 서버? 클라이언트?
🙋 **클라이언트(플레이어)** 입니다. 서버는 master.m3u8에 각 화질의 대역폭 정보를 담아 제공만 하고, 플레이어가 측정한 처리량/버퍼 상태에 따라 다음 세그먼트를 어느 variant에서 받을지 스스로 고릅니다. 그래서 HLS variant는 **목표 비트레이트(`-b:v`)** 로 등급을 명확히 나눠야 합니다 — CRF 가변이면 대역폭 예측이 안 돼 ABR 판단이 흐려집니다.

🧑‍💼 세그먼트를 6초로 잡았는데 2초로 줄이면 뭐가 좋고 나빠지나요?
🙋 2초면 화질 전환 반응이 빨라지고 초기 재생 지연(첫 세그먼트 다운로드)이 줄어듭니다. 대신 세그먼트 수가 늘어 HTTP 요청 수·플레이리스트 크기·인코딩 오버헤드가 커집니다. 6초는 일반적인 절충값이고, 라이브·저지연이 중요하면 짧게 갑니다.

🧑‍💼 코드에서 `-hls_list_size 0`을 줬는데 의미는?
🙋 플레이리스트에 세그먼트를 **전부 유지**한다는 뜻으로, VOD(다시보기)용입니다. 라이브면 최근 N개만 유지하도록 작은 값을 줘서 슬라이딩 윈도우로 운영합니다.

🧑‍💼 화질이 3개면 인코딩 비용이 3배인데, 어떻게 줄이죠?
🙋 코드의 `convertToM3U8For2CH`처럼 **한 번의 ffmpeg 실행에서 `split`으로 입력을 나눠 여러 화질을 동시 출력**하면, 디코딩은 한 번만 하고 인코딩만 화질 수만큼 합니다. 입력을 N번 다시 디코딩하는 것보다 효율적입니다. 더 줄이려면 CMAF로 코덱/오디오를 공유하거나, 인기 없는 영상은 lazy하게(요청 시) 변환하는 전략도 가능합니다.

### 시뮬레이션 E — `run()`의 버그 잡아내기 (코드 리뷰 능력)

🧑‍💼 (`FFMPEGUtil.run` 코드를 보여주며) 이 메서드에서 이상한 점 있나요?
🙋 네, `boolean result = process.waitFor(120, ...); if(result) process.destroyForcibly();` 부분이 의심스럽습니다. `waitFor(timeout)`은 **시간 내 정상 종료되면 `true`** 를 반환합니다. 그런데 `true`일 때 `destroyForcibly()`로 죽이고 있어요. 정상 종료된 프로세스를 굳이 죽이고, **정작 타임아웃(false)일 때는 죽이지 않는** 역전된 로직으로 보입니다.

🧑‍💼 정상 종료된 프로세스를 destroy하면 무슨 일이 생기죠?
🙋 이미 끝난 프로세스라 큰 부작용은 없을 수 있지만, 의도는 분명 "타임아웃 시 강제 종료"였을 겁니다. 지금은 **타임아웃 난 좀비 프로세스가 안 죽고 남아** GPU/CPU를 계속 점유할 위험이 있습니다. 조건을 `if(!result)`로 바꿔야 맞습니다.

🧑‍💼 그것 말고 또?
🙋 주석엔 "60초 내에 끝나지 않으면"이라고 적혀 있는데 실제 코드는 `120`초입니다. 주석-코드 불일치라 유지보수 시 혼란을 줍니다. 그리고 타임아웃이 고정값이라 긴 영상엔 부족합니다 — duration 기반 동적 타임아웃이 안전합니다.

🧑‍💼 좋네요. 이런 걸 어떻게 사전에 잡죠?
🙋 `waitFor`의 반환값 의미 같은 건 테스트로 고정합니다. 타임아웃 케이스(일부러 안 끝나는 더미 프로세스)와 정상 케이스를 각각 단위 테스트로 만들어 "타임아웃 시 프로세스가 실제로 종료되는지"를 검증하면 이 역전 버그가 바로 드러납니다.
