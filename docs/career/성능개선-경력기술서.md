# 경력기술서용 — 백엔드 성능 개선 / 운영 장애 대응 사례

> 택시 텔레매틱스(GPS·ADAS) 데이터 수집 및 안전운전점수 산정 백엔드.
> 기술 스택: **Spring Boot, Spring Batch, JPA(Hibernate), MyBatis, PostgreSQL, HikariCP, AWS S3**
> 데이터 규모: GPS 이력 테이블 약 **8,500만 행 / 24GB**, 분 단위·일 단위 배치 다수.

> ⚠️ 사용 안내: 아래 "성과"의 구조적 개선(트랜잭션 구조, 쿼리 수, 메모리 적재 방식 등)은 코드/쿼리로 확정된 사실입니다.
> 런타임 수치 중 일부는 모니터링으로 사후 측정해 채우면 신뢰도가 높아집니다(표시해 둠).

## 사례 1. 일배치 안전운전점수 산정 로직의 트랜잭션·메모리·쿼리 구조 개선

**[문제]**
- 안전운전점수 산정 배치(`caculeSafeScore`)가 **전체 이력을 하나의 거대 `@Transactional`** 안에서 루프 처리.
  - 배치가 도는 수십 분~시간 동안 **DB 커넥션 1개를 계속 점유** → HikariCP 풀 고갈에 기여.
  - 레코드 N건 처리 중 1건 예외 → **전체 롤백 + 루프 중단**으로 진행분 유실.
- 전체 대상 이력을 **메모리에 한 번에 적재**(수십만 행) → 메모리 압박/GC 부하.
- 레코드 1건당 **5개 쿼리(SELECT 4 + UPDATE 1)** 를 순차 실행 → 전형적 N+1.

**[조치]**
- 레코드 단위 처리를 **별도 빈의 `@Transactional(REQUIRES_NEW)`** 로 분리하고, 호출부는 **건별 `try/catch`로 실패 스킵**.
  → 한 건 실패가 전체에 전파되지 않고(장애 격리), **진행분 보존**, 커넥션을 건건이 짧게 점유.
- 전체 일괄 로딩을 **Spring Data `Slice` 기반 1,000건 페이징 순회**로 전환 → 메모리/GC 부하 감소.
- 건당 쿼리를 **5개 → 3개**로 축소(N+1 완화):
  - 같은 테이블·같은 시간구간을 따로 조회하던 2개 쿼리를 **PostgreSQL 조건부 집계(`FILTER`) 단일 쿼리**로 통합.
  - 행 자체 값으로 계산 가능한 집계 1개는 **SQL 조회 → 애플리케이션 산술 연산으로 대체**.
- 리팩터 전/후 **결과 동등성(점수 값 불변)을 쿼리 단위로 검증**하고, 백업 브랜치·단위 커밋으로 **안전한 롤백 경로** 확보.

**[성과]**
- 배치 실패 내성 확보: 단건 오류로 전체가 멈추거나 롤백되던 구조 → **진행분 보존 + 문제 건만 스킵**.
- 커넥션 점유 시간·동시 점유량 **구조적 감소**(거대 단일 트랜잭션 제거).
- 건당 DB 왕복 **40% 감소(5→3)**, 메모리 일괄 적재 제거.

**[사용 기술]** Spring 트랜잭션 전파(`REQUIRES_NEW`), JPA/QueryDSL, MyBatis, PostgreSQL `FILTER` 집계, 페이징(`Slice`).

## 사례 2. 거리계산 배치 쿼리의 전체 스캔 제거 (Prepared Statement Generic Plan 이슈)

**[문제]**
- GPS 구간별 거리·주행시간 계산 배치(`updateDistance`)가 **구간 1건당 약 10분** 소요(진단: `pg_stat_activity`의 `wait_event=DataFileRead`로 I/O 바운드 확인).
- 원인 규명: 쿼리가 `(SELECT * FROM uc_gps_history ORDER BY time) ...` **불필요한 정렬 서브쿼리**를 거쳐 필터링.
  - 단발 쿼리(literal 값)는 옵티마이저가 인덱스로 최적화했지만, 애플리케이션은 **Prepared Statement(서버 prepared) → generic plan** 으로 굳어 **8,500만 행 테이블을 전체 Seq Scan + Sort** 함을 `EXPLAIN (ANALYZE, BUFFERS)` 및 `plan_cache_mode=force_generic_plan` 재현으로 확정.

**[조치]**
- 정렬 서브쿼리를 제거하고 **테이블에 직접 인덱스 조건(device_id, time)** 을 걸도록 쿼리 재작성.
  - 윈도우 함수(`LAG`)가 자체 `ORDER BY`로 정렬하고 최종 `UPDATE`는 PK 매칭이므로 **결과 동등성 보장**(중간 정렬은 불필요했음을 논증).
  - custom/generic plan **양쪽 모두 인덱스 스캔**을 타도록 유도해 generic plan 함정 제거.

**[성과]**
- 구간당 전체 테이블 스캔(8,500만 행) 제거 → 인덱스 스캔으로 전환, **구간당 수 분 단위 소요 해소**.
- *(측정 권장)* 배포 후 배치 전체 소요시간 before/after 수치 기록 시 임팩트 강화.

**[사용 기술]** PostgreSQL 실행계획 분석(`EXPLAIN ANALYZE/BUFFERS`, `wait_event`), `plan_cache_mode`, 인덱스 설계, PgJDBC prepared statement 동작 이해.

## 사례 3. GPS 적재 테이블 PK 시퀀스 고갈(int4 한계) 운영 장애 대응 및 재발 방지

**[문제]**
- 운영 중 GPS 적재(`COPY`)가 **전면 실패**: `nextval: reached maximum value of sequence (2147483647)`.
- `uc_gps_history.id`가 **`integer`(int4)** 이고 시퀀스가 **21억 상한 소진** → 신규 INSERT 불가.
- **모순 포착으로 근본 원인 규명**: 실제 살아있는 행은 약 8,500만인데 시퀀스는 21억 → "넣었다 버리는 누수"가 있음을 추론.
  - `migrateDbFile`이 **ADAS 테이블 없는 파일을 GPS COPY '후'에 검사**해 예외 발생 → 트랜잭션 롤백(GPS 행은 사라지나 **시퀀스 값은 영구 소비**) → `migrate_flag='N'`으로 남아 **매분 동일 파일 재COPY** → 시퀀스 무한 소모.

**[조치]**
- *(코드, 재발 방지)*
  - **ADAS 존재 검사를 COPY 이전으로 이동** → 누락 파일은 COPY 자체를 생략(시퀀스 소비 0).
  - 누락 파일을 **별도 상태(`migrate_flag='E'`)로 표시**해 재시도 대상에서 제외(무한 루프 차단).
  - **레코드별 예외 격리**로 한 건 실패가 전체 배치를 중단시키지 않도록 개선.
- *(운영 DDL, 복구)*
  - `id` 컬럼 **`integer → bigint` 전환**(24GB 테이블, `ACCESS EXCLUSIVE` 락·전체 재작성) 수행 및 진행 모니터링.
  - 시퀀스 **`bigint` 전환 + MAXVALUE 상향**, `setval`로 기존 최대 id 위로 정렬(PK 충돌 방지).
  - 장기 DDL의 클라이언트 끊김 리스크(절전/세션 단절) 관리, `pg_stat_activity`로 진행/락 상태 실시간 확인.

**[성과]**
- 적재 전면 중단 장애 **복구** 및 **재발 방지**(누수 로직 제거 + 용량 한계 해소).
- 시퀀스 가용 범위 21억 → **9.2×10¹⁸(bigint)** 로 확장.

**[사용 기술]** PostgreSQL 시퀀스/`bigint` 마이그레이션, 대용량 테이블 DDL(락·재작성 영향), `COPY` 적재, 운영 장애 트러블슈팅, 근본원인 분석(RCA).

## 사례 4. 커넥션/스레드 점유 진단 체계 수립 및 문서화

**[문제]**
- "서버 로직이 안 도는 것 같다"류의 모호한 장애에서 **무엇이 커넥션/스레드를 점유하는지** 특정이 어려움.
- 모든 `@Scheduled` 잡이 **단일 스케줄러 스레드(pool size 1)** 를 공유 → 장기 배치가 점유하면 **매분 마이그레이션이 지연**되는 근본 구조 파악.

**[조치]**
- 3계층 진단 절차 정립:
  - **HikariCP 메트릭/누수 탐지**(`leak-detection-threshold`, `active/pending`)로 풀 고갈·점유 추세 파악.
  - **`pg_stat_activity`** (`idle in transaction`, `wait_event_type`)로 점유 주체·쿼리 특정.
  - **스레드 덤프**(`jstack`/`jcmd`/actuator)로 `scheduling-1` 등 블로킹 스레드·락 경합 분석.
- 메트릭 급증 시각 ↔ 배치 cron ↔ 스레드 스택 ↔ 실행 쿼리를 **교차 대조하는 런북** 문서화.

**[성과]**
- 장애 시 **원인 특정 시간 단축**, 재현 가능한 진단 절차 표준화(팀 공유용 런북 + 진단 SQL/모니터링 가이드 작성).

**[사용 기술]** HikariCP, Spring `@Scheduled`/Spring Batch, JVM 스레드 덤프 분석, PostgreSQL 모니터링, 운영 문서화.

## 경력기술서 복붙용 요약 불릿

- 일배치 안전운전점수 산정 로직의 **단일 거대 트랜잭션을 레코드 단위 독립 트랜잭션(`REQUIRES_NEW`)으로 분리**, 1건 실패 시 전체 롤백·중단되던 구조를 **진행분 보존·실패 격리**로 개선하고 DB 커넥션 점유 시간을 단축.
- 배치 처리의 **건당 쿼리를 5개 → 3개로 축소**(조건부 집계 통합·연산 인라인화로 N+1 완화), **전체 이력 일괄 로딩을 1,000건 페이징으로 전환**해 메모리/GC 부하 감소. 리팩터 전후 **결과 동등성 검증**.
- 거리계산 배치 쿼리가 **Prepared Statement generic plan에서 8,500만 행 테이블을 전체 스캔·정렬**하던 문제를 `EXPLAIN`·`wait_event` 분석으로 규명, **불필요한 정렬 서브쿼리 제거로 인덱스 스캔을 유도**해 구간당 약 10분 소요를 해소.
- GPS 적재 PK 시퀀스가 **int4 상한(21억) 소진으로 적재 전면 중단된 운영 장애**를 대응: **‘ADAS 누락 파일 롤백 후 매분 재적재’ 누수 로직을 근본 원인으로 규명**해 코드로 재발 방지하고, **id 컬럼·시퀀스 bigint 전환(24GB 테이블)** 으로 서비스 복구.
- **HikariCP 메트릭·`pg_stat_activity`·스레드 덤프** 기반 커넥션/스레드 점유 진단 절차를 표준화해 **런북으로 문서화**, 모호한 성능 장애의 원인 특정 시간을 단축.

## 기술 키워드 (이력서 스킬 섹션용)

`PostgreSQL` `EXPLAIN ANALYZE` `실행계획/인덱스 튜닝` `시퀀스/bigint 마이그레이션` `COPY 대량적재`
`Spring Boot` `Spring Batch` `JPA/Hibernate` `MyBatis` `트랜잭션 전파 설계`
`HikariCP 커넥션 풀` `N+1 최적화` `장애 대응/RCA` `JVM 스레드 덤프` `운영 문서화`
