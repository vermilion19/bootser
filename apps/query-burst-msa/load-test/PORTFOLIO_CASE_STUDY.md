# 동시성 상황에서 주문 정합성을 검증하고 개선한 경험

## 부제

k6 기반 부하 테스트로 확인한 중복 주문 방지와 과판매 대응

---

## 1. 프로젝트 소개

`query-burst-msa`는 주문, 재고, 랭킹, 분석 기능이 분리된 MSA 기반 프로젝트다.  
서비스는 `order-service`, `catalog-service`, `member-service`, `ranking-service`, `analytics-service`로 나뉘어 있고, 앞단에는 `nginx`, 저장소로는 `PostgreSQL`, 메시징에는 `Kafka`, 캐시에는 `Redis`를 사용한다.

이번 작업에서는 기능 자체보다 정합성에 집중했다.  
주문 시스템에서는 단건 요청이 잘 동작하는 것보다, 동시 요청이 몰리는 상황에서도 데이터가 깨지지 않는지가 더 중요하다고 생각했기 때문이다.

특히 아래 두 가지를 핵심 검증 대상으로 잡았다.

- 같은 주문 요청이 동시에 여러 번 들어와도 실제 주문은 한 번만 생성되는가
- 재고보다 많은 주문이 동시에 들어와도 예약 성공 수는 실제 재고를 넘지 않는가

---

## 2. 왜 이 테스트를 했는가

중복 주문과 과판매는 주문 시스템에서 가장 위험한 문제에 가깝다.

중복 주문이 발생하면 주문 데이터만 꼬이는 것이 아니라, 이후 결제, 배송, 통계 데이터까지 함께 오염될 수 있다.  
과판매는 더 직접적이다. 실제 재고보다 많은 주문을 받아버리면 취소, 환불, 고객 응대까지 전부 운영 이슈로 이어진다.

그래서 이번 테스트는 단순히 응답 속도를 재는 성능 테스트라기보다, 동시성 상황에서 실제 데이터 정합성이 유지되는지를 확인하는 검증 작업에 가까웠다.  
또한 k6 결과만 보고 판단하지 않고, 최종적으로는 DB를 직접 조회해서 실제 row 수까지 확인하는 방식으로 진행했다.

---

## 3. 테스트 환경

- Backend: `Java`, `Spring Boot`
- Infra: `Docker Compose`, `Nginx`
- Storage / Messaging: `PostgreSQL`, `Kafka`, `Redis`
- Load Test Tool: `k6`

테스트는 `docker compose`로 실행한 k6 컨테이너에서 수행했고, 모든 요청은 `query-burst-nginx`를 통해 서비스로 전달되도록 구성했다.

## 4. Idempotency 검증

### 4-1. 검증 목적

첫 번째로 확인한 것은 `Idempotency`였다.

같은 `Idempotency-Key`로 동시에 50개의 주문 요청이 들어왔을 때, 실제로는 주문 1건과 재고 예약 1건만 생성되는지를 확인하고 싶었다.

검증 기준은 아래처럼 잡았다.

- `customer_order`에서 해당 `idempotency_key` 기준 주문 수
- `inventory_reservation`에서 해당 `request_id` 기준 예약 수

### 4-2. 처음 실행했을 때 겪은 문제

처음에는 테스트가 바로 의미 있는 결과를 주지 않았다.

가장 먼저 확인한 문제는 요청 body가 API 계약과 맞지 않았다는 점이다.  
주문 요청에는 `items[].unitPrice`가 필요한데, 초기 k6 스크립트에는 이 값이 빠져 있었다. 그 결과, 동시성 문제가 아니라 요청 형식 오류 때문에 잘못된 결과가 나왔다.

그 다음으로는 더 구조적인 문제가 드러났다.  
catalog 쪽은 `request_id` unique 제약으로 예약 중복을 막고 있었지만, order-service는 주문 레벨에서 같은 키를 묶고 있지 않았다. 이 상태에서는 같은 예약을 참조하는 주문 row가 여러 개 생길 수 있었다.

즉, 예약은 하나인데 주문은 여러 건 생성될 수 있는 상태였다.

### 4-3. 수정 내용

이 문제를 해결하기 위해 order-service에 아래 내용을 반영했다.

- `customer_order.idempotency_key` 컬럼 추가
- `idempotency_key` unique 제약 추가
- 동일 키 요청이면 기존 주문을 재반환하도록 처리
- 동시에 같은 키 요청이 들어올 때를 막기 위해 Postgres advisory lock 적용

테스트 스크립트도 함께 수정했다.

- `unitPrice` 누락 수정
- 컨테이너 재기동 직후 `502`가 나오는 문제를 막기 위해 health check 대기 로직 추가
- 응답 횟수만으로 중복 생성 여부를 단정하지 않도록 판정 방식 수정

이 과정에서 알게 된 건, 응답이 여러 번 성공했다고 해서 실제 데이터가 여러 번 저장됐다고 볼 수는 없다는 점이었다.  
동일한 주문을 여러 번 재반환하는 경우도 응답만 보면 전부 `201`로 보일 수 있기 때문이다.

### 4-4. 최종 실행 결과

최종 실행에서는 50개의 동시 요청이 모두 `201 + STOCK_RESERVED`로 처리됐고, 서버 에러는 발생하지 않았다.

### 4-5. DB 직접 검증

k6 결과만으로는 충분하지 않다고 판단해서, 최종적으로 DB를 직접 확인했다.

확인 결과는 아래와 같았다.

- `inventory_reservation where request_id = sharedKey` → `1건`
- `customer_order where idempotency_key = sharedKey` → `1건`

즉, 응답은 여러 번 반환됐지만 실제 데이터는 주문 1건, 예약 1건만 생성됐다.  
결과적으로 주문 서비스의 idempotency는 의도한 방향으로 보강되었다고 판단했다.

## 5. Oversell 검증

### 5-1. 검증 목적

두 번째로 확인한 것은 `Oversell`이었다.

하나의 상품에 주문이 몰렸을 때, 예약 성공 수가 실제 시작 재고를 초과하지 않는지를 확인하고 싶었다.

시나리오는 아래처럼 구성했다.

- executor: `constant-arrival-rate`
- 요청률: `300 req/s`
- 지속 시간: `15초`
- 총 요청 수: 약 `4500`
- 각 요청은 서로 다른 `Idempotency-Key` 사용

### 5-2. 처음엔 실패처럼 보였던 이유

처음 결과만 보면 oversell이 발생한 것처럼 보였다.  
예약 성공 수가 기준 재고를 훨씬 넘었기 때문이다.

그런데 DB를 확인해보니 문제는 서비스가 아니라 테스트 코드 쪽에 있었다.  
당시 스크립트는 시작 재고를 `50`으로 하드코딩하고 있었는데, 실제 테스트에 사용된 상품 재고는 50이 아니었다.

즉, 이건 실제 버그가 아니라 잘못된 기준값 때문에 생긴 false positive였다.

### 5-3. 수정 내용

oversell 스크립트는 아래와 같이 수정했다.

- health check 대기 추가
- 첫 번째 상품을 무조건 쓰지 않고, 현재 `ACTIVE` 상품 중 재고가 가장 낮은 상품을 선택
- 하드코딩된 시작 재고 제거
- 실제 상품 재고를 읽어서 `기준 재고(실측)`로 사용
- 요약 출력에 실제 상품 ID와 실제 시작 재고를 반영

이 수정 이후부터는 테스트 시점의 실제 데이터에 맞춰 결과를 해석할 수 있게 됐다.

### 5-4. 최종 실행 결과

최종 실행에서는 아래와 같은 결과가 나왔다.

- 핫 상품 ID: `99700`
- 시작 재고: `701`
- `STOCK_RESERVED`: `701`
- `REJECTED`: `3800`

### 5-5. DB 직접 검증

테스트 이후 DB를 직접 조회해 확인한 결과는 아래와 같았다.

- `catalog_product.stock = 0`
- `status = SOLD_OUT`
- `stock < 0` row 수 = `0`
- 해당 상품 기준 `RESERVED` 건수 = `701`

정리하면 시작 재고 701개만큼만 예약이 성공했고, 그 이후 요청은 정상적으로 재고 부족으로 거절되었다.  
음수 재고는 발생하지 않았고, oversell도 확인되지 않았다.

## 6. 테스트하면서 겪은 문제와 배운 점

이번 작업은 단순히 부하 테스트를 몇 번 실행한 경험이라기보다, 테스트를 믿을 수 있는 상태로 다듬어가는 과정에 가까웠다.

실제로 중간에 확인한 문제는 아래와 같았다.

- 요청 body가 API 계약과 맞지 않아 의미 없는 실패가 발생했다.
- 서비스가 막 재기동된 직후 테스트가 시작돼 `nginx 502`가 발생했다.
- 응답 횟수만 보고 실제 데이터 생성 횟수를 판단하면 오판할 수 있었다.
- 잘못된 기준 재고 때문에 oversell false positive가 발생했다.

이번 작업을 하면서 가장 크게 느낀 점은, 부하 테스트는 숫자만 뽑는 작업이 아니라는 점이었다.  
어떤 결과를 성공으로 볼지, 그 기준이 실제 데이터와 맞는지, 그리고 그 결과를 어디까지 믿어도 되는지를 계속 검토해야 의미 있는 검증이 된다.

## 7. 최종 결과 정리

이번 작업을 통해 최종적으로 확인한 내용은 아래와 같다.

### Idempotency

- 동일 `Idempotency-Key` 동시 요청에서도 실제 주문은 1건만 생성됨
- 실제 재고 예약도 1건만 생성됨
- 나머지 요청은 동일 주문 재반환으로 처리됨

### Oversell

- 단일 상품 고부하 상황에서도 예약 성공 수가 시작 재고를 넘지 않음
- 재고는 정확히 0까지 소진됨
- 음수 재고는 발생하지 않음

결국 이번 작업은 단순히 테스트를 실행한 것이 아니라, 주문 시스템에서 가장 위험한 두 가지 문제를 실제 데이터 기준으로 검증하고, 필요한 부분은 코드와 테스트 스크립트 모두 수정하면서 정리한 경험이라고 말할 수 있다.

## 8. 마무리 문장 예시

이번 작업을 통해 동시성 문제는 단순히 API 응답만으로 판단할 수 없고, 테스트 설계와 DB 검증까지 함께 맞물려야 제대로 확인할 수 있다는 점을 배웠다.  
특히 중복 주문과 과판매처럼 실제 서비스에서 바로 문제가 될 수 있는 시나리오를 직접 검증하고, 코드와 테스트를 함께 수정해 최종적으로 정합성을 확인했다는 점에서 의미 있는 경험이었다.
