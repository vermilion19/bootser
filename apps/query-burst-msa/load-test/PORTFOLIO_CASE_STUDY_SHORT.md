# 동시성 상황에서 주문 정합성을 검증하고 개선한 경험

## 부제

k6 기반 부하 테스트로 확인한 중복 주문 방지와 과판매 대응

---

## 1. 프로젝트 소개

`query-burst-msa`는 주문, 재고, 랭킹, 분석 기능이 분리된 MSA 기반 프로젝트다.  
이번 작업에서는 단순 기능 테스트가 아니라, 동시 요청이 몰리는 상황에서도 주문과 재고 데이터가 깨지지 않는지를 확인하는 데 집중했다.

핵심 검증 대상은 두 가지였다.

- 같은 요청이 동시에 여러 번 들어와도 주문은 한 번만 생성되는가
- 재고보다 많은 주문이 들어와도 예약 성공 수는 실제 재고를 넘지 않는가

중복 주문과 과판매는 실제 서비스에서 바로 장애나 고객 이슈로 이어질 수 있기 때문에, 응답 결과만 보는 것이 아니라 최종적으로 DB를 직접 조회해 검증했다.

---

## 2. Idempotency 검증

첫 번째로 확인한 것은 `Idempotency`였다.  
같은 `Idempotency-Key`로 동시에 50개의 주문 요청이 들어왔을 때 실제 주문과 재고 예약이 각각 1건만 생성되는지를 보고 싶었다.

처음에는 테스트 결과가 곧바로 의미 있게 나오지 않았다.  
요청 body에 `unitPrice`가 빠져 API 계약과 맞지 않았고, order-service도 주문 레벨에서 같은 키를 기준으로 묶고 있지 않아 같은 예약을 바라보는 주문 row가 여러 개 생길 수 있는 상태였다.

이 문제를 해결하기 위해 아래 내용을 반영했다.

- `customer_order.idempotency_key` 컬럼 추가
- `idempotency_key` unique 제약 추가
- 동일 키 요청 시 기존 주문 재반환 처리
- Postgres advisory lock 기반 직렬화 적용
- 테스트 스크립트에 health check 대기 및 판정 보정 로직 추가

최종 실행에서는 50개의 동시 요청이 모두 정상 응답으로 처리됐고, 서버 에러도 발생하지 않았다.

권장 캡처: `Idempotency 부하 테스트 결과`

이후 DB를 직접 조회해 아래를 확인했다.

- `inventory_reservation where request_id = sharedKey` → `1건`
- `customer_order where idempotency_key = sharedKey` → `1건`

즉, 응답은 여러 번 반환됐지만 실제 데이터는 주문 1건, 예약 1건만 생성됐다.  
결과적으로 주문 서비스의 idempotency는 의도한 대로 보강되었다고 판단했다.

## 3. Oversell 검증

두 번째로 확인한 것은 `Oversell`이었다.  
하나의 상품에 대해 재고보다 훨씬 많은 주문이 동시에 들어왔을 때, 예약 성공 수가 실제 시작 재고를 초과하지 않는지를 확인했다.

테스트는 초당 300건씩 15초 동안 요청을 보내는 방식으로 구성했다.  
처음 결과만 보면 oversell이 발생한 것처럼 보였지만, 확인해보니 스크립트가 시작 재고를 50으로 하드코딩하고 있어서 잘못된 결론이 나온 것이었다.

이후 아래처럼 수정했다.

- health check 이후에만 테스트 시작
- 현재 `ACTIVE` 상품 중 재고가 가장 낮은 상품 선택
- 하드코딩된 기준 재고 제거
- 실제 상품 재고를 읽어 `기준 재고(실측)`으로 사용

최종 실행에서는 아래 결과를 확인했다.

- 핫 상품 ID: `99700`
- 시작 재고: `701`
- `STOCK_RESERVED`: `701`
- `REJECTED`: `3800`

권장 캡처: `Oversell 부하 테스트 결과`

DB 직접 조회 결과는 아래와 같았다.

- `catalog_product.stock = 0`
- `status = SOLD_OUT`
- `stock < 0` row 수 = `0`
- 해당 상품 기준 `RESERVED` 건수 = `701`

즉, 시작 재고만큼만 예약이 성공했고, 그 이후 요청은 정상적으로 거절되었다.  
음수 재고는 발생하지 않았고, oversell도 확인되지 않았다.

## 4. 정리

이번 작업을 통해 최종적으로 아래를 확인했다.

- 동일 `Idempotency-Key` 동시 요청에서도 실제 주문은 1건만 생성됨
- 실제 재고 예약도 1건만 생성됨
- 단일 상품 고부하 상황에서도 예약 성공 수는 시작 재고를 넘지 않음
- 재고는 정확히 0까지 소진되고, 음수 재고는 발생하지 않음

이번 경험에서 가장 중요했던 점은 테스트를 단순히 실행하는 데서 끝내지 않았다는 것이다.  
응답 결과만 보고 결론 내리지 않고, 테스트 스크립트의 전제조건을 수정하고, 서비스 코드도 함께 보완한 뒤, 최종적으로는 DB를 직접 조회해 데이터 정합성을 확인했다.

이 과정을 통해 동시성 문제는 단순한 API 성공 여부가 아니라, 실제 데이터 기준으로 검증해야 제대로 판단할 수 있다는 점을 분명하게 확인했다.
