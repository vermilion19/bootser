## 목표

`query-burst`를 한 번에 분해하지 않고, 운영 리스크가 낮은 영역부터 단계적으로 분리한다.

핵심 원칙:

- 읽기/집계 서비스 먼저 분리
- DB 공유 금지
- 이벤트 계약 먼저 안정화
- 주문-재고 쓰기 경계는 마지막에 재설계

## Phase 1. analytics-service 분리

### 목표

현재 `statistics` 패키지를 별도 애플리케이션으로 떼어낸다.

### 범위

- `StatisticsEventConsumer`
- `DailySalesSummary`
- `ProductDailySales`
- `StatisticsQueryService`
- `StatisticsController`

### TODO

- [ ] `statistics` 패키지를 신규 앱으로 이동할 설계안을 만든다
- [ ] 집계 전용 DB를 분리한다
- [ ] `order-events` 소비 설정을 신규 앱으로 이동한다
- [ ] 통계 API를 신규 서비스에서 직접 제공한다
- [ ] 기존 앱의 통계 API를 제거하거나 게이트웨이 경로로 전환한다
- [ ] 집계 데이터 초기 적재 방식(backfill)을 정의한다
- [ ] Consumer 멱등성 보완안을 반영한다

### 산출물

- `apps:query-burst-analytics` 또는 동등한 신규 앱
- 집계 DB 스키마
- 통계 API 라우팅 계획

### 완료 기준

- 주문 앱이 내려가도 통계 서비스는 독립 배포 가능하다
- 통계 서비스는 주문 서비스 DB에 직접 접근하지 않는다
- 기존 통계 API와 동등한 기능이 새 서비스에서 제공된다

### 리스크

- 멱등성 보완 없이 분리하면 중복/유실 이슈가 서비스 밖으로 확산된다
- backfill 전략이 없으면 운영 데이터 전환 시 조회 공백이 생긴다

### TODO

- [ ] 현재 비원자적인 `decrementSales()`를 보완한다

### 산출물

- `apps:query-burst-ranking` 또는 동등한 신규 앱
- Redis 운영 정책 문서
- 랭킹 조회 API 경로

### 완료 기준

- 실시간 랭킹은 주문 서비스와 독립 배포 가능하다
- Redis 장애/복구 시 랭킹 재구축 방식이 정의되어 있다
- 랭킹 계산 로직이 원자성 측면에서 현재보다 안전하다

### 리스크

- 알고리즘 자체의 비용 문제는 서비스 분리만으로 해결되지 않는다
- Redis 키 정책이 불명확하면 운영 중 메모리 사용량 추적이 어렵다

### TODO

- [ ] 재고 차감 책임을 분리 가능한 인터페이스로 추상화한다
- [ ] 동기 재고 예약 방식과 비동기 사가 방식 중 하나를 선택한다
- [ ] 재고 확보 실패 시 주문 처리 정책을 정의한다
- [ ] 분산 락/펜싱 토큰 책임이 어느 서비스로 이동할지 결정한다

### 산출물

- 주문-재고 경계 설계 문서
- 재고 예약 API 또는 이벤트 계약 초안

- 재고 확보 성공/실패의 소유 서비스가 명확하다

### 리스크

- 이 단계 없이 바로 분리하면 분산 트랜잭션 문제가 폭발한다
- 재고와 주문의 소유권이 불명확하면 장애 복구가 어려워진다

### 범위

- 카테고리 관리

### TODO

- [ ] 재고 예약 성공/실패 응답 계약을 구현한다
- [ ] 재고 이벤트 발행이 필요하면 outbox 전략을 적용한다
- [ ] 상품 조회 API 경로를 게이트웨이 기준으로 재정의한다

### 산출물

- `apps:query-burst-catalog` 또는 동등한 신규 앱
- catalog DB
- 주문-카탈로그 연동 구현

### 완료 기준

- 주문 서비스는 상품 DB를 직접 조회/수정하지 않는다
- 재고 차감은 catalog 서비스 책임으로 귀속된다
- 상품/카테고리 API는 독립 배포 가능하다

### 리스크

- 동기 호출을 선택하면 장애 전파 관리가 필요하다
- 비동기 사가를 선택하면 상태 머신과 보상 설계가 필요하다

- 앞 단계가 정리되지 않으면 order만 따로 정리해도 효과가 작다

## 최종 정리

이 실행 계획의 핵심은 "쉽게 떨어지는 read side부터 분리하고, write side는 마지막에 재설계한다"는 것이다.

가장 현실적인 다음 액션은 두 가지다.

1. `order-events` 계약 문서화
2. `statistics`를 별도 서비스로 분리하는 상세 설계 시작
