## 개요

현재 `query-burst`는 주문 서비스가 상품을 직접 조회하고 재고를 직접 차감한다.
이 구조는 모놀리식에서는 성립하지만, `catalog-service`를 분리하면 그대로 유지할 수 없다.

이 문서는 주문과 재고의 경계를 새로 정의하기 위한 설계 초안이다.

---

## 현재 구조 요약

현재 흐름:

1. `OrderFacade`가 상품별 Redis 분산 락 획득
2. `OrderService.createOrder()`가 `productRepository.findById()`로 상품 조회
3. `Product.decreaseStock()` 호출
4. 같은 트랜잭션 안에서 주문과 outbox 저장

fallback:

1. Redis 장애 발생
2. `findByIdWithLock()`로 DB 비관적 락
3. `decreaseStockFallback()` 호출

## 문제

`catalog-service` 분리 후에는 주문 서비스가 다음을 직접 할 수 없다.

- 상품 DB 직접 조회
- 재고 직접 차감
- 상품 엔티티 락 직접 제어

즉, 재고 확보는 별도 서비스 책임으로 이동해야 한다.

---

## 목표

- 주문 서비스와 카탈로그 서비스의 책임 경계를 명확히 한다
- 재고 확보 성공/실패를 서비스 간 계약으로 정의한다
- 모놀리식에서 사용하던 DB 공유 전제를 제거한다
- 향후 `catalog-service` 분리를 가능하게 한다

---

## 핵심 원칙

1. 주문 서비스는 재고를 직접 수정하지 않는다
2. 재고의 진실 소스는 catalog 쪽에 둔다
3. 재고 확보는 명시적인 API 또는 이벤트 계약으로 표현한다
4. 실패/보상 시나리오를 먼저 정의한다

---

## 선택지 비교

## 옵션 A. 동기 재고 예약 API

흐름:

1. `order-service`가 주문 요청 접수
2. `catalog-service`에 재고 예약 요청
3. 성공 시 주문 생성
4. 실패 시 주문 거절

### API 예시

```http
POST /internal/inventory/reservations
```

```json
{
  "requestId": "uuid",
  "orderId": 123456789,
  "memberId": 1001,
  "items": [
    {
      "productId": 2001,
      "quantity": 2
    }
  ]
}
```

성공 응답 예시:

```json
{
  "reservationId": "resv-001",
  "status": "RESERVED"
}
```

실패 응답 예시:

```json
{
  "status": "REJECTED",
  "reason": "INSUFFICIENT_STOCK"
}
```

장점:

- 구현이 단순하다
- 주문 성공/실패를 즉시 결정할 수 있다

단점:

- 서비스 간 동기 결합
- catalog 장애가 주문 경로에 직접 전파됨
- 타임아웃/재시도/중복 요청 관리가 필요

적합한 상황:

- 초기 분리 단계
- 주문량이 아주 크지 않은 상황

---

## 옵션 B. 비동기 사가 기반 예약

흐름:

1. `order-service`가 주문 요청 접수
2. `StockReserveRequested` 발행
3. `catalog-service`가 처리 후 `StockReserved` 또는 `StockRejected` 발행
4. `order-service`가 결과 이벤트에 따라 주문 확정 또는 실패 전이

장점:

- 느슨한 결합
- 서비스별 독립성이 높음
- 장애 격리에 유리

단점:

- 구현 복잡도 높음
- 주문 상태 머신이 필요함
- 최종 확정까지 시간이 걸림

적합한 상황:

- 높은 확장성 요구
- 이벤트 중심 운영 성숙도가 있는 경우

---

## 옵션 C. inventory-service 별도 분리

구조:

- `catalog-service`와 별도로 inventory만 독립

장점:

- 재고 정책이 아주 복잡할 때 적합

단점:

- 현재 규모에서는 과분할 가능성이 큼
- 서비스 하나가 더 늘어남

권장:

- 지금 당장은 선택하지 않는다

---

## 권장안

현 시점에서는 옵션 A를 먼저 채택하는 것이 현실적이다.

즉:

- 초기에는 동기 `reserve stock` API
- 경계가 안정되고 트래픽/조직 요구가 커지면 옵션 B로 확장

이 방식이 좋은 이유:

- 현재 주문 생성 API의 동기 UX를 유지할 수 있다
- 모놀리식 의존 제거를 가장 단순하게 달성할 수 있다
- 추후 비동기화할 때 계약을 진화시키기 쉽다

## 권장 책임 분리

### order-service 책임

- 주문 요청 접수
- 요청 멱등성 처리
- 주문 상태 관리
- outbox 발행
- 재고 예약 요청 호출

### catalog-service 책임

- 상품/카테고리 마스터
- 재고 보유
- 재고 예약/해제/확정
- 재고 관련 락/동시성 처리

---

## 제안하는 주문 상태 확장

현재는 `PENDING`, `PAID`, `SHIPPED`, `DELIVERED`, `CANCELED` 중심이다.
서비스 분리 후에는 예약 상태가 필요할 수 있다.

예시:

- `PENDING`
- `STOCK_RESERVED`
- `REJECTED`
- `PAID`
- `SHIPPED`
- `DELIVERED`
- `CANCELED`

초기에는 내부 상태만 늘리고 외부 API는 단순화할 수 있다.

## 재고 예약 API 설계 초안

### 1. 재고 예약

```http
POST /internal/inventory/reservations
```

request:

```json
{
  "requestId": "uuid",
  "orderId": 123456789,
  "items": [
    { "productId": 2001, "quantity": 2 },
    { "productId": 2002, "quantity": 1 }
  ]
}
```

response:

```json
{
  "reservationId": "resv-123",
  "status": "RESERVED"
}
```

### 2. 예약 해제

```http
POST /internal/inventory/reservations/{reservationId}/release
```

사용 시점:

- 주문 실패
- 결제 실패
- 취소

### 3. 예약 확정

```http
POST /internal/inventory/reservations/{reservationId}/commit
```

사용 시점:

- 주문이 최종 확정되는 시점

## 멱등성 정책

분리 후에도 중복 요청은 계속 발생할 수 있다.

권장:

- 주문 서비스는 `Idempotency-Key` 기준 멱등성 유지
- catalog 서비스는 `requestId` 또는 `reservationId` 기준 멱등성 유지

즉, 양쪽 서비스가 독립적으로 멱등성을 가져야 한다.

---

## 동시성 제어 책임

현재는 주문 서비스가:

- Redis 락
- 펜싱 토큰
- fallback

을 직접 관리한다.

분리 후 권장:

- 재고 동시성 제어는 catalog 서비스 내부 책임으로 이동
- 주문 서비스는 결과만 받는다

즉, 다음 항목은 catalog 쪽으로 이관하는 것이 맞다.

- `LOCK:product:*`
- `FENCE:product:*`
- `lastFenceToken`

## 실패 시나리오

### 시나리오 1. 재고 부족

- catalog 응답: `REJECTED`
- order 처리: 주문 실패 또는 생성 거절

### 시나리오 2. catalog 타임아웃

- order 처리: 재시도 또는 실패 응답
- 권장: 짧은 타임아웃과 명시적 에러 코드

### 시나리오 3. 주문 저장 실패, 재고는 이미 예약됨

- order 처리: 보상 호출로 예약 해제

### 시나리오 4. 예약 해제 실패

- 운영 보상 작업 필요
- retry queue 또는 관리자 API 필요

## 마이그레이션 단계

### 단계 1. 추상화

- 주문 서비스 내부에서 재고 차감 로직을 인터페이스로 추상화
- 직접 `productRepository`를 호출하는 경로를 감싼다

### 단계 2. 모놀리식 내부 어댑터

- 같은 애플리케이션 안에서 `InventoryReservationPort` 같은 형태로 분리
- 내부 구현은 기존 Product 기반 로직 유지

### 단계 3. 계약 확정

- 예약/해제/확정 API 또는 이벤트 계약 정의

### 단계 4. catalog 서비스 외부화

- 내부 어댑터를 HTTP/Kafka 기반 외부 어댑터로 교체

### 단계 5. 락 책임 이관

- Redis 락/펜싱 토큰 책임을 catalog 내부로 이동

## 최종 제안

`catalog-service` 분리의 본질은 상품 CRUD 분리가 아니라 "재고 소유권 이동"이다.
따라서 다음 실무 액션은 상품 서비스 분리보다 먼저 재고 예약 계약을 확정하는 것이다.

권장 1차 전략:

1. 동기 재고 예약 API 채택
2. 주문 서비스의 직접 재고 차감 제거
3. 이후 필요 시 이벤트 기반 사가로 확장
