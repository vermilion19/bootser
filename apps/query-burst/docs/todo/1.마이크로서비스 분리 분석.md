# query-burst 마이크로서비스 분리 분석

작성일: 2026-04-09

---

## 목적

현재 `query-burst`는 하나의 애플리케이션 안에 여러 도메인이 패키지 단위로 분리되어 있다.
이 문서는 현재 구조를 기준으로:

- 어떤 부분이 마이크로서비스로 분리하기 좋은지
- 어떤 부분은 아직 모놀리식으로 유지하는 편이 나은지
- 실제로 분리한다면 어떤 순서가 적절한지

를 정리한다.

---

## 현재 도메인 경계

현재 패키지 구조는 이미 다음과 같은 서비스 경계를 암시한다.

- `member`: 회원 관리
- `product` + `category`: 상품/카테고리 관리
- `order`: 주문 생성, 상태 전이, 재고 차감, 아웃박스 발행
- `ranking`: Redis 기반 실시간 랭킹
- `statistics`: CQRS 기반 집계 조회
- `lock`, `common.kafka`, `common.ratelimit`: 공통 인프라 성격

즉, 코드 구조만 보면 이미 "모듈형 모놀리식"에 가깝다.
문제는 `order`가 `member`, `product`, `outbox`, `lock`에 직접 결합되어 있다는 점이다.

---

## 분리 판단 기준

마이크로서비스 분리 우선순위는 다음 기준으로 보는 것이 합리적이다.

1. 다른 도메인 DB를 직접 수정하지 않아도 되는가
2. 이미 비동기 이벤트 기반으로 동작하고 있는가
3. 읽기 모델과 쓰기 모델이 분리되어 있는가
4. 독립 배포/독립 스케일링 가치가 큰가
5. 분리 시 분산 트랜잭션 복잡도가 급격히 커지지 않는가

이 기준으로 보면 `statistics`, `ranking`은 분리 가치가 높고, `order`는 가장 늦게 분리하는 편이 맞다.

---

## 추천 분리 우선순위

1. `analytics-service`
2. `ranking-service`
3. `catalog-service`
4. `member-service`
5. 마지막에 `order-service`

---

## 서비스 후보 분석

### 1. analytics-service

가장 먼저 분리하기 좋은 후보다.

이유:

- 이미 Kafka 이벤트를 소비해 자체 집계 테이블을 갱신한다
- 주문 원본 테이블을 직접 수정하지 않는다
- 읽기 모델이 별도 테이블로 존재한다
- 독립 배포와 독립 스케일링 가치가 높다

현재 포함 책임:

- 일별 카테고리 매출 집계
- 날짜별 상품 판매량 집계
- 상품별 기간 추이 조회

현재 구현 근거:

- `statistics/event/StatisticsEventConsumer`
- `statistics/domain/DailySalesSummary`
- `statistics/domain/ProductDailySales`
- `statistics/application/StatisticsQueryService`
- `statistics/web/StatisticsController`

분리 후 구조:

- 자체 DB 보유
- `order-events` 소비
- 집계용 read model만 유지
- 조회 API만 외부에 제공

장점:

- 주문 서비스와 완전히 다른 부하 패턴을 분리할 수 있다
- 읽기 최적화 모델을 독립 운영할 수 있다
- CQRS 구조와 잘 맞는다

주의점:

- 이벤트 중복/순서/재처리 정책을 별도 서비스 수준에서 확정해야 한다
- 지금은 `isDuplicate() -> 처리 -> markProcessed()` 방식이라 멱등성 보완이 필요하다

결론:

`query-burst`에서 가장 먼저 독립 서비스화할 만한 영역이다.

---

### 2. ranking-service

두 번째로 분리하기 좋은 후보다.

이유:

- 이미 이벤트 소비 기반으로 동작한다
- 주문 DB에 직접 쓰지 않는다
- Redis 중심 워크로드라 스케일링 특성이 주문/통계와 다르다

현재 포함 책임:

- 시간대별 판매량 누적
- 슬라이딩 윈도우 인기 상품 조회

현재 구현 근거:

- `ranking/event/RankingEventConsumer`
- `ranking/application/RankingService`
- `ranking/web/RankingController`

분리 후 구조:

- Redis를 주 저장소처럼 사용
- `order-events` 소비
- 랭킹 조회 API 제공

장점:

- 실시간 조회 트래픽과 주문 쓰기 트래픽을 분리할 수 있다
- Redis 운영 정책을 독립적으로 가져갈 수 있다
- 캐시/랭킹 만료 전략을 별도 서비스에서 실험하기 쉽다

주의점:

- 현재 `decrementSales()`는 비원자적이라 분리 전후 관계없이 보완이 필요하다
- 조회 복잡도는 여전히 Redis 키 합산 비용에 좌우되므로, 서비스 분리만으로 알고리즘 문제가 해결되지는 않는다

결론:

`analytics-service` 다음으로 분리 가치가 높다.

---

### 3. catalog-service

`product`와 `category`는 함께 가는 것이 맞다.

이유:

- 상품과 카테고리는 같은 도메인 언어를 공유한다
- 상품 조회/관리 API가 독립적이다
- 향후 검색, 가격 정책, 재고 정책이 커지면 분리 가치가 커진다

현재 포함 책임:

- 상품 CRUD
- 카테고리 계층 관리
- 가격 변경
- 상태 변경
- 재고 보유

현재 구현 근거:

- `product/application/ProductService`
- `product/application/CategoryService`
- `product/domain/Product`
- `product/domain/Category`
- `product/web/ProductController`
- `product/web/CategoryController`

분리 난이도가 높은 이유:

- 현재 주문 서비스가 상품 엔티티를 직접 조회하고 재고를 차감한다
- 분산 락과 펜싱 토큰도 사실상 상품 재고 보호를 위해 존재한다
- 즉, 지금 구조에서는 `order`와 `product`가 트랜잭션 경계를 공유하고 있다

분리 후 바뀌어야 하는 점:

- 주문이 상품 DB를 직접 수정하지 않아야 한다
- 재고 차감 책임은 `catalog-service` 또는 별도 `inventory-service`로 이동해야 한다
- 주문은 "재고 확보 요청"의 결과를 받아 주문을 확정해야 한다

가능한 방향:

- 동기 방식: `order-service -> catalog-service` 재고 예약 API 호출
- 비동기 방식: `StockReserveRequested -> StockReserved/StockRejected` 이벤트 흐름

장점:

- 상품/재고/카테고리 운영이 명확히 분리된다
- 카탈로그 조회 부하를 독립적으로 처리할 수 있다

주의점:

- 이 단계부터는 진짜 분산 시스템 문제를 다뤄야 한다
- 주문과 재고 사이에 사가, 보상, 타임아웃, 중복 처리 설계가 필요하다

결론:

분리 가치는 크지만, `analytics`/`ranking`보다 먼저 하면 위험하다.

---

### 4. member-service

분리는 가능하지만 우선순위는 높지 않다.

이유:

- 현재는 CRUD와 참조 데이터 성격이 강하다
- 주문/상품에서 `memberId`, `sellerId`를 통해 참조하지만, 복잡한 협업 로직은 많지 않다

현재 포함 책임:

- 회원 생성/수정/삭제
- 페이지네이션 기반 회원 조회

현재 구현 근거:

- `member/application/MemberService`
- `member/domain/Member`
- `member/web/MemberController`

장점:

- 사용자 마스터 데이터 책임이 명확해진다
- 인증/권한 시스템과 결합하기 쉬워진다

주의점:

- 지금 당장 병목 도메인은 아니다
- 회원 서비스만 따로 분리해도 주문/상품 서비스는 결국 회원 정보를 부분 복제하거나 API 호출해야 한다

결론:

조직 경계나 인증 체계 확장이 있을 때 분리하는 편이 낫다.

---

### 5. order-service

가장 나중에 분리하는 것이 맞다.

이유:

- 현재 시스템의 핵심 복잡성이 집중된 영역이다
- 멱등성, 분산 락, 펜싱 토큰, Redis fallback, Outbox가 모두 주문 경로 중심이다
- 다른 도메인과의 동기 의존도 가장 높다

현재 포함 책임:

- 주문 생성
- 상태 전이
- 재고 차감
- Producer 멱등성
- Outbox 저장 및 이벤트 발행

현재 구현 근거:

- `order/application/OrderFacade`
- `order/application/OrderService`
- `order/application/IdempotencyService`
- `order/event/OutboxMessageRelay`
- `lock/RedisDistributedLock`

왜 마지막인가:

- 먼저 분리하면 `product`, `member`, `statistics`, `ranking`과의 경계를 한 번에 모두 재설계해야 한다
- 트랜잭션 일관성을 잃고 보상 로직을 도입해야 한다
- 가장 많은 장애 시나리오가 생긴다

결론:

`order`는 분리의 종착점이지 출발점이 아니다.

## 추천 최종 서비스 구조

현실적인 목표 구조는 다음과 같다.

- `order-service`
  - 주문 생성
  - 주문 상태 전이
  - 아웃박스 발행
- `catalog-service`
  - 상품
  - 카테고리
  - 재고
- `member-service`
  - 회원 마스터 데이터
- `analytics-service`
  - 일별 매출 집계
  - 상품 추이
- `ranking-service`
  - Redis 기반 실시간 랭킹

필요하다면 이후에 `inventory-service`를 `catalog-service`에서 별도 분리할 수 있다.

## 이벤트 중심 분리 전략

현재 구조에서 서비스 분리를 가능하게 하는 가장 중요한 기반은 이미 존재한다.

- `order-events` 토픽
- Outbox 발행
- consumer 기반 read model 구축

이벤트 계약은 사실상 다음 필드가 핵심이다.

- `eventType`
- `orderId`
- `memberId`
- `totalAmount`
- `orderStatus`
- `occurredAt`
- `items[].productId`
- `items[].categoryId`
- `items[].quantity`
- `items[].unitPrice`

즉, `analytics-service`와 `ranking-service`는 주문 서비스 DB를 보지 않고도 독립적으로 동작할 수 있다.

## 가장 큰 리스크: 주문과 재고의 분리

현재 구조에서 마이크로서비스 전환 시 가장 어려운 지점은 주문과 재고다.

현재는:

1. 주문이 상품을 직접 조회한다
2. 주문이 재고를 직접 차감한다
3. 주문이 Redis 분산 락과 펜싱 토큰까지 관리한다

이 구조는 모놀리식에서는 강하지만, 서비스 분리 후에는 성립하지 않는다.

분리 후에는 다음 셋 중 하나로 가야 한다.

### 방식 A. catalog-service 동기 호출

- `order-service`가 `catalog-service`에 재고 확보 요청
- 성공 시 주문 생성
- 실패 시 주문 거절

장점:

- 이해하기 쉽다
- 일관성 모델이 명확하다

단점:

- 동기 결합이 생긴다
- 장애 전파 가능성이 크다

### 방식 B. 사가 기반 비동기 예약

- 주문 요청 접수
- `StockReserveRequested`
- `StockReserved` 또는 `StockRejected`
- 결과에 따라 주문 확정/실패

장점:

- 서비스 간 느슨한 결합
- 확장성 좋음

단점:

- 구현 복잡도 높음
- 결국 상태 머신과 보상 트랜잭션이 필요함

### 방식 C. inventory-service 별도 분리

- 재고만 독립 서비스로 분리
- 주문/카탈로그가 모두 inventory를 참조

장점:

- 재고 규칙이 복잡해질 때 적합

단점:

- 너무 일찍 도입하면 과분한 분해가 될 수 있음

현 시점 추천:

처음에는 `catalog-service` 안에 재고를 두고, 정말 재고 정책이 복잡해질 때만 별도 분리한다.

## 단계별 마이그레이션 권장안

### 1단계

`analytics-service` 분리

- `statistics` 패키지를 별도 앱으로 이동
- 자체 DB 사용
- `order-events` 구독 유지
- 기존 주문 앱에서는 통계 API 제거 또는 프록시 전환

### 2단계

`ranking-service` 분리

- `ranking` 패키지를 별도 앱으로 이동
- Redis 키 관리 책임 이관
- 실시간 랭킹 API를 독립 제공

### 3단계

`catalog-service` 분리 준비

- 주문 서비스가 상품 엔티티를 직접 수정하지 않도록 추상화
- 재고 차감 로직을 명시적 인터페이스로 분리
- 동기/비동기 재고 확보 전략 결정

### 4단계

`catalog-service` 분리

- 상품/카테고리/재고 DB 분리
- 주문 서비스와 재고 확보 프로토콜 연결

### 5단계

필요 시 `member-service` 분리

- 회원 마스터 데이터 API 제공
- 다른 서비스는 member snapshot 또는 캐시 활용

### 6단계

마지막으로 `order-service`를 중심 쓰기 서비스로 정리

- 진짜 서비스 경계 기준으로 아웃박스/멱등성/사가 재설계

## 지금 당장 하지 말아야 할 것

- `order`와 `product`를 한 번에 찢기
- DB를 공유한 채 서비스만 나누기
- `member-service`를 먼저 떼고 핵심 쓰기 경로는 그대로 두기
- Outbox 없이 이벤트 발행부터 직접 하도록 바꾸기

이런 접근은 운영 복잡도만 늘리고 아키텍처 이득은 작을 가능성이 높다.

## 최종 결론

`query-burst`는 마이크로서비스 후보가 분명한 구조다.
다만 모든 도메인을 한 번에 나누는 방식은 좋지 않다.

가장 좋은 출발점:

- `analytics-service`
- `ranking-service`

그 다음 후보:

- `catalog-service`

후순위:

- `member-service`

가장 마지막:

- `order-service`

즉, 이 프로젝트는 "읽기/집계 서비스 먼저 분리하고, 쓰기 코어는 마지막에 재설계"하는 전략이 가장 안전하다.
