## 개요

현재 `query-burst` 내부의 `ranking` 패키지를 별도 `ranking-service`로 분리한다.

이 서비스의 역할은 다음 두 가지다.

- 주문 이벤트를 소비해 Redis 기반 랭킹 데이터를 유지
- 슬라이딩 윈도우 기준 인기 상품 조회 API 제공

이 서비스는 주문 DB를 소유하지 않는다.
랭킹은 이벤트 기반 read model로 취급한다.

---

## 배경 (Why)

랭킹 기능은 주문 코어와는 다른 특성을 가진다.

- 데이터 저장소가 Redis 중심이다
- 실시간 조회 트래픽이 발생할 수 있다
- 주문 쓰기 경로와 스케일링 기준이 다르다
- 장애 시 랭킹만 별도로 degrade 시키는 것이 가능해야 한다

현재도 이미 이벤트 소비 기반이므로 `analytics-service` 다음 분리 후보로 적합하다.

---

## 목표

- 랭킹 기능을 별도 배포 단위로 분리
- Redis 키 관리 책임을 독립 서비스로 이동
- `order-events` 기반으로 랭킹을 유지
- 기존 `/api/rankings/realtime` 기능을 유지

## 기능 요구사항

### FR-1. 주문 이벤트 소비

- `order-events` 토픽을 소비해야 한다
- `ORDER_CREATED` 수신 시 상품 판매량을 증가시켜야 한다
- `ORDER_CANCELED` 수신 시 상품 판매량을 감소시켜야 한다
- 지원하지 않는 이벤트 타입은 무시해야 한다

### FR-2. 시간대별 랭킹 유지

- 시간 단위 key로 상품 판매량을 누적해야 한다
- key TTL은 현재 정책과 동일하게 25시간 수준을 유지한다

### FR-3. 슬라이딩 윈도우 조회

- `windowHours` 기준으로 최근 N시간 랭킹을 계산해야 한다
- 조회 파라미터는 최소 현재 API와 호환돼야 한다

### FR-4. Top N 조회 API 제공

- `GET /api/rankings/realtime`
- `windowHours`
- `size`

를 지원해야 한다

### FR-5. 재구축 가능성

- Redis 유실 또는 초기 배포 시 랭킹을 재구축할 수 있는 운영 절차가 필요하다

---

## 비기능 요구사항

### NFR-1. 독립 저장소

- 랭킹 데이터는 Redis에만 저장한다
- 주문 DB 직접 조회를 런타임 경로에 두지 않는다

### NFR-2. Eventually Consistent

- 주문 직후 즉시 반영이 아니어도 된다
- 최종적으로 최근 이벤트가 반영돼야 한다

### NFR-3. 멱등성

- 중복 이벤트 수신 시 랭킹 점수가 왜곡되지 않아야 한다

### NFR-4. 관측 가능성

- 소비 지연
- 이벤트 처리 성공/실패 수
- Redis key 수
- 랭킹 조회 지연

을 추적할 수 있어야 한다

## 저장 모델

### 1. 시간대별 집계 키

패턴:

- `RANK:hourly:{yyyyMMddHH}`

의미:

- 특정 시간대의 상품별 누적 판매량

member:

- `productId`

score:

- 해당 시간대 누적 판매량

TTL:

- 25시간

### 2. 조회용 임시 키

패턴:

- `RANK:window:{windowHours}h:{timestamp}`

의미:

- 여러 시간대 key를 합산한 임시 결과

현재 구현:

- 조회 시 생성
- 조회 직후 삭제

---

## 현재 구현의 제약

### 제약 1. 감소 연산 비원자성

현재 `decrementSales()`는 `ZSCORE -> 계산 -> ZADD` 방식이다.
동시 증가/감소가 섞이면 덮어쓰기 위험이 있다.

권장:

- 음수 `ZINCRBY` 기반으로 변경
- 필요 시 Lua Script로 음수 방지 포함

### 제약 2. 조회 비용

현재 윈도우 집계는 각 시간대 key의 전체 엔트리를 순회해 합산한다.
즉, 문서 수준에서 단순 `O(log N)`이라고 표현하기는 어렵다.

분리 후에도 이 비용 구조는 그대로 남는다.

### 제약 3. 재구축 전략 부재

Redis만 날아가면 랭킹은 재생성 전까지 비어 있을 수 있다.
초기 분리 시 운영 절차가 필요하다.

## API 설계

### 실시간 랭킹 조회

```http
GET /api/rankings/realtime?windowHours=6&size=10
```

응답 예시:

```json
[
  {
    "productId": 2001,
    "score": 152.0,
    "rank": 1
  },
  {
    "productId": 2002,
    "score": 131.0,
    "rank": 2
  }
]
```

검증 규칙:

- `windowHours`는 1~24
- `size`는 운영 정책상 상한 설정 권장

---

## 이벤트 처리 규칙

### ORDER_CREATED

입력:

- `items[].productId`
- `items[].quantity`

처리:

- 현재 시각 기준 시간대 key에 수량 가산

### ORDER_CANCELED

입력:

- `items[].productId`
- `items[].quantity`

처리:

- 현재 시각 기준 시간대 key에 수량 차감

주의:

- 현재 구현은 취소 시 `occurredAt`가 아니라 처리 시점 `now()` 기준 시간대에 반영한다
- 이 의미가 맞는지 도메인적으로 재검토할 필요가 있다

---

## 서비스 경계

### ranking-service가 소유하는 것

- Redis 랭킹 key
- 랭킹 조회 API
- 주문 이벤트 소비 로직

### ranking-service가 소유하지 않는 것

- 상품 원본 정보
- 주문 원본 정보
- 통계용 DB

즉, 이 서비스는 product metadata 없이 `productId`만으로도 동작 가능해야 한다.
상품명 같은 표시 정보가 필요하면 별도 조회 또는 캐시 전략이 필요하다.

## 마이그레이션 전략

### 단계 1. 코드 분리

- `ranking` 패키지를 신규 앱으로 이동

### 단계 2. Redis 키 정책 확정

- prefix
- TTL
- 임시 key 삭제 정책
- 장애 시 flush/rebuild 절차

### 단계 3. Kafka consumer 이관

- `order-events` 소비 설정 이동
- consumer group 정책 유지

### 단계 4. API 전환

- `/api/rankings/**`를 신규 서비스로 라우팅

### 단계 5. 재구축 절차 수립

옵션:

- A. 랭킹은 휘발성으로 간주하고 자연 회복
- B. 최근 이벤트 replay로 복구
- C. analytics 데이터 기반으로 보조 재생성

권장:

- 초기에는 A 또는 B
- 운영 요구가 커지면 C 검토

## 리스크

### R-1. 중복 이벤트

원인:

- 현재 consumer 멱등성 한계

대응:

- 멱등성 개선

### R-2. 감소 연산 경합

원인:

- 현재 `decrementSales()` 비원자성

대응:

- 음수 `ZINCRBY` 또는 Lua Script

### R-3. Redis 유실

원인:

- Redis는 영속 저장소가 아님

대응:

- 재구축 절차 문서화

### R-4. 조회 비용 증가

원인:

- 윈도우 조회 시 다수 key 엔트리 합산

대응:

- 조회 패턴 모니터링
- 필요 시 pre-aggregation 또는 별도 캐시
