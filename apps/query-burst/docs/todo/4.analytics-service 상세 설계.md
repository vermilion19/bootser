## 개요

현재 `query-burst` 내부의 `statistics` 패키지를 별도 `analytics-service`로 분리한다.

이 서비스의 역할은 다음 두 가지다.

- 주문 이벤트를 소비해 집계용 read model 유지
- 집계 조회 API 제공

이 서비스는 주문 원본 데이터를 직접 소유하지 않는다.
오직 이벤트 기반으로 자체 집계 데이터를 구축하고 제공한다.

---

## 배경 (Why)

현재 통계 기능은 도메인상으로는 주문 쓰기 흐름과 분리되어 있다.
하지만 물리적으로는 같은 애플리케이션 안에 있어 다음 문제가 있다.

- 주문 서비스와 통계 서비스가 같은 배포 단위를 공유한다
- 읽기 부하와 쓰기 부하를 별도로 스케일링하기 어렵다
- 향후 서비스 분리의 첫 단계가 되는 read side를 독립 운영할 수 없다

현재 구조는 이미 CQRS read model 성격을 띠므로, 분리 비용 대비 효과가 가장 큰 영역이다.

---

## 목표

- 주문 서비스와 집계 서비스를 배포 단위로 분리
- 집계 데이터 저장소를 독립 DB로 분리
- `order-events`만으로 집계 read model을 유지
- 기존 통계 API 기능을 유지

## 기능 요구사항

### FR-1. 주문 이벤트 소비

- `order-events` 토픽을 소비해야 한다
- `ORDER_CREATED` 수신 시 집계를 누적해야 한다
- `ORDER_CANCELED` 수신 시 집계를 역산해야 한다
- 지원하지 않는 이벤트 타입은 안전하게 무시해야 한다

### FR-2. 일별 카테고리 매출 조회

- 날짜 범위 기준으로 `daily_sales_summary` 조회 API를 제공해야 한다

### FR-3. 날짜별 상품 판매 TOP 조회

- 특정 날짜의 상품 판매량 TOP N 조회 API를 제공해야 한다

### FR-4. 상품별 기간 추이 조회

- 특정 상품의 기간별 판매량/매출 추이 조회 API를 제공해야 한다

### FR-5. 재처리 가능성 고려

- 같은 이벤트가 중복 전달될 수 있음을 전제로 설계해야 한다
- 재배포/재시작 시 집계 손상이 없어야 한다

### FR-6. 초기 적재

- 신규 서비스 최초 배포 시 과거 주문 이벤트 또는 DB 백필 중 하나로 집계 데이터를 초기화할 수 있어야 한다

---

## 비기능 요구사항

### NFR-1. 독립 데이터 저장

- 주문 서비스 DB와 별도 DB를 사용해야 한다
- 직접 cross-db query를 금지한다

### NFR-2. Eventual Consistency

- 통계는 주문 생성 직후 즉시 반영되지 않아도 된다
- 대신 최종 일관성을 보장해야 한다

### NFR-3. 멱등성

- 중복 이벤트 수신 시 집계가 이중 반영되지 않아야 한다

### NFR-4. 관측 가능성

- consumer lag
- 처리 성공/실패 건수
- 집계 반영 시간
- 재처리 건수

를 추적할 수 있어야 한다

## 데이터 모델

### 1. daily_sales_summary

역할:

- 날짜 + 카테고리 기준 집계

주요 컬럼:

- `date`
- `category_id`
- `total_amount`
- `order_count`

### 2. product_daily_sales

역할:

- 날짜 + 상품 기준 집계

주요 컬럼:

- `date`
- `product_id`
- `sold_count`
- `revenue`

현재와 동일하게 unique 인덱스를 유지하는 것이 전제다.

---

## 이벤트 처리 규칙

### ORDER_CREATED

입력:

- `occurredAt`
- `items[].categoryId`
- `items[].productId`
- `items[].quantity`
- `items[].unitPrice`

처리:

- `occurredAt.toLocalDate()` 기준으로 집계 일자 결정
- 카테고리 집계 누적
- 상품 집계 누적

### ORDER_CANCELED

입력:

- 생성 때와 동일 필드

처리:

- 같은 날짜 기준으로 카테고리 집계 차감
- 같은 날짜 기준으로 상품 집계 차감
- 음수 방지는 현재 구현 정책을 따른다

## API 설계

### 1. 일별 매출 조회

```http
GET /api/statistics/daily-sales?from=2026-04-01&to=2026-04-09
```

응답 예시:

```json
[
  {
    "date": "2026-04-01",
    "categoryId": 301,
    "totalAmount": 1250000,
    "orderCount": 42
  }
]
```

### 2. 날짜별 상품 TOP 조회

```http
GET /api/statistics/top-products?date=2026-04-09
```

응답 예시:

```json
[
  {
    "date": "2026-04-09",
    "productId": 2001,
    "soldCount": 120,
    "revenue": 3400000
  }
]
```

### 3. 상품별 기간 추이

```http
GET /api/statistics/products/2001/trend?from=2026-04-01&to=2026-04-09
```

## 서비스 경계

### analytics-service가 소유하는 것

- 집계용 DB
- 통계 API
- 주문 이벤트 소비 로직

### analytics-service가 소유하지 않는 것

- 주문 원본 데이터
- 회원 원본 데이터
- 상품 원본 데이터
- 재고

필요한 참조 데이터는 이벤트 payload 안의 snapshot 수준 정보만 사용한다.

## 마이그레이션 전략

### 단계 1. 코드 분리

- `statistics` 패키지를 신규 앱으로 복사/이동
- 공통 라이브러리 의존성을 최소화

### 단계 2. DB 분리

- analytics 전용 DB 생성
- 집계 테이블 스키마 이관

### 단계 3. 이벤트 소비 연결

- `order-events` consumer 설정 이관
- 소비 그룹 이름 유지 또는 신규 이름 부여

### 단계 4. 백필

선택지:

- A. 과거 주문 DB를 한 번 읽어 집계 테이블 생성
- B. 과거 이벤트 replay

현실적으로는 A가 더 단순할 가능성이 높다.

### 단계 5. API 전환

- 게이트웨이 또는 라우팅 레벨에서 `/api/statistics/**`를 analytics-service로 연결

### 단계 6. 기존 코드 제거

- 모놀리식 내부 `statistics` 패키지 제거 또는 deprecated 처리

---

## 백필 전략 제안

### 옵션 A. DB 기반 백필

방법:

- 기존 `orders`, `order_item`, `product`를 읽어 집계 테이블 생성

장점:

- 한 번성 작업으로 단순함

단점:

- 신규 서비스가 초기 적재 시점에 한시적으로 주문 DB 접근이 필요할 수 있음

### 옵션 B. 이벤트 replay

방법:

- 과거 `order-events`를 다시 소비해 집계 재구성

장점:

- 이벤트 기반 아키텍처 원칙에 더 잘 맞음

단점:

- retention, 재생 범위, 재처리 속도 문제를 해결해야 함

권장:

- 초기 1회는 DB 기반 백필
- 이후 운영 재구축은 replay 또는 snapshot 전략 추가 검토

## 리스크

### R-1. 중복 집계

원인:

- 현재 consumer 멱등성 구현 한계

대응:

- 선점 기반 멱등성 또는 after-commit 마킹 설계 반영

### R-2. 초기 데이터 공백

원인:

- 백필 없이 서비스만 먼저 띄우는 경우

대응:

- cut-over 전 백필 완료 필수

### R-3. 이벤트 계약 변경

원인:

- 주문 서비스 payload 변경

대응:

- `order-events` 계약 문서와 버전 정책 유지

### R-4. 운영 추적 어려움

원인:

- 집계 지연, 실패, 재처리를 관측하지 못함

대응:

- lag, 처리 건수, 실패 건수, 지연 시간 메트릭 추가

---

## 완료 기준

- `analytics-service`가 주문 서비스와 독립 배포 가능하다
- 통계 API가 신규 서비스에서 제공된다
- 주문 서비스 DB를 런타임에 직접 조회하지 않는다
- 백필 후 기존 통계 결과와 샘플 비교가 가능하다
- 멱등성/재처리 정책이 문서화되어 있다
