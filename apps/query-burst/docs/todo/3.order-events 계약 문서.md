# query-burst order-events 계약 문서

작성일: 2026-04-09

---

## 개요

`order-events`는 주문 도메인에서 발행하는 핵심 통합 이벤트 토픽이다.
현재 구현에서 이 토픽은 다음 읽기/집계 계층이 공통으로 소비한다.

- `statistics-consumer-group`
- `ranking-consumer-group`

향후 `analytics-service`, `ranking-service`, `notification-service` 같은 분리된 서비스들도 이 계약을 기준으로 연동하게 된다.

주의:

- 이 문서는 `query-burst`의 전체 이벤트 시스템 문서가 아니다
- 현재 범위는 Kafka 토픽 `order-events`의 계약만 다룬다
- 플래시 세일 전용 비동기 주문 생성 흐름은 별도 토픽 `flash-sale-orders`를 사용하며, 이는 별도 문서에서 다룬다

---

## 목적

- 주문 생성/취소/상태 변경 사실을 외부 소비자에게 전달
- 통계/랭킹 같은 읽기 모델이 주문 DB에 직접 접근하지 않도록 지원
- 서비스 분리 시 공용 이벤트 계약의 기준점 제공

---

## Producer

현재 producer는 주문 서비스 내부의 Outbox 패턴을 통해 구현된다.

흐름:

1. `OrderService`가 주문 생성 또는 상태 변경 시 `OutboxEvent`를 저장
2. `OutboxMessageRelay`가 `PENDING` 이벤트를 주기적으로 조회
3. Kafka `order-events` 토픽에 payload를 발행
4. 성공 시 `PUBLISHED`, 실패 누적 시 `FAILED`

관련 구현:

- [OrderService.java](C:/Users/NCand/Documents/bootser/apps/query-burst/src/main/java/com/booster/queryburst/order/application/OrderService.java)
- [OutboxEvent.java](C:/Users/NCand/Documents/bootser/apps/query-burst/src/main/java/com/booster/queryburst/order/domain/outbox/OutboxEvent.java)
- [OutboxMessageRelay.java](C:/Users/NCand/Documents/bootser/apps/query-burst/src/main/java/com/booster/queryburst/order/event/OutboxMessageRelay.java)

비고:

- 플래시 세일 주문 생성 요청은 `OutboxMessageRelay`를 통하지 않고 `flash-sale-orders` 토픽으로 직접 발행된다
- 따라서 `order-events`는 "주문 도메인 전체의 모든 이벤트"가 아니라 "주문 read model 반영용 통합 이벤트"에 더 가깝다

## Consumer

현재 consumer는 두 개다.

### statistics-consumer-group

역할:

- `ORDER_CREATED` 수신 시 집계 테이블 누적
- `ORDER_CANCELED` 수신 시 집계 테이블 역산

관련 구현:

### ranking-consumer-group

역할:

- `ORDER_CREATED` 수신 시 Redis 랭킹 증가

## 전달 보장

현재 구조의 전달 보장은 다음과 같다.

- Outbox 기반 발행
- Kafka 발행 성공 후 `PUBLISHED` 마킹
- 전체적으로 `At-Least-Once`

즉, 중복 전달 가능성을 전제로 소비자가 멱등성을 가져야 한다.

주의:

- 현재 consumer 멱등성 구현은 `isDuplicate() -> 처리 -> markProcessed()` 방식이다
- 따라서 TOCTOU와 커밋 타이밍 이슈가 존재할 수 있다
- 이 계약 문서는 현 상태를 설명하며, 소비자 구현이 완전한 exactly-once를 보장한다는 의미는 아니다

---

## 토픽 정보

- Topic: `order-events`
- Key: `String.valueOf(orderId)`
- Value: `OrderEventPayload`

키를 `orderId`로 두므로, 동일 주문 기준의 이벤트는 파티션 전략에 따라 순서 보장을 기대할 수 있다.
다만 파티션 수 변경, 재처리, consumer 재시작까지 포함한 전역 순서 보장을 의미하지는 않는다.

## 이벤트 타입

현재 코드상 정의된 이벤트 타입:

- `ORDER_CREATED`
- `ORDER_CANCELED`
- `ORDER_STATUS_CHANGED`

실제 소비 여부:

- `statistics-consumer-group`: `ORDER_CREATED`, `ORDER_CANCELED`만 처리
- `ranking-consumer-group`: `ORDER_CREATED`, `ORDER_CANCELED`만 처리
- `ORDER_STATUS_CHANGED`는 현재 두 consumer 모두 무시

## 이벤트별 의미

### ORDER_CREATED

의미:

- 주문이 생성되었고, outbox에 기록되어 외부 read model이 반영 가능한 상태

현재 payload 특성:

- `items` 포함
- 통계와 랭킹 계산에 필요한 최소 필드를 모두 포함

사용 소비자:

- statistics
- ranking

### ORDER_CANCELED

의미:

- 주문이 취소되었고, 기존 집계/랭킹을 되돌릴 필요가 있음

현재 payload 특성:

- `items` 포함
- 생성 시 집계했던 항목을 역산할 수 있도록 구성

사용 소비자:

- statistics
- ranking

### ORDER_STATUS_CHANGED

의미:

- 결제, 배송, 배송완료 같은 상태 전이 발생

현재 payload 특성:

- `items`는 빈 리스트일 수 있음
- 현재 통계/랭킹 소비자는 사용하지 않음

예상 용도:

- 알림
- 주문 타임라인
- 운영 모니터링

## 이벤트 플로우

```text
Client
  -> OrderController
  -> OrderFacade
  -> OrderService
     -> Orders / OrderItem 저장
     -> OutboxEvent 저장

OutboxMessageRelay
  -> outbox_event(PENDING) 조회
  -> Kafka publish(order-events)
  -> 성공 시 PUBLISHED

Kafka order-events
  -> statistics-consumer-group
     -> 집계 테이블 갱신
  -> ranking-consumer-group
     -> Redis 랭킹 갱신
```

## 멱등성 규칙

소비자는 다음 키 기준으로 중복 처리를 판단한다.

`CONSUMER:{groupId}:{orderId}:{eventType}`

의미:

- consumer group마다 독립된 멱등성 범위
- 같은 주문이라도 consumer group이 다르면 서로 영향 없음

현재 한계:

- 사후 마킹 방식
- 트랜잭션 커밋 전 마킹 가능성
- 동시 처리 선점 부재

따라서 이 계약을 사용하는 새 서비스는 "중복 수신 가능"을 기본 전제로 설계하는 것이 안전하다.

## 호환성 정책

현재는 별도 버전 필드가 없다.
서비스 분리를 고려하면 다음 정책이 필요하다.

권장:

- `eventVersion` 필드 추가
- 필드 삭제보다 추가 중심의 하위 호환 유지
- 신규 consumer는 미지원 `eventType`을 무시
- `items`를 비워 보내는 경우를 허용하는 규칙 명문화

## 결론

`order-events`는 `query-burst`를 마이크로서비스로 나눌 때 가장 중요한 공용 계약이다.
