## 개요

이 문서는 두 가지를 함께 다룬다.

- `member-service`를 별도 서비스로 분리할지에 대한 상세 판단
- 마지막 단계에서 `order-service`를 어떤 책임으로 재정의할지에 대한 설계 방향

이 둘을 함께 다루는 이유는, `member`는 분리 우선순위가 낮고 `order`는 분리 우선순위가 가장 늦기 때문이다.
즉, 둘 다 "후반부 정리 대상"이라는 공통점이 있다.

---

## Part 1. member-service 상세 설계

## 배경 (Why)

현재 회원 도메인은 비교적 단순하다.

- 회원 생성/수정/삭제
- 페이지네이션 조회
- 주문자/판매자 참조 데이터

즉, 독립 서비스로 만들 수는 있지만 지금 당장의 병목 도메인은 아니다.

---

## 목표

- 회원 마스터 데이터를 별도 서비스로 소유할 수 있게 설계
- 향후 인증/권한 시스템과 결합 가능한 기반 제공

---

## 비목표

- 인증 서비스 구현
- OAuth, 세션, 토큰 발급
- 권한 시스템 구축

---

## 현재 범위

- `member/domain/Member`
- `member/application/MemberService`
- `member/web/MemberController`

---

## 기능 요구사항

- 회원 생성
- 회원 수정
- 회원 삭제
- Page/Slice/Cursor 기반 회원 조회
- 다른 서비스가 참조 가능한 member master 제공

---

## 비기능 요구사항

- 회원 DB 독립 소유
- 다른 서비스의 direct join 금지
- 참조 전략 문서화

---

## 서비스 경계

### member-service가 소유하는 것

- 회원 마스터 데이터
- 회원 조회 API

### member-service가 소유하지 않는 것

- 주문
- 상품
- 재고

---

## 다른 서비스와의 연동

### order-service

- 주문 생성 시 `memberId` 유효성 검증이 필요할 수 있다

### catalog-service

- `sellerId` 유효성 검증이 필요할 수 있다

권장:

- 초기에는 동기 조회 또는 로컬 캐시
- 이후 필요 시 member snapshot 이벤트 검토

---

## 분리 가치 평가

### 장점

- 회원 데이터 소유권 명확화
- 인증/권한과 연동하기 쉬움
- 조직 경계가 생기면 독립성 확보 가능

### 단점

- 지금 당장 성능 이득이 크지 않음
- 주문/상품 쪽에서 member 참조 방식이 추가로 필요

결론:

- 지금 바로 분리할 이유는 약하다
- 조직 경계 또는 인증 확장 요구가 생기면 분리하는 편이 낫다

---

## member-service 완료 기준

- 회원 DB가 독립 서비스에 귀속된다
- 다른 서비스가 member DB를 직접 조회하지 않는다
- member 참조 전략이 API 또는 캐시로 정리된다

---

## Part 2. order-service 재정의 설계

## 배경 (Why)

현재 `order`는 단순 주문 도메인이 아니라, 시스템 전체의 쓰기 오케스트레이터 역할을 하고 있다.

포함 책임:

- 주문 생성
- 주문 상태 전이
- 요청 멱등성
- 재고 차감 조율
- 분산 락
- Redis fallback
- outbox 저장
- 이벤트 발행

이 상태로는 "주문 서비스"라기보다 "주문 중심 조율 계층"이다.

서비스 분리 후에는 이 책임을 더 명확히 재정의해야 한다.

---

## 목표

- order-service를 중심 쓰기 서비스로 재정의
- 다른 서비스 DB 직접 접근 제거
- 주문 상태 머신을 서비스 경계 기준으로 재정리

---

## order-service가 최종적으로 가져야 할 책임

- 주문 요청 접수
- 요청 멱등성 보장
- 주문 상태 관리
- 재고 예약 요청 조율
- 결제/배송 등 후속 상태 전이
- 도메인 이벤트 발행

---

## order-service가 버려야 할 책임

- 상품 DB 직접 조회
- 상품 DB 직접 수정
- 상품 락 직접 제어
- 재고 동시성 세부 구현

즉, 현재 `OrderService` 안의 재고 차감 구현은 서비스 분리 완료 시점에는 제거되어야 한다.

---

## 제안 상태 머신

현재:

- `PENDING`
- `PAID`
- `SHIPPED`
- `DELIVERED`
- `CANCELED`

분리 후 후보:

- `PENDING`
- `STOCK_RESERVING`
- `STOCK_RESERVED`
- `REJECTED`
- `PAID`
- `SHIPPED`
- `DELIVERED`
- `CANCELED`

이렇게 되면 주문 서비스는 "상품 재고 확보 전/후"를 명시적으로 표현할 수 있다.

---

## 제안 이벤트

현재 `order-events`는 읽기 모델 중심 이벤트다.
향후 order-service를 중심 서비스로 만들려면 내부/외부 이벤트를 구분하는 것도 고려할 수 있다.

후보:

- `OrderCreated`
- `OrderRejected`
- `OrderPaid`
- `OrderShipped`
- `OrderDelivered`
- `OrderCanceled`

초기에는 현재 이벤트를 유지해도 되지만, 장기적으로는 더 명확한 이벤트 네이밍이 좋다.

---

## 멱등성 전략

현재:

- `Idempotency-Key`
- Redis 상태 저장
- `PROCESSING` / `COMPLETED`

분리 후에도 order-service는 클라이언트 요청 멱등성의 1차 책임을 유지해야 한다.

추가로 필요한 것:

- 외부 서비스 호출의 멱등성 전파
- 재고 예약 `requestId`와 연계

즉:

- API 멱등성
- 내부 명령 멱등성

둘 다 필요하다.

---

## Outbox 전략

order-service는 분리 후에도 Outbox를 유지하는 것이 맞다.

이유:

- 주문 상태 변경은 여러 하위 소비자에게 전달될 가능성이 크다
- analytics, ranking, notification, audit 등의 소비자가 늘어날 수 있다

즉, Outbox는 분리 이후에도 core pattern으로 유지해야 한다.

---

## 마이그레이션 전략

### 단계 1. 현재 책임 분리

- `OrderFacade`의 조율 책임과 `OrderService`의 도메인 책임을 더 명확히 나눈다

### 단계 2. 재고 의존 제거

- 직접 `productRepository` 접근 제거

### 단계 3. 상태 머신 확장

- 재고 예약 단계 반영

### 단계 4. 외부 호출/이벤트 계약 정리

- catalog와의 계약 확정

### 단계 5. order 중심 이벤트 재정비

- 외부 공개 이벤트와 내부 도메인 이벤트를 구분

---

## 리스크

### R-1. 책임 과다

원인:

- 주문 서비스가 계속 재고 세부사항까지 품는 경우

대응:

- 재고 책임 명시적 이관

### R-2. 상태 머신 복잡화

원인:

- 분리 후 주문 상태가 늘어남

대응:

- 공개 상태와 내부 상태를 구분

### R-3. 외부 서비스 장애 전파

원인:

- catalog 연동 실패

대응:

- timeout, retry, 보상, fallback

---

## 최종 우선순위 판단

### member-service

- 분리 가능
- 하지만 우선순위는 낮음

### order-service 재정의

- 반드시 해야 함
- 하지만 가장 마지막 단계

즉, `member`는 "필요하면 분리", `order`는 "마지막에 반드시 재정의"가 맞다.

---

## 완료 기준

- member-service 분리 여부가 명확한 기준으로 결정된다
- order-service의 최종 책임 경계가 문서화된다
- 주문 서비스가 다른 서비스 DB 직접 접근을 제거할 계획을 갖는다

---

## 구현 전 체크리스트

- [ ] member 분리 필요성을 조직/인증 요구와 함께 재평가
- [ ] order 상태 머신 확장 필요성 결정
- [ ] order 외부 이벤트 네이밍 개선 여부 검토
- [ ] order 내부/외부 책임 분리안 작성

---

## 최종 제안

`member-service`는 지금 당장 떼기보다 필요 시점에 분리하는 편이 맞다.
반면 `order-service`는 가장 마지막 단계에서 반드시 재정의해야 한다.

즉, 후반부 설계의 핵심은 "member를 떼는 것"보다 "order를 더 작고 선명한 책임으로 재구성하는 것"이다.
