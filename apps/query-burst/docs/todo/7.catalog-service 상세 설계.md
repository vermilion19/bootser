## 개요

`catalog-service`는 상품, 카테고리, 재고를 소유하는 서비스다.
현재 모놀리식 안의 `product` 도메인과 `category` 도메인을 중심으로 구성하되, 마이크로서비스 분리 이후에는 "상품 CRUD 서비스"가 아니라 "상품/카테고리/재고의 진실 소스"가 되어야 한다.

---

## 배경 (Why)

현재 `query-burst`에서 상품 도메인은 다음 책임을 동시에 갖고 있다.

- 상품 조회/생성/수정/삭제
- 카테고리 계층 관리
- 가격 및 상태 변경
- 재고 보유
- 주문 시 재고 차감 대상

문제는 주문 서비스가 상품 DB를 직접 읽고 수정한다는 점이다.
따라서 진짜 서비스 분리를 하려면 상품 도메인을 떼는 것보다 먼저 재고 소유권을 catalog로 이동해야 한다.

---

## 목표

- 상품/카테고리/재고의 소유 서비스를 분리
- 주문 서비스가 상품 DB를 직접 읽거나 수정하지 않도록 전환
- 카탈로그 조회 API와 재고 관련 API를 독립 배포 가능하게 구성

## 기능 요구사항

### FR-1. 상품 관리

- 상품 생성
- 상품 조회
- 상품 삭제
- 상품 가격 변경
- 상품 상태 변경

### FR-2. 카테고리 관리

- 카테고리 생성
- 카테고리 조회
- 카테고리 삭제
- 계층 구조 관리

### FR-3. 재고 관리

- 현재 재고 보유량 관리
- 재고 예약
- 재고 확정 차감
- 재고 해제

### FR-4. 주문 연동

- 주문 서비스의 재고 확보 요청을 처리해야 한다
- 요청별 멱등성을 가져야 한다
- 부족 재고, 중복 요청, 타임아웃을 처리해야 한다

## 비기능 요구사항

### NFR-1. 독립 데이터 소유권

- `product`
- `category`
- 재고/예약 관련 저장소

는 catalog 서비스가 소유해야 한다.

### NFR-2. 동시성 안전성

- 같은 상품 재고에 대한 동시 요청을 안전하게 처리해야 한다

### NFR-3. API/이벤트 계약 안정성

- 주문 서비스가 catalog 내부 구현을 몰라도 연동 가능해야 한다

### NFR-4. 관측 가능성

- 재고 예약 성공/실패율
- 재고 부족 빈도
- 예약 해제 실패
- 타임아웃

을 추적할 수 있어야 한다.

## 데이터 소유 모델

### 1. Product

주요 필드:

- `id`
- `name`
- `price`
- `stock`
- `status`
- `categoryId`
- `sellerId`
- `lastFenceToken`

주의:

- 분리 후에도 `lastFenceToken` 유지 여부는 재고 동시성 전략에 따라 달라진다

### 2. Category

주요 필드:

- `id`
- `name`
- `parentId`
- `depth`

### 3. Inventory Reservation

분리 후 추가가 권장되는 모델:

- `reservationId`
- `requestId`
- `orderId`
- `productId`
- `reservedQuantity`
- `status`
- `expiresAt`

상태 예시:

- `RESERVED`
- `COMMITTED`
- `RELEASED`
- `FAILED`

## 주문 서비스와의 경계

### order-service가 하면 안 되는 것

- product DB 직접 조회
- product DB 직접 수정
- 재고 락 직접 제어
- fence token 직접 비교

### catalog-service가 책임져야 하는 것

- 재고 동시성 제어
- 부족 재고 판단
- 예약/해제/확정
- 재고 상태와 상품 상태 일관성

---

## 동시성 전략

선택지는 세 가지다.

### 전략 A. DB 비관적 락

장점:

- 단순하다

단점:

- 고TPS에 불리하다

### 전략 B. Redis 락 + 펜싱 토큰

장점:

- 현재 구현을 가장 잘 재사용할 수 있다

단점:

- Redis 의존성과 fallback 설계가 필요하다

### 전략 C. 예약 테이블 + 원자적 업데이트

장점:

- 서비스 경계가 분명하다

단점:

- 구현 복잡도가 증가한다

초기 권장:

- catalog 내부에서는 B 또는 C를 선택
- order 서비스는 그 구현을 몰라야 한다

---

## 이벤트 설계

초기에는 내부 API 중심으로 시작해도 되지만, 이후에는 다음 이벤트를 검토할 수 있다.

- `StockReserved`
- `StockRejected`
- `StockReleased`
- `StockCommitted`
- `ProductPriceChanged`
- `ProductStatusChanged`

초기 분리 단계에서는 재고 예약은 동기 API, 상품 변경 전파는 필요 시 이벤트로 가는 것이 현실적이다.

---

## 마이그레이션 전략

### 단계 1. 내부 포트 도입

- 주문 서비스에서 재고 차감 경로를 `InventoryReservationPort` 같은 추상화로 감싼다

### 단계 2. 모놀리식 내부 구현 유지

- 포트의 구현은 아직 기존 product 도메인을 사용

### 단계 3. catalog 앱 생성

- 상품/카테고리 API와 재고 예약 API를 신규 앱으로 이동

### 단계 4. 외부 연동 전환

- order 서비스가 internal API를 호출하도록 전환

### 단계 5. direct repository 제거

- order 서비스의 `productRepository` 직접 의존 제거

## 완료 기준

- catalog 서비스가 상품/카테고리/재고의 공식 소유 서비스가 된다
- 주문 서비스는 상품 DB를 직접 다루지 않는다
- 재고 예약 계약이 구현된다
- 카탈로그 API가 독립 배포 가능하다

## 최종 제안

`catalog-service`는 상품/카테고리 분리가 아니라 "재고 소유권 이동"까지 포함해서 설계해야 한다.
따라서 이 서비스 설계는 반드시 재고 예약 설계와 함께 움직여야 한다.
