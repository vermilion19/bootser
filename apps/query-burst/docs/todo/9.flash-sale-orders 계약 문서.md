## 개요

`flash-sale-orders`는 플래시 세일 주문 생성 요청을 비동기로 전달하는 전용 Kafka 토픽이다.

이 토픽은 일반 `order-events`와 역할이 다르다.

- `order-events`: 주문 결과를 외부 read model에 전달
- `flash-sale-orders`: 플래시 세일 주문 생성을 비동기 작업으로 넘김

즉, `flash-sale-orders`는 통합 이벤트라기보다 내부 명령성 토픽에 가깝다.

---

## 목적

- 플래시 세일 요청에서 주문 생성을 비동기로 분리
- Redis 선차감 이후 실제 DB 주문 생성을 후속 consumer에게 위임
- 동기 분산 락 기반 주문과 플래시 세일 주문을 다른 처리 경로로 운영

---

## Producer

현재 producer는 `FlashSaleService`다.

흐름:

1. 요청 검증
2. Redis Lua Script로 플래시 세일 재고 선차감
3. 성공 시 `orderId` 생성
4. `flash-sale-orders` 토픽에 payload 발행
5. 클라이언트에는 `202 Accepted` 성격으로 응답

관련 구현:

- 즉 `order-events`와 달리 DB 트랜잭션과 발행 보장을 같이 묶는 구조가 아니다

## 전달 보장

현재 구조는 다음 성격을 가진다.

- Kafka 발행은 direct send
- consumer는 멱등성 체크 수행
- 실패 시 재고 보상 로직 수행

주의:

- producer 쪽은 Outbox 기반이 아니므로 `order-events`와 동일한 수준의 발행 보장을 갖는다고 보기 어렵다
- consumer는 예외가 발생해도 현재 구현상 `markProcessed()`를 호출하므로, 재고 보상 후 재시도를 막는 구조다

즉, 이 토픽은 "재시도 가능한 비동기 명령"이라기보다 "한 번 처리 후 성공 또는 보상 종료" 패턴에 가깝다.

## 처리 플로우

```text
Client
  -> FlashSaleOrderController
  -> FlashSaleService.requestOrder()
     -> Redis Lua Script 선차감
     -> Kafka publish(flash-sale-orders)
     -> 202 Accepted 응답

Kafka flash-sale-orders
  -> FlashSaleOrderConsumer
     -> FlashSaleService.processOrder()
        -> OrderService.createFlashSaleOrder()
           -> DB 주문 생성
     -> 실패 시 compensateStock()
```

## 멱등성 규칙

consumer는 다음 키 기준으로 중복 처리를 판단한다.

`CONSUMER:{groupId}:{orderId}:{eventType}`

현재 한계:

- `isDuplicate() -> 처리 -> markProcessed()` 구조
- 처리 성공 시뿐 아니라 보상 후에도 `markProcessed()` 호출

이 의미는 다음과 같다.

- 같은 메시지가 다시 와도 재처리되지 않음
- 대신 보상 이후 재도전도 자동으로 일어나지 않음

따라서 운영 정책상 "보상 후 종료"가 맞는지 반드시 의도적으로 확인해야 한다.

## 실패 시나리오

### 1. Redis 선차감 실패

- producer 단계에서 즉시 요청 실패
- Kafka 발행 없음

### 2. Kafka 발행 실패

- 현재는 direct send이며 outbox가 아님
- 선차감 후 발행 실패 시 처리 공백 가능성을 별도 검토해야 함

### 3. consumer 처리 실패

- `compensateStock()`으로 Redis 재고 보상
- `markProcessed()` 호출
- 동일 메시지 재처리 차단

### 4. 보상 실패

- 현재 자동 재시도나 DLQ 흐름은 문서화돼 있지 않다
- 운영 재처리 절차가 필요할 수 있다

따라서 장기적으로는 다음 둘 중 하나로 정리하는 것이 좋다.

- catalog/order 경계 재설계 후 보다 명확한 예약/확정 이벤트 체계로 발전

## 결론

`flash-sale-orders`는 현재 `query-burst`에서 플래시 세일 처리 경로를 분리하는 핵심 토픽이다.
다만 `order-events`와 달리 Outbox 기반 통합 이벤트가 아니라 비동기 명령성 토픽에 가까우므로, 계약과 운영 정책을 별도로 관리하는 것이 맞다.
