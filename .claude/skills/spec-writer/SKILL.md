## 작업

사용자의 요구사항을 분석하여 개발 가능한 스펙 문서로 정리합니다.

### 입력

- `$ARGUMENTS`로 전달된 요구사항
- 대화 중 언급된 기능 요청
- 참조할 기존 코드나 문서

### 스펙 문서 형식

```markdown
# [기능명] 스펙

## 개요
한 줄 요약

## 배경 (Why)
- 왜 이 기능이 필요한지
- 해결하려는 문제

## 요구사항

### 기능 요구사항 (Functional)
- [ ] FR-1: 구체적인 기능 설명
- [ ] FR-2: ...

### 비기능 요구사항 (Non-Functional)
- [ ] NFR-1: 성능, 보안, 확장성 등
- [ ] NFR-2: ...

## 영향 범위

### 변경 대상
| 레이어 | 파일/모듈 | 변경 내용 |
|--------|----------|----------|
| Domain | Entity.java | 필드 추가 |
| Application | Service.java | 메서드 추가 |

### 의존성
- 영향받는 다른 서비스/모듈
- 필요한 외부 라이브러리

## API 설계 (해당 시)

### Endpoint
```
POST /api/v1/resource
```

### Request/Response
간단한 JSON 예시

## 데이터 모델 (해당 시)
- 새로운 엔티티/테이블
- 기존 스키마 변경

## 테스트 시나리오
- 정상 케이스
- 예외 케이스
- 경계값 케이스

## 미결정 사항 (TBD)
- 추가 논의 필요한 항목

## 참고
- 관련 문서, 링크
```

### 작성 규칙

1. **구체적으로**: 모호한 표현 대신 측정 가능한 기준 사용
2. **DDD 관점**: 도메인, 애플리케이션, 인프라 레이어 구분
3. **프로젝트 컨벤션 준수**: CLAUDE.md의 패키지 구조 참고
4. **작게 시작**: MVP 범위로 먼저 정의

### 사용 예시

```
/spec-writer 웨이팅 호출 시 SMS 알림 기능
/spec-writer 식당 영업시간 관리 기능 추가
/spec-writer                              # 대화 내용 기반으로 정리
```
