## 작업

작성된 스펙 문서를 리뷰하고, 불필요한 부분을 제거하여 바로 개발에 착수할 수 있는 형태로 정제합니다.

### 입력

- `$ARGUMENTS`로 전달된 스펙 문서 경로
- 또는 직전에 `/spec-writer`로 생성된 스펙

### 리뷰 관점

#### 1. 실현 가능성 (Feasibility)
- 현재 아키텍처에서 구현 가능한가?
- 기술적 제약사항은 없는가?
- 의존성 충돌은 없는가?

#### 2. 범위 적정성 (Scope)
- MVP로 줄일 수 있는 부분은?
- 나중에 해도 되는 기능은?
- 오버엔지니어링 요소는?

#### 3. 명확성 (Clarity)
- 모호한 요구사항은 없는가?
- 개발자가 바로 이해할 수 있는가?
- 빠진 정보는 없는가?

#### 4. 일관성 (Consistency)
- 기존 코드베이스 패턴과 맞는가?
- 네이밍 컨벤션 준수하는가?
- DDD 원칙에 부합하는가?

### 출력 형식

```markdown
## 스펙 리뷰 결과

### 요약
- 전체 평가: (Ready / Needs Work / Major Revision)
- 예상 복잡도: (Low / Medium / High)

### 간소화 제안

#### 제거 권장
- 항목: 이유

#### 단순화 권장
- 항목: 대안

#### 후순위 권장
- 항목: 이유

### 보완 필요

#### 명확화 필요
- 항목: 질문

#### 누락된 정보
- 항목

### 정제된 스펙

---
[간소화된 핵심 스펙만 추출]
---

### 구현 순서 제안
1. 첫 번째 단계 (필수)
2. 두 번째 단계
3. ...

### 체크리스트
- [ ] 도메인 엔티티 설계
- [ ] Repository 인터페이스
- [ ] Service 로직
- [ ] Controller/API
- [ ] 테스트 코드
```

### 간소화 원칙

1. **YAGNI**: 당장 필요 없는 기능 제거
2. **MVP First**: 핵심 기능만 남기기
3. **Incremental**: 점진적 확장 가능하게
4. **Actionable**: 바로 코딩 시작할 수 있게

### 사용 예시

```
/spec-review docs/waiting-sms-spec.md    # 특정 스펙 파일 리뷰
/spec-review                              # 직전 스펙 리뷰
```
